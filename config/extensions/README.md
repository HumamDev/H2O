# config/extensions/ — per-host/per-browser extension config

Established in Phase 8G-2 (2026-05-19) as a future home for per-host /
per-browser extension configuration that does NOT belong with the frozen
chatgpt+chrome legacy configs at the top of `config/`.

## What goes here

When a new host+browser combo is added, its config files land at:

```
config/extensions/<host>/<browser>/
├── keys.json           Chrome SPKI public keys + IDs OR Firefox gecko IDs
├── dev-order.tsv       (optional) per-host loader order
├── loader-deps.json    (optional) per-host dependency map
└── loader-tiers.json   (optional) per-host tier config
```

## Chrome `keys.json` format

```json
{
  "<variant>": {
    "key": "<base64-encoded-SPKI-public-key>",
    "id": "<derived-chrome-extension-id>"
  }
}
```

Generated by `tools/product/extensions/_shared/generate-chrome-key.mjs`
(future, Phase 8G-5).

## Firefox `keys.json` format

```json
{
  "<variant>": {
    "gecko_id": "h2o-<host>-<variant>-firefox@h2ocockpitpro.com",
    "strict_min_version": "109.0"
  }
}
```

Hand-edited. Firefox does NOT derive ID from a key; it uses the gecko_id
string directly.

## What does NOT go here

- **chatgpt+chrome legacy configs** stay at the top of `config/`:
  - `config/dev-order.tsv`
  - `config/loader-deps.json`
  - `config/loader-tiers.json`
  - `config/extension-keys.json` (Phase 8A-1 Chrome keys for the 8 existing chatgpt+chrome variants)
- **Operator-local secrets** stay at `config/local/`.

## Why a separate file per host+browser

Per-host/per-browser keys files isolate risk: rotating a `claude/firefox/`
gecko_id has zero blast radius against the chatgpt+chrome OAuth-Google
manifest key (Phase 8A-1) registered in Supabase Auth.
