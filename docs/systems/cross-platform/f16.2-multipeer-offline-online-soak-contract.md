# F16.2 - Multi-Peer / Offline-Online Soak Contract

Status: F16.2.a docs-only contract.

F15 is structurally closed. F16.1 is closed and provides the runtime conflict gate, preflight early-warning integration, settlement fail-closed integration, bulk row conflict classification, and closure validators that require the runtime conflict gate. F16.2 defines how those gates must be proved under deterministic multi-peer, offline/online replay conditions.

## 1. Architecture Verdict

F16.2 is a soak-proof phase, not a new product-feature phase. It must prove that F16.1 runtime conflict behavior holds when peers diverge locally, replay in different orders, replay exact duplicates, replay stale operations, and encounter settlement blockers before mutation-side effects.

F16.2 is split into four phases:

| Phase | Scope |
|---|---|
| F16.2.a | Docs-only multi-peer/offline-online soak contract |
| F16.2.b | Pure JS deterministic multi-peer soak proof harness and validator |
| F16.2.c | Studio/Tauri runtime proof runner integration |
| F16.2.d | Closure validator hardening requiring lightweight soak proof |

F16.1 proves isolated runtime conflict gates. F16.2 must prove peer-local divergence, replay ordering, delayed settlement, duplicate/idempotent replay, stale-base replay, out-of-order replay, and fail-closed side-effect boundaries.

The simulator must call the real F16.1 runtime conflict gate APIs where available:

- `H2O.Desktop.Sync.evaluateLibraryRuntimeConflict`
- `H2O.Desktop.Sync.evaluateLibraryCatalogRuntimeConflict`
- `H2O.Desktop.Sync.evaluateLibraryBindingRuntimeConflict`
- `H2O.Desktop.Sync.classifyLibraryBulkRuntimeConflictRows`

The simulator must also reuse real replay, watermark, and consumed-operation validation kernels where available. It must not reimplement an alternate fake conflict model that can drift away from runtime behavior.

## 3. Simulated Peer State Model

Each simulated peer must carry only hash-safe state:

| Field | Contract |
|---|---|
| `peerIdHash` | Hash identity for the simulated sync peer |
| `installIdHash` | Hash identity for the install |
| `deviceIdHash` | Hash identity for the device |
| `syncPeerIdHash` | Optional hash identity if the runtime exposes one |
| `catalogState` | Hash-keyed catalog state with lifecycle, revision/base hash, `nameHash`, and `colorHash` |
| `bindingState` | Hash-keyed binding edge state with `bindingKind`, left endpoint hash, and right endpoint hash |
| `cacheState` | Materialized `chats.category_id` observation only; never conflict authority |
| `folderBridgeState` | F7/F15 identity shadow and delegation flag state |
| `outbox` | Envelopes pending replay |
| `offlineQueue` | Ordered local actions while disconnected |
| `watermarks` | Subject/peer/revision forward-only state |
| `consumedOperations` | Event/dedupe ledger |
| `f5Reviews` | Pending or closed terminal review state plus evidence digest |
| `bulkBatches` | Batch hash, row classifications, and retry status |
| `clock` | Deterministic logical clock; scenario logic must not call `Date.now()` |

The peer model is a deterministic simulator state, not a persistence model. It must not write business tables, SQLite rows, watermarks, consumed operations, or cache state except inside proof-safe in-memory fixtures.

## 4. Replay Modes

F16.2.b must model these replay modes:

| Replay mode | Required behavior |
|---|---|
| Peer A online / Peer B offline | Online peer can settle first; offline peer replays later against advanced state |
| Both peers offline | Both peers queue operations with local expected state and reconcile later |
| Delayed replay | A valid local operation arrives after another peer has advanced the subject |
| Duplicate replay | Exact same operation/dedupe identity replays more than once |
| Stale-base replay | Operation expected base/revision no longer matches current state |
| Out-of-order replay | Later operation arrives before an earlier queued operation |
| Exact replay idempotency | Exact duplicate is idempotent, not a second mutation |
| Conflicting replay fail-closed before settlement mutation | Runtime conflict blocker prevents ledger, cache, publication, apply, and settlement side effects |

## 5. Harness Rules

The F16.2.b harness must:

- Call the real F16.1 conflict gate APIs for catalog, binding, bulk, cache, and F5 conflict decisions where available.
- Reuse real replay, watermark, and consumed-operation validation kernels where available.
- Keep default validation lightweight and deterministic.
- Use seeded deterministic randomness only in heavy opt-in mode.
- Keep all fixture data hash-only in output.
- Assert blocked settlement has zero ledger and side-effect deltas.
- Treat cache drift as warning-only and never as conflict authority.
- Preserve the F15 one-subject-per-proposal and binding-as-subject model.
- Preserve `folder.metadata` for folder endpoints; folders are not `library.catalog`.
- Preserve the deferred `folder_bindings` trigger decision from F15.11.

Each scenario must document the winner/loser rule, expected blocker or warning code, refresh/re-propose requirement, retry safety, side-effect expectation, cache authority rule, and privacy/redaction requirement.

| Case | Scenario | Replay mode | Winner / loser rule | Expected code | Refresh / re-propose | Retry safe | Side-effect expectation | Cache authority | Privacy |
|---:|---|---|---|---|---|---|---|---|---|
| 1 | Two peers create same label/tag/category | Peer A online / Peer B offline | First valid settlement wins; loser blocks | `library-catalog-cross-install-name-collision` | Required | Yes after refresh | Loser has zero mutation deltas | Not involved | Hash-only catalog state |
| 2 | Offline rename vs online rename | Delayed stale replay | Online settlement advances revision; offline replay blocks | `library-catalog-cross-install-stale-base` | Required | Yes after refresh | Stale replay has zero mutation deltas | Not involved | No raw names |
| 3 | Recolor vs archive | Out-of-order or delayed replay | First valid settlement wins; stale loser blocks | `library-catalog-cross-install-stale-base` or `library-catalog-cross-install-lifecycle-conflict` | Required | Yes after refresh | Stale replay has zero mutation deltas | Not involved | No raw color |
| 4 | Duplicate chat-label/chat-tag bind | Duplicate replay | Exact replay is idempotent; non-identical duplicate edge blocks | `library-binding-cross-install-duplicate-edge` | Not required for exact replay; required for conflict | Exact replay yes | Duplicate conflict has zero mutation deltas | Not involved | No raw chat/catalog IDs |
| 5 | Bind vs unbind same edge | Stale-base or out-of-order replay | Expected edge state mismatch blocks stale side | `library-binding-cross-install-state-conflict` or stale-base equivalent | Required | Yes after refresh | Blocked side has zero mutation deltas | Not involved | No raw endpoint IDs |
| 6 | Chat-category replacement race | Both peers offline | Independent unbind + bind; one-active-per-chat conflict blocks stale side | `library-binding-cross-install-state-conflict` | Required | Yes after refresh | Blocked side has zero mutation deltas | `library.binding` is source of truth; cache is non-authoritative | No `category_id` or `chats.category_id` |
| 7 | Chat-folder replacement race | Both peers offline | Same as chat-category; endpoint remains `folder.metadata` | `library-binding-cross-install-state-conflict` | Required | Yes after refresh | Blocked side has zero mutation deltas | Not cache-authoritative | No folder-as-catalog, no raw folder data |
| 8 | F7 fallback vs F15 delegated path | Mixed fallback/delegated replay | Bridge conflict warns or blocks based on active state; fallback remains available | `library-binding-f7-f15-identity-conflict` | Required if blocker | Yes after refresh | Blocked bridge conflict has zero mutation deltas | Not involved | Exercise delegation flag in both positions; hash-only identity shadow |

| 12 | Cache drift after reconnect | Delayed replay | Drift is warning-only; binding remains source of truth | `library-cache-cross-install-drift` | Refresh required | Yes | Cache repair is not automatic in soak proof | Cache never decides conflict authority | No `category_id` or `chats.category_id` |
| 13 | Conflict runtime unavailable during replay | Settlement replay | Settlement fails closed when required gate is unavailable | `library-conflict-runtime-required-unavailable` or current module-local required-unavailable blocker | Required | Yes after gate available | Zero mutation deltas before settlement writes | Not involved | Redacted error codes only |
| 14 | Settlement blocker before consumed-op/watermark | Conflicting replay fail-closed | Blocker occurs before consumed-op, watermark, bookkeeping, cache, publication, journal, apply, relay/outbox | Existing conflict blocker from case context | Required if stale/conflict | Yes after refresh where applicable | All mutation side-effect flags false | Cache unchanged | Hash-only report |

## 8. Ledger Delta Expectations

Blocked operations must show zero delta for:

- Consumed-operation writes.
- Watermark writes.
- Bookkeeping writes.
- Cache refresh writes.
- Publication terminal mutations.
- Relay terminal mutations.
- Outbox terminal mutations.
- Native calls.
- F5 calls.
- Apply side effects.
- Journal side effects.
- SQL execution for blocked bulk rows.

These deltas must be explicit in scenario output or summarized by `sideEffectSummary`. A blocked scenario that cannot prove zero deltas is not a passing scenario.

## 9. Privacy Boundary

Soak reports must be hash-only. They must not emit raw:

- Chat IDs.
- Catalog IDs.
- Folder IDs.
- Names.
- Titles.
- Folder names.
- Colors.
- Paths.
- Filenames.
- Bundle filenames.
- URLs or tokens.
- Content, messages, or attachments.
- `category_id`.
- `chats.category_id`.
- `folder_id`.
- `chat_id`.

F16.2.b must include a central leak scan over all scenario outputs. The leak scan must cover both success rows and blocked rows because blocker reports are often where raw fixture values accidentally leak.

## 10. Performance Boundaries

F16.2 focuses on correctness under replay, not maximum scale.

Default validator mode:

- Lightweight.
- Two peers.
- Fourteen required scenarios.
- Bounded simulated events.
- No 10k-scale stress.
- Suitable for regular validation runs.

Heavy mode:

- Opt-in only, for example `F16_SOAK_HEAVY=1`.
- Uses seeded deterministic randomness.
- Reports the seed in `performanceSummary`.
- May expand peer count, replay permutations, and event counts.
- Performance budget exceedance is warning-only in F16.2.

Large-scale stress targets, including 10k chats and broad bundle stress, are reserved for F16.3.

## 11. Validator Strategy

F16.2.b should create:

```text
tools/validation/sync/validate-f16-library-multipeer-soak.mjs
```

The validator must assert:

- The soak module exists.
- APIs and markers exist.
- The required scenario names exist.
- The required replay modes exist.
- Peer model fields exist.
- Privacy guardrails exist.
- Side-effect flags and ledger-delta false checks exist.
- Runtime conflict gate APIs are referenced.
- A lightweight deterministic proof runs in Node/VM.

F16.2.d should update closure only after the soak proof exists and is stable. Closure must require the lightweight soak proof, not heavy soak.

## 12. Risks and Mitigations

| Risk | Mitigation |
|---|---|
| Simulator diverges from runtime semantics | Reuse the real F16.1 gate and real replay/watermark/consumed-op kernels where available |
| Heavy soak slows default validators | Keep heavy mode env-gated with `F16_SOAK_HEAVY=1` |
| Cache drift is accidentally treated as authority | Keep explicit warning-only cache drift scenario and assert `library.binding` remains source of truth |
| F5 race proof is too synthetic | Use a terminal first-wins state machine scenario with evidence digests |
| Folder F7/F15 ambiguity | Keep endpoint as `folder.metadata`, exercise the delegation flag both ways, and do not model folders as `library.catalog` |
| Raw fixture values leak into reports | Run a central leak scan over all scenario outputs |
| Blocked replay mutates ledgers | Require zero-delta assertions for every blocked scenario |

## 14. Explicit Deferrals

F16.2.a does not implement:

- Runtime soak harness.
- Validator.
- Closure updates.
- Loader or pack wiring.
- Runtime conflict resolver changes.
- SQLite trigger changes.
- `folder_bindings` trigger protection.
- F7 deletion.
- UI controls.
- 10k-scale stress.

F16.2.b is the next phase and should implement the deterministic multi-peer soak proof harness plus validator.

## 15. Authoritative Summary

F16.2 proves that the F16.1 runtime conflict gate remains correct when peers diverge and later replay operations under delayed, duplicate, stale, and out-of-order conditions. The proof must use real runtime conflict APIs and real replay/watermark/consumed-op kernels where available, keep all output hash-only, prove blocked settlement has zero mutation deltas, and keep default validation lightweight. Heavy soak and large-scale performance stress are separate, opt-in concerns and do not belong in F16.2.a.
