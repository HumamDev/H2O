# F15.12 — Cross-Install Conflict Resolution Contract

**Status:** Architecture contract. F15.12.a is docs-only.
**Predecessor closure:** F15.11.f closed at `c0ebcf5318b083c10473bccf34e5b25434839a15` (`test(sync): add folder binding absorption proof`).
**Scope:** `library.catalog` and `library.binding` cross-install / multi-peer conflict resolution.
**Non-scope:** Runtime implementation, validators, SQLite triggers, store rewiring, F7 deletion, UI mutation controls.

## 1. Architecture Verdict

F15.12 is a doc-first contract followed by proof extension. F15.12.a defines the conflict-resolution contract only. F15.12.b will extend runtime proof and validators.

No new kernel primitive is required. F15.12 preserves the F14/F15 one-subject-per-proposal model and the F15 binding-as-subject model. Conflict handling composes existing kernel and execute-lane concepts: replay defense, consumed-operation idempotency, watermark monotonicity, lifecycle validation, uniqueness gates, F5 terminal closure rules, and operator-mediated retry.

`folder_bindings` SQLite trigger protection remains deferred to F15.11.g / F16 after runtime soak. F15.12 must not add those triggers and must not weaken the existing F15.8 trigger/sentinel model for protected F15 tables.

## 3. Deterministic Decision Rules

1. **First valid settlement wins** unless the later event is an exact idempotent replay.
2. **No field-level auto-merge in F15.** Rename, recolor, lifecycle, bind, and unbind conflicts do not merge partial fields across peers.
3. **Stale `baseHash` / expected current state fails closed.** A proposal generated against an older subject revision must refresh and re-propose.
4. **Same `eventDigest` / `dedupeKey` is idempotent.** Exact replay does not apply twice.
5. **Watermark is forward-only.** Regression and equal-conflict block.
6. **Consumed-op exact duplicates are idempotent.** Conflicting consumed state blocks.
7. **Binding uniqueness is logical, not only subjectId.** The uniqueness key is `bindingKind` plus endpoint subject hashes, with special one-active-per-chat rules for `chat-category` and `chat-folder`.
8. **Catalog uniqueness is logical.** Active uniqueness is `(originAccountIdHash, catalogKind, nameHash)`.
9. **F5 terminal closure is idempotent only for the same closure.** Different terminal decisions for the same review conflict.
10. **Cache is never conflict authority.** `chats.category_id` may be stale and can warn, but it never decides whether a binding operation wins.

## 5. Operator-Visible Conflict States

| State | Meaning | Operator action |
|---|---|---|
| `conflict-blocked` | Settlement/preflight refused an unsafe operation | Refresh and inspect current state |
| `refresh-required` | Local proposal was based on stale mirror/base state | Refresh, then re-propose if still desired |
| `retry-safe` | Exact replay or partial bulk recovery can be retried safely | Retry without creating duplicates |
| `partial-migration` | Bulk import/import recovery settled only a subset | Review blocked rows and retry remaining rows |
| `cache-drift` | Read cache differs from binding state | Refresh cache; do not treat as source-of-truth conflict |
| `pending-review` | F5 catalog tombstone is awaiting decision | Wait for review or inspect F5 panel |
| `terminal-f5-conflict` | F5 review already closed differently | No retry without a new operator proposal |
| `identity-bridge-warning` | F7/F15 chat-folder bridge is consistent but dual identity evidence exists | Monitor; no migration trigger protection yet |

## 6. Retry, Refresh, and Re-Propose Rules

- A stale loser must refresh local catalog/binding state and generate a new proposal from the current state.
- Exact replay must not re-apply, re-close F5, re-write consumed-op rows, or advance watermarks.
- Partial bulk import can retry idempotently; already-settled rows are skipped and failed rows are retried.
- Cache drift is warning-only and refreshable. It does not require re-proposal unless the binding source of truth also changed.
- Conflicting F5 terminal decisions block operator action. Recovery requires a new explicit proposal if the lifecycle still permits it.
- Chat-category and chat-folder replace remain two proposals: unbind old, bind new. If one step fails, the partial state is surfaced rather than auto-rolled back.
- No silent fallback from F15 delegated folder binding to F7. Explicit fallback remains available only under the guarded F15.11.d contract.

## 7. Replay, Watermark, and Consumed-Operation Behavior

Replay defense treats exact duplicates as safe and conflicting same-subject events as unsafe. Settlement must compare `eventDigest`, `dedupeKey`, `lineageId`, `subjectId`, operation kind, base/target hashes, and actor peer metadata.

Watermarks are scoped by subject and peer. A proposed watermark must be forward-only. Regression, subject-scope mismatch, peer-scope mismatch, and equal-conflict block. Equal idempotent replay is safe only when the revision, lineage, event digest, and dedupe key agree.

Consumed-operation rows are append-only. An exact duplicate operation is idempotent. A consumed row with the same event identity but conflicting status, subject, operation kind, or validation summary blocks with the existing consumed-operation conflict path.

Stale-base fail-closed behavior is mandatory. If a proposal's expected current state does not match the local settled state, the operation does not try to infer the operator's intent against the new state.

## 8. F5 Conflict Rules

- First terminal closure wins.
- Same terminal closure replay is idempotent.
- Conflicting terminal closure blocks with `library-catalog-f5-review-conflict`.
- Pending F5 review cannot execute catalog tombstone.
- `approved-restore` closes the review without Native tombstone apply.
- `approved-seal` requires the Native tombstone apply path after closure.
- `auto-expired` maps to the seal path if supported by the active F5 bridge.
- Duplicate F5 review rows with identical review/evidence digest are warning-only; rows with conflicting decision/evidence block.

## 9. Cache Conflict Rules

`library.binding` is the source of truth. `chats.category_id` is a materialized local read cache.

Cache drift is warning-only. The cache never decides whether a binding conflict exists, never overrides binding state, and never emits proposals. The only write authority for cache refresh remains the execute-settlement/cache bridge path protected by F15.8 sentinel rules.

Read-your-own-write is local only: after local binding settlement, local cache refresh is synchronous. Cross-install reads are eventually consistent and may show stale category assignment until remote settlement and cache refresh arrive.

## 10. Bulk Migration Conflict Rules

- Bulk migration uses deterministic batch and edge identity.
- Re-importing the same bundle across installs is idempotent.
- Duplicate rows are skipped or reported as idempotent, not duplicated.
- Conflicting rows are reported as partial conflicts, not silently overwritten.
- Retry is safe because settled rows are skipped and failed rows retain deterministic identity.
- Bulk envelopes and reports must never expose raw bundle filename, path, title, raw chat ID, raw catalog ID, raw catalog name, content, URL, or token.
- Bulk migration does not bypass catalog/binding uniqueness, replay, watermark, consumed-op, privacy, or settlement rules.

## 11. F7 Fallback vs F15 Chat-Folder Rules

F7 fallback remains available. F15 chat-folder delegation remains guarded and defaults off unless explicitly enabled by the F15.11.d flag/accessor contract.

Folder endpoints remain `folder.metadata`. Folders are not `library.catalog`; only the binding edge is absorbed into `library.binding` as `bindingKind: 'chat-folder'`.

`folder_bindings` trigger protection remains deferred to F15.11.g / F16. F15.12 does not protect `folder_bindings` with SQLite triggers because F7 fallback still writes through the legacy path.

Old F7 identity and F15 identity are bridged through migration shadow events. If both identities map to the same `(chatSubjectId, folderSubjectId)` and the migration digest agrees, the state is a warning/observability condition. If F7 and F15 identities imply different active folders for the same chat, the conflict blocks delegated settlement and surfaces `library-binding-f7-f15-identity-conflict`.

## 12. Proof Expectations for F15.12.b

F15.12.b should prove:

- Same catalog create on two peers: one wins, one blocks.
- Rename vs rename blocks stale loser.
- Recolor vs recolor blocks stale loser.
- Archive vs rename blocks stale loser.
- Tombstone approve-seal vs approve-restore conflict blocks.
- Tombstone makes new bind fail while existing binding remains warning-only.
- Duplicate chat-label/chat-tag/tag-category bind is idempotent or blocked.
- Chat-category replacement race surfaces partial state.
- Chat-folder replacement race uses F15.11 bridge and no F7 deletion.
- F7 fallback flag off remains compatible.
- F15 delegated chat-folder conflict uses `folder.metadata`.
- Category cache drift remains warning-only.
- Same bundle import on two installs is idempotent.
- Partial bulk import retry does not duplicate rows.
- No raw names, IDs, titles, folder names, paths, content, URLs, or tokens appear in conflict proof output.
- No direct store writes or trigger weakening are introduced.

## 15. Authoritative Summary

F15.12 resolves the provisional cross-install language from F15.0.x by choosing conservative convergence semantics: first valid settlement wins, exact replay is idempotent, stale or conflicting operations fail closed, and operators refresh/re-propose from current state. Catalog and binding remain single-subject lanes. Cache remains a local materialized read view. F5 terminal closure is authoritative for catalog tombstone decisions. F7 folder binding compatibility remains guarded and trigger protection remains deferred.
