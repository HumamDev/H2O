# F17.3 Migration / Rollback Contract

## Architecture Verdict

F17.3 is a release-safety proof phase.

Default F17.3 checks must be:

- Read-only.
- Fixture-based.
- Temp-file based.
- Explicitly isolated from real user databases.

Live DB backup/restore and runtime rollback drills remain:

- Explicit.
- Manual.
- Opt-in.
- Operator-attested.

F17.3 does not implement production backup automation by default. It defines and proves the release contract first, then a later phase can decide whether to add runtime backup-before-migration behavior.

## Current Repo Findings

Current repository findings:

- SQLite migrations are registered through `tauri_plugin_sql::Builder::default().add_migrations("sqlite:studio-v1.db", studio_migrations())` in `apps/studio/desktop/src-tauri/src/lib.rs`.
- `studio_migrations()` contains versions `1` through `13`.
- F17.2 already validates migration ordering and `v13` guarded trigger DDL presence.
- The SQLite DB URL used by Rust code is `sqlite:studio-v1.db`.
- The actual on-disk DB path is plugin-managed and is not explicitly resolved in app code today.
- No repo-level `.bak` backup-before-migration mechanism exists today.
- Current rollback proof patterns use `sqlite::memory:` or explicit transaction rollback.
- The `v13` trigger guard exists as `f16_folder_bindings_trigger_guard`.
- The `v13` guard is default-off.
- When active, the guard allows `f15.execute-settlement-writer` and `f16.folder-legacy-fallback`.
- The writer identity sentinel `h2o_writer_identity()` exists and is proved.
- No F17 migration/rollback validator exists yet.

## Backup-Before-Migration Strategy

The production strategy is to copy the active Studio DB before migration-sensitive release startup.

Required backup behavior:

- Copy `studio-v1.db` to a timestamped `.bak` file before migration-sensitive release startup.
- Include sidecars if present:
  - `studio-v1.db-wal`
  - `studio-v1.db-shm`
- Emit a backup manifest with:
  - Original path.
  - Backup path.
  - Size.
  - mtime.
  - sha256.
  - Source app version.
  - Target app version.
  - Observed timestamp.

Restore rules:

- Restore must happen only while the app and DB pool are closed.
- Restore copies the `.bak` back to the original DB path.
- Restore includes matching `-wal` and `-shm` sidecars only when those sidecars were backed up coherently.

Default validator behavior:

- Use temp fixture files only.
- Never resolve or touch the real plugin-managed `studio-v1.db` path.
- Prove manifest shape and copy/restore mechanics against temp files.

Runtime implementation, if added later, must run before the first migration-triggering DB open. Because Tauri SQL migrations are registered on `sqlite:studio-v1.db`, backup automation must happen before the app opens that DB through the plugin.

## Old-DB Compatibility Proof Strategy

F17.3.b should use synthetic fixture DB definitions for:

- Empty DB.
- `v1`-only DB.
- `v3` folder-era DB.
- `v12` pre-folder-trigger DB.
- `v13` current DB.

Default proof should validate:

- Migration order by source parse.
- Fixture metadata classification as old/current/unknown.
- Backup manifest creation before simulated migration.
- No real DB path was touched.

Optional Rust/cargo proof may apply real SQL migrations to `sqlite::memory:` or a temp DB if practical. It must not use a production DB path by default.

## v13 Trigger Guard Rollback Strategy

Rollback for `v13` means:

- Disable the guard.
- Do not drop the triggers.

Rules:

- The guard table remains installed.
- `f16_folder_bindings_trigger_guard.enabled = 0` restores legacy-compatible write behavior.
- Active guard blocks unauthorized `folder_bindings` insert/update/delete.
- Active guard permits:
  - Settlement identity: `f15.execute-settlement-writer`
  - Scoped legacy fallback identity: `f16.folder-legacy-fallback`

Rollback proof should assert:

- Default-off.
- Active-blocking.
- Authorized-pass.
- Disable-restores-legacy.

## Default Vs Optional Checks

Default checks:

- Source-parse migration ordering.
- Backup manifest/copy/restore drill on temp files.
- Old-DB fixture classification.
- `v13` guard rollback contract.
- No real DB path touched.
- F17.1 and F17.2 compatibility.

Optional checks:

- `--cargo` Rust proof/compile.
- `--live-db-attested` operator-attested real runtime proof.
- `--restore-drill` if restore drill becomes heavier than the default temp-file drill.

Production DB mutation is never allowed by default.

## Validator / Proof Strategy

F17.3.b should output schema:

```text
h2o.release.migration-rollback-validation.v1
```

Output should include:

- Migration order result.
- Old-DB fixture result.
- Backup manifest result.
- Restore drill result.
- `v13` guard rollback result.
- No-real-DB-touched proof.
- Blockers.
- Warnings.
- Optional check states.

Blockers:

- Missing migration version.
- Duplicate migration version.
- Missing `v13` guard DDL.
- Backup manifest missing hash/path/size.
- Restore drill failed.
- Guard disable lever missing.
- Validator touched real DB path.

Evidence should clearly separate default temp-file proof from live/operator-attested proof.

## Risks And Mitigations

| Risk | Mitigation |
|---|---|
| Backup runs after migrations | Require backup before the first migration-triggering DB open |
| Real user DB touched by validator | Default validator uses temp files and synthetic fixtures only |
| WAL sidecar inconsistency | Backup sidecars only when DB is closed or sidecars are copied coherently |
| `v13` rollback misunderstood as trigger removal | Contract defines rollback as guard disable, not trigger drop |
| Old DB fixture too synthetic | Add optional anonymized fixture or live operator attestation later |
| Restore drill corrupts state | Default restore drill runs only on temp files |
| Plugin-managed DB path ambiguity | Contract documents the gap and requires an explicit implementation decision before runtime automation |
