# F19.1 Chrome/Desktop Library Parity Contract

Status: F19.1.a contract and read-only diagnostic foundation.

## Architecture Verdict

F19 reopens Sync Architecture under the stricter premium-sync standard. The F15/F16/F17 work proves Desktop-local Library Sync kernels, settlement safety, conflict gates, simulated replay, migration safety, privacy, and Desktop runtime proof health. It does not prove that Chrome Studio and Desktop Studio automatically share one live Library state.

F19.1.a adds a read-only parity contract and diagnostic. It must not mutate Chrome storage, Desktop SQLite, sync-folder files, backend state, identity state, or any propagation path. The diagnostic only captures a hash-only snapshot from the current surface and compares it with a caller-supplied peer snapshot.

Premium sync remains incomplete after F19.1.a. Completion requires later propagation and live two-surface proof.

## Current Root Cause

Chrome Studio and Desktop Studio currently read different live sources:

| Surface | Current source | Notes |
|---|---|---|
| Chrome Studio | Chrome/MV3 archive services, `chrome.storage.local`, Studio IndexedDB/localStorage, and native ChatGPT broadcast projection | Chrome can see linked-only native records and Chrome-local folders/categories. |
| Desktop Studio | Tauri SQLite `sqlite:studio-v1.db` via Desktop `store/*.tauri.js` modules | Desktop `LibraryIndex` refreshes from SQLite-backed entity stores and intentionally skips Chrome-style index persistence. |
| Shared live backend | None currently proven for Library state | Supabase/identity surfaces exist elsewhere, but no Library backend/source-of-truth is currently wired for Chrome/Desktop parity. |
| Sync folder bridge | Optional local file bridge using `latest.json` and `chrome-latest.json` | Disabled or user-configured by default; not premium automatic live sync. |

This explains the observed mismatch: Chrome can show `10 total / 9 saved / 1 linked` with folders/categories while Desktop shows `7 total / 4 saved / 0 linked` and no folders/categories because each surface is hydrating from its own local source.

## Snapshot Shapes

### Chrome Source Snapshot

Schema: `h2o.studio.sync.library-parity-snapshot.v1`

Required fields:

- `schema`
- `version`
- `surface`: `chrome-studio`
- `sourceType`: `chrome-library-index`
- `sourceAvailable`
- `sourceMetadata`
- `counts`
- `hashes`
- `warnings`
- `observedAtIso`

The Chrome snapshot should read from the in-memory `H2O.LibraryIndex` cache and, when available, read-only `H2O.LibraryWorkspace` catalog APIs. It must not trigger export/import, write `chrome.storage.local`, write IndexedDB, or call propagation APIs.

### Desktop Source Snapshot

Schema: `h2o.studio.sync.library-parity-snapshot.v1`

Required fields:

- `schema`
- `version`
- `surface`: `desktop-studio`
- `sourceType`: `desktop-sqlite-library-index`
- `sourceAvailable`
- `sourceMetadata`
- `counts`
- `hashes`
- `warnings`
- `observedAtIso`

The Desktop snapshot should read from the in-memory `H2O.LibraryIndex` cache and, when available, read-only Desktop store/catalog APIs. It must not write SQLite, trigger settlement, import sync folders, export bundles, refresh caches with writes, or call propagation APIs.

## Parity Schema

Schema: `h2o.studio.sync.chrome-desktop-library-parity.v1`

Output fields:

- `schema`
- `version`
- `ok`
- `status`
- `chrome`
- `desktop`
- `mismatches`
- `blockers`
- `warnings`
- `observedAtIso`

`ok:true` means both snapshots are present, both sources are available, and all compared counts/hashes match. `ok:false` is diagnostic only in F19.1.a; it must not trigger repair or sync.

## Compared Fields

The diagnostic compares hash-only/redacted evidence:

- total library count
- saved count
- linked count
- pinned count
- archived count
- folder count and folder list hash
- label count and label list hash
- category count and category list hash
- project count and project list hash, if available
- recent chats hash/fingerprint
- source metadata showing Chrome vs Desktop source and availability
- identity/workspace known vs unknown status

## Privacy Boundary

Parity output must be hash-only. It must not emit raw:

- chat IDs
- snapshot IDs
- catalog IDs
- folder IDs
- project IDs
- names
- titles
- folder names/colors
- paths
- URLs/tokens
- content/messages/attachments
- `category_id`
- `chats.category_id`
- `folder_id`
- `chat_id`

Raw values may be used only transiently inside the current process to compute hashes. They must not appear in returned diagnostic JSON.

## Why Prior Proofs Did Not Catch This

F15/F16/F17 proofs validated the local sync architecture and release safety:

- catalog/binding canonicalizers, diagnostics, preflight, settlement, and store cutover
- runtime conflict gate fail-closed behavior
- simulated multi-peer replay and performance stress
- Desktop runtime proof health
- migration/rollback and release evidence

They did not launch Chrome Studio and Desktop Studio as two live surfaces, capture both visible Library states, and assert convergence. F16.2 multi-peer soak is a deterministic simulated proof, not a real MV3 Chrome storage to Tauri SQLite parity check. F17.4 was a Desktop runtime smoke, not a Chrome/Desktop parity smoke.

## F19 Completion Dependency

F19.1.a only creates the read-only foundation. Premium sync completion requires later phases to add:

- a live Chrome snapshot capture path
- a live Desktop snapshot capture path
- a documented identity/workspace boundary
- automatic propagation or a shared backend/source-of-truth
- conflict-gated cross-surface replay
- a live parity proof that fails when Chrome and Desktop visible Library state diverge
