## 1. Executive Summary

This doc defines exactly how operations in one F15 lane affect the other lane, and how those consequences surface as diagnostics, blockers, warnings, or operator-visible UI affordances.

Top-level decisions:

| Decision | Verdict | Status |
|---|---|---|
| Cross-lane principle | **Catalog lifecycle is observed by bindings; bindings are NEVER automatically cascaded by catalog lifecycle. Operator action is required to bring orphaned bindings to a coherent state.** | **final** |
| Catalog rename → bindings | **Zero impact** (bindings reference catalog by subjectId, not by name or revisionHash). No warnings, no orphan diagnostics, no preflight gates trigger. | **final** |
| Catalog recolor → bindings | **Zero impact**. Display-only change. | **final** |
| Catalog archive → existing bindings | **Info-level warning** only. Bindings remain in `bound` state. UI dims related rows but does not surface action. | **final** |
| Catalog archive → new bind attempt | **Blocker** at preflight (`binding-endpoint-not-active`). New binds require an active catalog endpoint. | **final** |
| Catalog tombstone (into `retained`) → existing bindings | **Warning** + orphan diagnostic (`binding-orphaned-catalog-retained`). Bindings remain bound; UI surfaces amber badge + "Unbind to clean" action. | **final** |
| Catalog `expired` / `tombstoned` → existing bindings | **Warning** + escalated orphan diagnostic (`-expired` / `-tombstoned` codes). Bindings remain bound for audit. | **final** |
| Catalog non-active → new bind attempt | **Blocker** (`binding-endpoint-not-active`). Applies to archived, retained, expired, AND tombstoned. | **final** |
| Catalog tombstoned → unbind permitted? | **Yes — unbind is permitted regardless of catalog state.** This is the operator's only path to clean up orphans. | **final** |
| Binding apply gate (active uniqueness) | **One active binding per (subjectType, bindingKind, leftSubjectId, rightSubjectId).** Re-bind blocked. | **final** |
| chat-category uniqueness | **One active `chat-category` binding per chat (leftSubjectId).** Re-bind blocked with `chat-category-conflict`. | **final** |
| chat-folder uniqueness (post-F15.5) | **One active `chat-folder` binding per chat.** Same shape as chat-category. | **final** |
| `relatedSubjects` annotation storage | **Diagnostic envelope only.** Not stored in canonical envelopes (`revisionHash`). Aggregated digest may appear in bookkeeping rows. | **final** |
| `relatedSubjects` max count | **50 per diagnostic emit.** Beyond 50: truncate + emit `related-subjects-truncated` info diagnostic. | **final** |
| Materialized cache (`chats.category_id`) | **Diagnostic-only treatment in F15.0.1.** Full contract in F15.0.2. Source of truth = `library.binding` state. Chat.metadata domain is a read-only consumer of the cache. | **final** |
| Cross-lane consequence cascade | **Strictly forbidden.** Operator action required for every catalog-induced binding cleanup. | **final** |
| F15.0.0 §6.3 vs §7.3 ambiguity | **Resolved here: §7.3's "active only" wins.** §6.3's "not tombstoned" is the authoring error; F15.0.1 supersedes F15.0.0 on this specific gate. | **final** |
| Replace-category UX | **Two-proposal model** at lane level (unbind old + bind new); single operator click at UI level. UI tracks the pair as a logical transaction but lanes see independent proposals. | **final** |

**Bottom line:** the cross-lane interaction surface is small, structured, and operator-mediated. Catalog lifecycle propagates to bindings only as diagnostic warnings + new-bind blockers — never as automatic state mutation. The `relatedSubjects` annotation contract is privacy-safe, count-limited, and confined to diagnostic output. Materialized cache for `chats.category_id` is diagnostic-only at this phase; the full read-cache contract lands in F15.0.2.

## 2. Cross-Lane Principle

**The single binding rule that all other F15.0.1 decisions derive from:**

> Catalog lifecycle is observed by bindings; bindings are NEVER automatically cascaded by catalog lifecycle. Operator action is required to bring orphaned bindings to a coherent state.

This principle has three immediate consequences:

1. **No auto-unbind.** When a catalog is tombstoned or archived, its existing bindings remain in `bound` state. They become "orphaned" relative to a non-active catalog, but the binding lane does NOT emit unbind proposals on their behalf.
2. **New binds require active catalogs.** Once a catalog leaves the `active` state, the binding lane refuses new binds against it. This prevents fresh orphans from accumulating during a catalog's deprecation window.
3. **Unbind is always permitted.** Regardless of the right-side catalog's state, the binding lane allows unbind. This is the operator's exclusive cleanup path. Without it, orphaned bindings would be eternally stuck.

The principle deliberately rejects two appealing alternatives:

- **Auto-cascade unbind on catalog tombstone.** This is the natural "intuitive" choice but is high-blast-radius: tombstoning a popular label could remove N hundred bindings at apply time, creating an apply storm in the execute lane and unobservable mass state change for operators on other installs. F15.0.0 §7.3 explicitly forbade this.
- **Hard-block unbind when catalog is tombstoned.** Initially appealing for audit integrity, but creates a permanent orphan state with no operator recovery. Rejected.

The principle aligns with F14's "explicit operator action for consequential changes" pattern (per F14.5.5 F5 review queue contract: operators must explicitly approve seal / restore).

## 3. Catalog → Binding Matrix

For each catalog operation, the impact on the binding lane:

| Catalog operation | Catalog lifecycle effect | Impact on existing bindings | Impact on new bind attempts | Diagnostic emitted on bindings | UI affordance |
|---|---|---|---|---|---|
| `create` | (initial → `active`) | No bindings exist yet | n/a (no binding to be made yet) | none | none |
| `rename` | no lifecycle change (nameHash + revisionHash change) | **zero impact** — bindings reference by subjectId | bindings can still be made | none | none |
| `recolor` | no lifecycle change | **zero impact** — display only | bindings can still be made | none | none |
| `archive` | `active → archived` | info-level warning per related binding | **blocked**: `binding-endpoint-not-active` | `binding-orphaned-catalog-archived` (info) | row dimmed; tooltip "catalog archived" |
| `restore-from-archived` | `archived → active` | warnings cleared | new binds allowed again | warnings cleared | row un-dimmed |
| `tombstone` (proposal) | `active \| archived → retained` (via F5 review queue) | warning per related binding | **blocked**: `binding-endpoint-not-active` | `binding-orphaned-catalog-retained` (warning) | amber badge + "Unbind to clean" |
| `restore-from-retained` | `retained → active` | warnings cleared | new binds allowed again | warnings cleared | badges cleared |
| (auto) `retention-expired` | `retained → expired` | escalated warning | **blocked** | `binding-orphaned-catalog-expired` (warning) | orange badge + "Unbind to clean" |
| (auto) `terminal-sealed` | `expired → tombstoned` | terminal warning | **blocked** | `binding-orphaned-catalog-tombstoned` (warning) | red badge + "Unbind to clean" |

### 3.1 Why archive is non-cascading

Archive is operator-reversible without time pressure. An archived label can be restored indefinitely. Existing bindings against an archived label are "soft orphans" — they still semantically describe a relationship that may become active again. Auto-unbinding on archive would be over-eager.

### 3.2 Why retained / expired / tombstoned are non-cascading

The F14.5.2 retention amendment explicitly designed retention as a **deferred-action** model: operators have a window to reconsider. If bindings were auto-unbound on tombstone proposal, the F5 review queue's restore path would have nothing to restore to (the bindings would already be gone). Keeping bindings bound preserves the F5-restore path's idempotency.

### 3.3 Why new binds are blocked against all non-active catalogs

Per the cross-lane principle, fresh orphans during a catalog's deprecation window add to the operator's cleanup burden. Blocking new binds is the smallest-surface defense: it prevents the orphan set from growing during the F5 review window. The blocker code is uniform (`binding-endpoint-not-active`) regardless of which non-active state the catalog is in; the diagnostic specificity comes from the related-subjects annotation.

### 3.4 Doc correction relative to F15.0.0

F15.0.0 §6.3 specified the `endpointsResolvable` binding preflight gate as "their canonical subjects are not tombstoned." F15.0.0 §7.3 contradicted this with "block NEW bind proposals against a non-active catalog endpoint."

**F15.0.1 resolution:** §7.3's tighter "active only" rule is final. F15.0.0 §6.3 is **superseded by F15.0.1 §4.2** below on this specific gate. The F15.0.0 doc text should be amended at next opportunity (or treat F15.0.1 as the source of truth for this gate). All other F15.0.0 §6.3 gates remain in force.

## 4. Binding → Catalog Matrix

For each binding operation, the catalog-state requirements:

### 4.1 `bind` operation

| Requirement | Detail | Blocker code if violated |
|---|---|---|
| Right-side catalog endpoint must exist | Canonical catalog subject resolvable; revisionHash readable | `binding-endpoint-missing` |
| Right-side catalog must be `active` | (per F15.0.1 §3.4 amendment) | `binding-endpoint-not-active` |
| Left-side chat endpoint must be resolvable | Chat subjectId hash present in local snapshot | `binding-chat-endpoint-missing` |
| Uniqueness invariant | Per bindingKind, see §4.3 | `binding-already-bound` / `chat-category-conflict` / `chat-folder-conflict` |
| Cross-account safety | originAccountIdHash matches localAccountIdHash | `cross-account-binding-attempt` |
| Watermark monotonicity | Replay defense passes | `binding-replay-detected` |
| Privacy scan | `kernel.scanDomainForbiddenFields('library.binding', envelope)` returns ok | `library-binding-row-contains-forbidden-field` |

### 4.2 `unbind` operation

| Requirement | Detail | Blocker code if violated |
|---|---|---|
| Target binding must exist | Canonical binding subject resolvable | `binding-not-found` |
| Target binding must currently be `bound` | Already-unbound binding is a noop | `binding-not-bound` |
| Right-side catalog state | **Any state permitted** — this is the orphan cleanup path | (no gate) |
| Left-side chat state | **Any state permitted** — including tombstoned chats | (no gate) |
| Cross-account safety | originAccountIdHash matches localAccountIdHash | `cross-account-unbind-attempt` |
| Watermark monotonicity | Replay defense passes | `binding-unbind-replay-detected` |
| Privacy scan | `kernel.scanDomainForbiddenFields('library.binding', envelope)` returns ok | `library-binding-row-contains-forbidden-field` |

### 4.3 Uniqueness invariants by bindingKind

| bindingKind | Endpoint types | Uniqueness rule | Blocker code on violation |
|---|---|---|---|
| `chat-label` | `left = chat.metadata`, `right = library.catalog` (label) | At most one active `(chat, label)` pair | `chat-label-already-bound` |
| `chat-tag` | `left = chat.metadata`, `right = library.catalog` (tag) | At most one active `(chat, tag)` pair | `chat-tag-already-bound` |
| `chat-category` | `left = chat.metadata`, `right = library.catalog` (category) | At most one active `chat-category` binding per chat (across ALL categories) | `chat-category-conflict` |
| `tag-category` | `left = library.catalog` (smaller-id), `right = library.catalog` (larger-id) | At most one active `(tag, category)` pair | `tag-category-already-bound` |
| `chat-folder` (post-F15.5) | `left = chat.metadata`, `right = library.catalog` (folder) | At most one active `chat-folder` binding per chat | `chat-folder-conflict` |

### 4.4 The chat-category / chat-folder "replace" workflow

For chat-category and chat-folder, the uniqueness rule means an operator who wants to move a chat from category A to category B must:

1. Unbind the existing `chat-category` binding to A.
2. Bind a new `chat-category` binding to B.

The UI surfaces this as a single operator action ("Move to category B"). Internally, the UI emits two binding proposals and tracks them as a logical transaction at the UI layer (failure of either should ideally roll back, but the lane sees them as independent — see Risk §10.4). At lane level, there is no "replace-category" operation; there are only `bind` and `unbind`.

|---|---|---|---|---|---|---|

| `binding-orphaned-catalog-archived` | info | binding | yes (dim badge) | no | no | Right-side catalog is archived; binding still valid |

| `binding-unbind-replay-detected` | blocker | binding | hidden | yes | yes | Replay defense on unbind |
| `library-binding-row-contains-forbidden-field` | blocker | binding | hidden | yes | yes | Privacy scan caught forbidden field in binding envelope |

### 5.1 Severity semantics

- **info**: diagnostic emitted for audit / observability; UI does NOT surface; proposals proceed unaffected
- **warning**: diagnostic emitted; UI surfaces with non-blocking badge or banner; proposals proceed
- **blocker**: diagnostic emitted; proposal generation refused at preflight; if somehow reached apply, execute lane rejects

### 5.2 Owning lane semantics

- **catalog**: emitted by catalog lane diagnostics module
- **binding**: emitted by binding lane diagnostics module
- **both**: emitted by either lane when condition is detected; cross-lane diagnostics surface in either lane's diagnostic output
- **aggregate (cross-lane)**: emitted only by a future F15.x cross-lane aggregator (not a per-lane diagnostic)
- **diagnostic envelope**: meta-diagnostic about the diagnostic process itself

## 6. relatedSubjects Contract

### 6.1 Canonical shape

```
relatedSubjects: [
  {
    relationKind:       enum,          // see §6.2
    sourceSubjectType:  enum,          // 'library.catalog' | 'library.binding'
    sourceSubjectId:    sha256,        // who emitted / observed the relation
    targetSubjectType:  enum,          // 'library.catalog' | 'library.binding' | 'chat.metadata'
    targetSubjectId:    sha256,        // who is being referenced
    impactKind:         enum,          // see §6.4
    severity:           'info' | 'warning' | 'blocker',
    diagnosticCode:     string,        // matches §5 taxonomy
    evidenceDigest:     sha256,        // hash over canonical evidence content
    observedAtIso:      ISO (hour-coarsened)
  }
]
```

### 6.2 `relationKind` enum (final)

- `orphans-binding` — source (catalog) state change creates an orphaned target (binding)
- `requires-active-endpoint` — source (binding) operation requires target (catalog) be active
- `requires-resolvable-endpoint` — source (binding) operation requires target (chat) be resolvable
- `uniqueness-conflicts-with` — source (proposed binding) collides with existing active target binding
- `name-collides-with` — source (proposed catalog rename) collides with target's nameHash
- `references-cache-stale` — source observes target materialized cache is out of date

### 6.3 `impactKind` enum (final)

- `blocks-new-binds` — source state prevents future binds against it (applies to non-active catalog states)
- `orphan-warning` — source state leaves target in orphan position (UI surfaces but no blocker)
- `cleanup-suggested` — source diagnostic recommends operator cleanup of target
- `blocks-proposal` — source diagnostic blocks proposal generation
- `blocks-apply` — source diagnostic blocks execute-lane apply (rare; usually `blocks-proposal` is sufficient)
- `informational-only` — source diagnostic provides context but no actionable effect

### 6.4 Forbidden fields in `relatedSubjects` entries

The following must NEVER appear in a `relatedSubjects` entry, enforced via `kernel.scanDomainForbiddenFields('library.binding', relatedSubjectsEntry)`:

- Raw catalog name, raw color, raw description, raw operator notes
- Raw chat title, raw chat content, raw turn data
- Raw account identifier, raw user identifier
- Raw catalog ID, raw chat ID, raw binding ID (only sha256 hashes allowed)
- Raw timestamps at sub-hour precision (`observedAtIso` is hour-coarsened)
- Any payload field that would expand into an audit trail of raw data

### 6.5 Storage policy

| Storage location | `relatedSubjects` treatment |
|---|---|
| Canonical envelope (`revisionHash`) | **NOT included.** `revisionHash` is identity; `relatedSubjects` is annotation. Including would couple identity to cross-lane state, breaking the F14.2 one-subject-per-proposal contract. |
| Bookkeeping ledger row body | **NOT stored.** Bookkeeping rows are immutable evidence chains; cross-lane state can change after the fact. |
| Bookkeeping ledger row metadata | **Aggregated digest only.** A `relatedSubjectsDigest` (sha256 of canonicalJSON of full set at emit time) MAY appear, providing tamper evidence without storing the set itself. |
| Diagnostic envelope | **Full storage.** This is the canonical home. Diagnostic output may carry up to 50 entries; >50 truncates with `related-subjects-truncated`. |
| Preflight result metadata | **Inline.** Preflight may return `relatedSubjects` directly in result `metadata` for caller inspection. |
| ApplyEvent receipt | **Inline.** Receipts carry `relatedSubjects` reflecting the snapshot at receipt-build time. |

### 6.6 Max-count + truncation

| Rule | Value |
|---|---|
| Maximum entries per emit | **50** |
| Truncation behavior | Sort by `severity` (blocker > warning > info) then by `observedAtIso` (newest first); keep first 50 |
| Truncation diagnostic | Emit `related-subjects-truncated` (info severity) with `truncatedCount: N` |
| Truncation determinism | Sort comparison fields are stable per emit; same input produces same truncation |

### 6.7 Provisional: cross-install reconciliation

The full contract for how `relatedSubjects` annotations from two installs reconcile during cross-install sync is deferred to F15.6 (per F15.0.0 §12.6). F15.0.1 treats `relatedSubjects` as **install-local** — each install computes its own annotation set based on its own observed state. Cross-install agreement on annotations is not guaranteed before F15.6.

## 7. Materialized Cache Preview

This section previews the F15.0.2 materialized cache contract for `chats.category_id`. **Full contract lands in F15.0.2.** F15.0.1 commits to the framing only.

**Pointer:** The committed materialized cache contract is now at `f15.0.2-materialized-category-cache-contract.md`. F15.0.2 fulfills the deferral in §7.5 below; it does not supersede F15.0.1 on any other point.

### 7.1 Source-of-truth alignment

- **Source of truth:** `library.binding` lane state for `chat-category` bindings.
- **Read cache:** `chats.category_id` column (SQLite) is a materialized view derived from binding state.
- **Source-of-truth NEVER:** chat.metadata domain. Chat.metadata may READ the materialized cache for display but never WRITES `category_id`.

### 7.2 Cache lifecycle

- Cache is refreshed on every successful `library.binding` apply event whose `bindingKind = 'chat-category'`.
- Refresh is performed by the F14.6 execute settlement writer after the binding apply lands.
- Cache reads in chat.metadata UI are best-effort: stale reads are tolerated and detected as `materialized-cache-drift-detected` diagnostic.

### 7.3 Drift detection (F15.0.1 scope)

- A drift probe (to be specified in F15.0.2 / implemented in F15.2.a binding diagnostics module) compares `chats.category_id` against the currently-bound `chat-category` binding's `rightSubjectId`.
- Drift exceeding a threshold or persisting beyond a refresh-cycle window emits `materialized-cache-drift-detected` (warning).
- Drift is NEVER auto-corrected. The cache is rebuilt only on the next binding apply event (which proves the lane was authoritative).

### 7.4 Chat.metadata coupling boundary

F15.0.1 reaffirms F15.0.0 §1's coupling boundary:

- Chat.metadata domain has **no chat-category mutation operation**.
- Chat.metadata canonical envelope MAY carry a `categoryIdHash` field for display purposes (sha256 of the bound category's subjectId, present iff `chat-category` binding is `bound`), but this field is DERIVED from binding state and is NEVER a write source.
- The chat.metadata lane's diagnostic surface (F14.3.2) may emit `materialized-cache-drift-detected` if the chat-metadata-side read of categoryIdHash drifts from binding state, but this is a read-side diagnostic only.

### 7.5 Out of F15.0.1 scope

These specifics belong to F15.0.2:

- Cache rebuild trigger algorithm (event-driven vs poll vs hybrid)
- Cache invalidation semantics across install boundaries
- Cache-vs-binding race resolution during concurrent applies
- Schema migration for existing `chats.category_id` data on first F15 deployment

### 8.1 Replace-category modal contract

The chat-category-conflict modal has a strict contract:

- Always shows the **current** category's display name (rendered from local store; never from envelope).
- Always shows the **proposed** category's display name.
- Explicitly states: "This will unbind from [X] first, then bind to [Y]. Both steps will be recorded in the audit log."
- "Move" button emits TWO independent proposals: `unbind chat-category(chat, X)` then `bind chat-category(chat, Y)`. The UI tracks them as a single logical transaction.
- If the unbind proposal fails (e.g., binding was already unbound by another install), the UI surfaces an error and does NOT proceed with the bind.
- If the unbind succeeds but the bind fails (e.g., Y was tombstoned in the meantime), the chat is left without a category. UI surfaces this clearly: "Chat is no longer in any category. Choose a new one?"

### 8.2 Orphan cleanup view

A dedicated UI view (F15.10 scope) lists all orphan bindings with:

- shortHash of binding subjectId
- shortHash of left endpoint (chat)
- shortHash of right endpoint (catalog) + catalog's `currentState`
- bindingKind enum value
- `boundAtIso` (hour-coarsened)
- "Unbind" action button

## 9. Proof Expectations

The F15.9 binding and catalog proof harnesses must directly assert each of the following:

### 9.1 Catalog lifecycle → binding non-impact

- `catalog rename produces zero binding-side blockers / warnings`: emit a catalog rename proposal; assert that the binding lane's diagnostic on a related binding returns `binding-ok` and no warnings.
- `catalog recolor produces zero binding-side impact`: same shape.
- `catalog archive emits info-level diagnostic only`: assert `binding-orphaned-catalog-archived` (info) on related bindings; no blockers; binding remains in `bound` state.
- `catalog tombstone (into retained) emits warning + orphan diagnostic`: assert `binding-orphaned-catalog-retained` (warning); binding remains bound; UI affordance state matches §8.

### 9.2 Catalog non-active → new bind blocked

- `new bind to archived catalog is blocked with binding-endpoint-not-active`.
- `new bind to retained catalog is blocked`.
- `new bind to expired catalog is blocked`.
- `new bind to tombstoned catalog is blocked`.
- `new bind to active catalog is permitted`.

### 9.3 Unbind always permitted on bound bindings

- `unbind on binding whose right catalog is retained / expired / tombstoned is permitted`.
- `unbind on already-unbound binding returns binding-not-bound (idempotent noop, not a hard error)`.

### 9.4 Uniqueness invariants enforced

- `chat-category-conflict blocks second active chat-category binding`.
- `chat-label-already-bound blocks duplicate active chat-label`.
- `chat-tag-already-bound blocks duplicate active chat-tag`.
- `tag-category-already-bound blocks duplicate active tag-category`.
- `chat-folder-conflict (post-F15.5) blocks second active chat-folder`.

### 9.5 Materialized cache contract

- `chats.category_id materialized cache drift is detected as warning, not silently corrected`.
- `chat.metadata domain reads cache but never writes`: assert the chat.metadata canonicalizer + proposal generators carry no chat-category mutation operation.
- `cache rebuild happens on chat-category binding apply event`: assert sequence at execute lane settlement.

### 9.6 relatedSubjects privacy & truncation

- `relatedSubjects entries pass kernel.scanDomainForbiddenFields('library.binding', entry)`: inject an entry containing a raw name and assert it's blocked.
- `relatedSubjects truncates at 50 with related-subjects-truncated emitted`.
- `relatedSubjects is NOT included in revisionHash`: render the canonical envelope with and without `relatedSubjects`; assert `revisionHash` is identical.
- `relatedSubjects digest in bookkeeping is sha256 of canonical full set`: assert byte-identical digest computation.

### 9.7 Cross-lane orchestration during F5 review

- `during catalog tombstone F5 review, related bindings see orphaned-catalog-retained warning`.
- `if operator approves restore, bindings see warning cleared (binding-ok)`.
- `if operator approves seal, bindings escalate to orphaned-catalog-tombstoned (still bound)`.

### 9.8 Replace-category two-proposal model

- `replace-category emits exactly two binding proposals (unbind, then bind) — never a single combined proposal`.
- `if unbind succeeds but bind fails, chat is left without category, no rollback at lane level`.
- `proof verifies the UI-layer transaction is purely cosmetic; the lane sees independent operations`.

### 9.9 Side-effect summary

- `all F15.0.1-scoped diagnostic emits carry sideEffectSummary with all 8 flags false`.
- `cross-lane diagnostic emission does not write to bookkeeping ledger of either lane`.
- `relatedSubjects emission does not advance any watermark`.

## 10. Risks and Mitigations

**10.1 Orphan accumulation.** If operators don't clean up orphan bindings, the diagnostic noise grows. UI shows "[N] orphans" indefinitely.

*Mitigation*: `cross-lane-orphan-count-high` warning fires above a threshold (recommend N=100 per install); dashboard widget directs operators to bulk cleanup view (§8.2). Aggregate warning is non-blocking.

**10.2 Operator confusion on "orphan binding" semantic.** Operators may not understand why a binding exists when its catalog is gone.

*Mitigation*: badge tooltip explains: "Catalog [shortHash] was tombstoned. This binding remains for audit. Click 'Unbind to clean' to remove." Operator UX docs (F15.10 scope) include a "What is an orphan binding?" explainer.

**10.3 Replace-category atomicity gap.** If unbind succeeds but bind fails, the chat is left without a category. This is a real consistency hole at the lane level.

*Mitigation*: UI surfaces the half-completed state immediately with a clear message and a "Choose a category" prompt. The lane records both events in bookkeeping, so audit is complete. Operators can resolve by choosing a new category. No automatic rollback (auto-rollback would require multi-subject coordination, which violates F14.2 one-subject-per-proposal).

**10.4 Replace-category cross-install race.** Operator on install A clicks "Replace X → Y" while operator on install B clicks "Replace X → Z". Both unbind proposals collide; one of the bind proposals collides with the other's bind.

*Mitigation*: standard F14 replay-defense + uniqueness invariant. First-to-apply wins; second-to-apply receives blocker; UI surfaces "category was concurrently changed; refresh and try again." Full cross-install resolution contract in F15.6 / §6.7.

**10.5 relatedSubjects metadata leak risk.** Even with privacy scan, a future bug could inject a raw field into a `relatedSubjects` entry's `evidenceDigest` source data.

*Mitigation*: `relatedSubjects` entries are scanned at three points: emit time, ledger-store-time (digest only), and read-time before egress. Triple-defense matches F14.3 chat lane convention. Privacy reviewer (workflow agent) verification at F15.2.a diagnostics implementation phase.

**10.6 chats.category_id cache drift duration.** If cache rebuild is async and operators read during the rebuild window, they see stale category assignment. With many concurrent binding changes, drift could be persistent.

*Mitigation*: full contract in F15.0.2. F15.0.1 commits only to drift-detection-as-diagnostic. Operators see "view may be out of date — refresh" rather than silently-wrong data.

**10.7 chat-folder forward compatibility.** F15.0.1 specifies chat-folder uniqueness now, but F15.5 absorption is when chat-folder actually starts existing. Specifying it now risks designing for a future scenario that may need revision.

*Mitigation*: chat-folder rows in this doc are gated with "(post-F15.5)" markers. F15.0.1 reserves the bindingKind enum value and the conflict-diagnostic code; actual binding-side gate enforcement does not activate until F15.5. F15.5 may emit a small amendment to F15.0.1 if the bridge work reveals a different shape, but the API contract is stable.

**10.8 Auto-tagging proposal storm.** Per F15.0.0 §12.8, auto-tagging emits proposals that must flow through library.binding. Many proposals against already-bound (chat, tag) pairs would emit `chat-tag-already-bound` blockers, generating diagnostic noise.

*Mitigation*: auto-tagging engine pre-checks against the current binding state and only emits proposals for genuinely-new (chat, tag) pairs. UI for operator-driven auto-tag acceptance shows only the proposed new pairs. `chat-tag-already-bound` blockers from auto-tagging are silently dropped at the diagnostic surface (not surfaced to operators).

**10.9 Diagnostic taxonomy growth.** As F15.x evolves, new diagnostic codes will be added. The §5 table is final for F15.0.1 scope but will grow.

*Mitigation*: F15.x amendments must extend the §5 table here, not duplicate it. Diagnostic code authority lives in this doc; F15.x diagnostic modules reference §5 by name.

**10.10 F15.0.0 §6.3 vs §7.3 contradiction resolved here.** F15.0.0 was internally inconsistent on the `endpointsResolvable` gate. F15.0.1 resolves it but creates a small precedent for "F15.0.x docs can supersede prior F15.0.x docs."

*Mitigation*: F15.0.0 should receive a one-line corrigendum at next opportunity pointing to F15.0.1 §3.4. Future F15.0.x docs should NOT silently override prior F15.0.x docs; if they need to, they must explicitly cite the prior doc and the resolution (as F15.0.1 does for the §3.4 case).
