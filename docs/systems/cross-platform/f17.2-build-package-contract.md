# F17.2 Build / Package Validation Contract

## Architecture Verdict

F17.2 is a build/package validation orchestrator and contract. It should prove that the Node-runnable build chain produces coherent, version-consistent, migration-complete packages before release.

F17.2 should:

- Reuse existing build, release, archive, and validation tooling.
- Follow the `tools/validation/identity/run-identity-release-gate.mjs` `spawnSync` orchestration precedent.
- Assert the existing build chain instead of reinventing it.
- Keep native, heavy, or platform-specific builds opt-in.
- Produce JSON evidence suitable for F17.5 release evidence bundling.

F17.2 is not a new sync-feature phase. It should not change Library Sync runtime behavior, SQLite schema, release tooling behavior, or package contents during the contract phase.

## Default Vs Optional Checks

Default Node-safe checks:

- Parse migration source and verify `v1` through `v13` are sequential with no gaps or duplicate migration versions.
- Verify `v13` guarded `folder_bindings` trigger DDL remains present.
- Verify Studio asset pack / production extension build path is present and callable.
- Verify built-vs-source comparison support through `pack-studio.mjs` and `compareArchiveWorkbenchToSource` when available.
- Verify `prepare-dist.mjs` stale-build guard and filename/reference rewrite rules remain present.
- Verify version consistency across bundle-relevant app files.
- Verify userscript `@version` is checked on its own axis and is not conflated with app semver.
- Verify artifact naming conventions are documented and inspected when artifacts exist.
- Verify default checks do not run Tauri, EAS, or heavy native builds.

Optional checks:

- `--cargo`: run Rust compile / migration-carrier check.
- `--tauri-build`: run full desktop bundle build.
- `--artifacts`: inspect existing generated artifacts.
- Mobile/EAS: manual only unless a later phase adds a safe dedicated runner.

## Build-Vs-Inspect Decision

The default F17.2 validator should run or inspect Node-runnable build steps only. It should not run full Tauri, Rust, or EAS builds by default.

Default behavior:

- Source-parse build and migration carriers.
- Run lightweight Node checks where they are already safe and deterministic.
- Inspect artifacts only if present or explicitly requested.
- Avoid generating large platform-specific outputs unless a flag requests it.

Heavy/native behavior:

- `--cargo` may run `cargo check`.
- `--tauri-build` may run `npm run tauri:build` from `apps/studio/desktop/`.
- `--artifacts` may inspect existing output directories.
- Mobile/EAS remains a manual production readiness item unless a later dedicated mobile release validator is added.

## Version-Stamp Plan

Canonical app/bundle version sources for Desktop Studio:

- `apps/studio/desktop/package.json`
- `apps/studio/desktop/src-tauri/tauri.conf.json`

Current state:

- `apps/studio/desktop/package.json` version is `0.1.0`.
- `apps/studio/desktop/src-tauri/tauri.conf.json` version is `0.1.0`.
- Root `package.json` has no app release version and should be treated as workspace-root tooling metadata, not the Desktop app semver source.

F17.2.b should document this explicitly in validator output. If a future release wants root versioning, that should be fixed as a separate release-process decision, not inferred silently.

Userscript `@version` is a separate axis. App semver and userscript versions must not be conflated.

## Migration Inclusion Checks

SQLite migrations are compiled into the Rust binary through `studio_migrations()` in `apps/studio/desktop/src-tauri/src/lib.rs`.

F17.2.b should verify:

- `studio_migrations()` exists.
- Migration versions `v1` through `v13` appear in order.
- No migration version is duplicated.
- No migration version is skipped.
- `v13` guarded `folder_bindings` trigger strings remain present.
- `f16_folder_bindings_trigger_guard` remains present.
- Trigger definitions for guarded insert, update, and delete remain present.
- Optional `--cargo` confirms the Rust migration carrier still compiles.

## F17.5 Evidence

F17.5 should capture JSON evidence from `validate:build --json`, including:

- Migration count/order result.
- `v13` trigger presence.
- Studio pack / production build result.
- Built-vs-source comparison result.
- `prepare-dist` stale-build guard result.
- `prepare-dist` rewrite/sanitization guard result.
- Version consistency table.
- Artifact naming result.
- Optional check state and pass/fail status.

The evidence should distinguish default checks from opt-in checks so that omitted native/heavy builds are explicit rather than silently assumed.

## Risks And Mitigations

| Risk | Mitigation |
|---|---|
| Tauri build accidentally runs by default | Default validator is Node-safe; `--tauri-build` required |
| Stale Studio bundle is silently shipped | Validate `prepare-dist.mjs` stale-build guard and pack comparison hooks |
| Migration drift | Parse `studio_migrations()` for sequential `v1` through `v13` and required `v13` trigger DDL |
| Version mismatch | Compare Desktop package version and Tauri config version |
| Userscript/app version conflation | Treat userscript `@version` as a separate axis |
| Chrome/Desktop divergence | Validate Studio pack path, prod extension build path, and Desktop dist staging contract |
| Mobile/EAS untested | Keep mobile/EAS manual unless a dedicated future runner is added |
| Build outputs written to gitignored repo paths | Keep default checks lightweight and document artifact inspection with `--artifacts` |
