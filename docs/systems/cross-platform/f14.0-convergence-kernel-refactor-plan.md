# F14.0 - Convergence kernel refactor plan

> **Docs-only architecture plan.** F14.0 introduces no runtime code, no kernel
> implementation, no domain refactor, no label/category convergence, no remote
> apply, no automatic merge, and no mobile write-back.

F10 through F13 proved the convergence model across five folder-related
domains:

- color
- rename
- move
- delete
- folder binding add

Those lanes intentionally favored safety and explicitness over reuse. F14
exists to extract the repeated low-level primitives into a shared convergence
kernel before adding labels, categories, chats, snapshots, or captures.

## 1. Why F14 is needed

The current convergence system is safe enough for manual Desktop-local dogfood,
but it is expensive to extend. Each new domain currently tends to copy:

- canonical JSON and hashing helpers
- peer envelope validation
- forever-no privacy scans
- append-only ledger normalization
- blocker/warning code handling
- result shape helpers
- replay/dedupe checks
- proof harness scaffolding
- UI status reducers

That duplication was acceptable while proving the safety model one domain at a
time. It becomes risky before F15 because labels/categories and later chats,
snapshots, and captures will multiply the same patterns across more entity
types. Without a shared kernel, future lanes will drift in blocker vocabulary,
hash semantics, privacy guards, and ledger behavior.

F14 is therefore a behavior-preserving refactor track. It reduces drift without
expanding authority.

## 2. Current state

### Domain-specific convergence lanes

Each completed domain has its own lane:

| Domain | Main shape | Current status |
|---|---|---|
| Color | planner entry -> preflight -> local color apply -> applyEvent -> bookkeeping -> UI | Complete |
| Rename | materialization -> preflight -> proposal candidate -> local rename -> applyEvent -> bookkeeping -> proof/UI | Complete |
| Move | materialization -> preflight -> proposal candidate -> local move -> applyEvent -> bookkeeping -> proof/UI | Complete |

| Binding | identity/cardinality -> materialization -> preflight -> proposal candidate -> reviewed bind add -> applyEvent -> bookkeeping -> proof/UI | Complete |

### Shared ledgers

The platform already has shared ledger concepts:

- relay outbox
- relay inbox
- relay index
- publication ledger
- convergence watermarks
- consumed-operation ledger

These are conceptually shared, but each domain still contains repeated code for
calling them and interpreting their results.

The repeated primitives are deliberate extraction candidates:

## 3. Kernel scope

The convergence kernel should provide shared primitives only. It must not own
domain policy.

| Primitive group | Purpose |
|---|---|
| Canonical JSON | Stable key ordering and canonical serialization. |
| Hashing | Web Crypto SHA-256, hash validation, state-hash helpers, UUID helpers if kept local. |
| Peer envelope validation | Validate F2/F10 peer envelopes and derive redacted peer IDs. |
| Ledger append/list | Shared append-only storage normalization, row validation scaffolds, immutable append helpers. |
| Privacy scans | Shared forever-no field scanning with domain-specific optional allowlists. |
| Result shaping | Consistent `ok`, `blockers`, `warnings`, counts, rows, and schema handling. |
| Blocker/warning codes | Shared add/dedupe helpers and vocabulary drift checks. |
| Status reducers | Derive current lifecycle state from rows without mutating rows. |
| Replay/dedupe helpers | Shared eventDigest/dedupeKey duplicate detection, stale/expired helpers, replay summaries. |

The kernel should be boring: small pure helpers first, then storage helpers,
then reducers. Domain-specific gates remain in domain modules.

## 4. Non-goals

F14 must not introduce:

- automatic merge
- remote apply
- mobile write-back
- WebDAV authority
- hidden polling
- automatic sync loops
- new domain behavior
- relaxed F5/F6/F7 gates
- changes to applyEvent semantics
- changes to domain safety gates
- label/category/chats/snapshots/captures convergence

F14.0 itself is documentation only.

## 5. Refactor safety rules

| Rule | Requirement |
|---|---|
| Docs first | Every extraction group gets a scoped plan before code moves. |
| Preserve behavior | Extraction must not change runtime output shape, blocker codes, ledger schemas, or apply behavior. |
| One primitive group at a time | Do not extract hashing, ledger IO, privacy scans, and reducers in one commit. |
| Validator parity | Existing validators must pass before and after each extraction. |
| Domain parity | Existing runtime proof commands must remain valid. |
| No widened authority | Shared helpers cannot make Chrome/mobile/native more capable. |
| No domain rewrite first | Do not rewrite a domain lane until the target primitive has been extracted and proven. |
| Redaction unchanged | Privacy scans must get stricter or equal, never looser. |
| Explicit imports/loading | Kernel modules must load before consumers and must no-op safely outside Tauri where appropriate. |
| Rollback-friendly commits | Each extraction must be independently revertible. |

Candidate modules:

| `peer-envelope.tauri.js` | Peer envelope construction, hashing, validation, peer-id derivation. |

| `status-reducer.tauri.js` | Reducers for candidate/publication/inbox/outbox/consumed/watermark current status. |

Initial namespace suggestion:

```txt
H2O.Desktop.Sync.kernel
```

Kernel modules should expose functions, not mutate domain state. Storage helpers
may write only when explicitly called by an existing ledger module.

## 7. Migration sequence

### F14.1 - Canonical hashing, JSON, and privacy

Extract:

- canonical JSON ordering
- SHA-256 helpers
- hash validators
- forever-no scans
- basic string/object helpers only if needed

Acceptance:

- No ledger or domain behavior changes.
- Existing `node --check`, cross-platform repo scan, envelope validator, and
  F7 hash parity validator pass.
- One domain module can adopt the helpers as a proof without changing output.

### F14.2 - Ledger IO

Extract:

- storage `get/set`
- append-only ledger normalization
- immutable row append helpers
- list helpers
- row summary scaffolds

Acceptance:

- Apply, consumed, watermark, publication, inbox/outbox ledger schemas remain
  unchanged.
- Dedupe behavior remains unchanged.
- Existing ledger APIs remain backward-compatible.

### F14.3 - Blocker and result helpers

Extract:

- blocker/warning add/dedupe
- code list normalization
- standard `failure()` / `baseResult()` patterns
- shared schema-safe result builders

Acceptance:

- Existing blocker codes remain stable.
- Validator output remains unchanged.
- No domain starts swallowing blockers as warnings.

### F14.4 - Lifecycle and current-state reducers

Add read-only reducers over:

- proposal candidate ledger
- conflict candidate ledger
- publication ledger
- relay outbox
- relay inbox
- consumed-operation ledger
- convergence watermarks

Acceptance:

- Reducers are read-only.
- They produce status summaries only.
- They do not enqueue, publish, apply, or advance watermarks.

### F14.5 - Reconciliation diagnostics

Add diagnostics for partial lifecycle states:

- applied but no applyEvent
- applyEvent but no consumed row
- consumed row but no watermark
- published but not in outbox
- outbox pending too long
- inbox pending-review too long
- duplicate/replay rows requiring operator cleanup

Acceptance:

- Diagnostics only.
- No repair or auto-cleanup.
- Output is redacted and counts-first.

### F14.6 - Deterministic proof fixture harness

Build deterministic proof fixtures that do not require mutating the live dogfood
workspace by default.

Acceptance:

- Proofs can run in fixture mode.
- Live mode remains explicit/operator-approved.
- Negative proofs are deterministic rather than dependent on existing tombstone
  or duplicate state.

## 8. Risk analysis

| Risk | Mitigation |
|---|---|
| Behavior drift | Extract one primitive group at a time and compare outputs before/after. |
| Accidental authority widening | Kernel helpers must not call apply, publish, enqueue, WebDAV, or mobile APIs. |
| Hidden coupling between domains | Keep domain policy in domain modules; kernel provides mechanics only. |
| Validator false negatives | Add focused parity checks for migrated helpers, not only broad scans. |
| Privacy regression | Central privacy scan must be stricter-or-equal and run before returns. |
| Ledger schema drift | Ledger helpers must preserve existing schemas and row fields exactly. |
| Proof gaps | Add fixture-mode proofs before extending to new domains. |
| Load-order regressions | Kernel modules must be registered before domain modules in `studio.html` and `pack-studio.mjs`. |
| Over-abstraction | Avoid a generic convergence engine until reducers and proofs are proven. |
| Large rollback blast radius | Small commits and no multi-domain rewrites in one phase. |

## 9. Acceptance criteria for each extraction

Every F14 extraction must satisfy:

- no package/lockfile changes
- no mobile write-back
- no remote apply
- no WebDAV behavior change
- no automatic merge
- no new domain behavior
- `git diff --check`
- `node --check` for changed JS files
- cross-platform repo scan passes
- cross-platform envelope validator passes
- F7 folder metadata hash parity validator passes
- affected runtime proof commands still have the same public API
- existing ledger keys and schemas remain unchanged unless a later migration doc
  explicitly authorizes a versioned migration

For behavior-preserving refactors, the acceptance bar is not "tests pass only."
The public diagnostic/apply/proof API shape must remain stable.

## 10. How F14 prepares F15

F15 should add labels/categories convergence only after F14 provides shared
kernel primitives. Labels and categories are structurally similar to a subset of
the completed lanes:

- identity/cardinality resembles binding
- rename resembles folder rename
- color/style metadata resembles color convergence
- assignment policies resemble binding cardinality
- delete requires F5-like review if destructive

With a kernel, F15 can implement labels/categories by defining domain adapters:

```txt
identity -> materialization -> preflight -> proposal -> apply -> applyEvent -> bookkeeping -> proof -> UI
```

The domain adapter should own policy. The kernel should supply hashing, ledger,
privacy, blocker, peer, replay, and reducer mechanics.

Chats, snapshots, and captures should wait longer:

- chats may include content/privacy and richer ownership
- snapshots imply restore/recovery semantics
- captures should remain evidence-first until content redaction and retention
  policy are stronger

## 11. MUST / SHOULD / DEFER

### MUST

- Keep F14.0 docs-only.
- Extract mechanics, not policy.
- Preserve runtime behavior and public APIs during F14.1-F14.3.
- Keep Desktop as the only apply authority.
- Keep `applyEvent` evidence-only.
- Keep all domain safety gates intact.
- Keep privacy scans redacted and stricter-or-equal.
- Keep each extraction independently revertible.

### SHOULD

- Add small parity fixtures for canonical JSON/hash/privacy helpers.
- Add a kernel namespace with explicit install/version markers.
- Add current-state reducers before new domain convergence.
- Add reconciliation diagnostics before automatic cleanup or repair.
- Consolidate UI status reducers after ledger reducers exist.
- Document each adopted helper in the consuming module's phase note.

### DEFER

- Generic convergence engine.
- Labels/categories implementation.
- Chats/snapshots/captures mutation.
- Automatic sync.
- Automatic merge.
- Remote apply.
- Mobile write-back.
- Ledger schema migrations.
- Cleanup/compaction of append-only ledgers.

## 12. F14.0 status

F14.0 defines the refactor plan only. No kernel files exist yet, and no domain
lane behavior is changed.
