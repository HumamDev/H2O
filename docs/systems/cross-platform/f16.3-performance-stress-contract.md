# F16.3 - Performance Stress Contract

Status: F16.3.a docs-only contract.

F16.2 is closed. F16.3 defines the performance and large-scale stress proof contract for F15/F16 Library Sync without changing runtime behavior.

## 1. Architecture Verdict

F16.3 is a synthetic-data stress harness measuring real modules. It is not a production load test.

The stress proof should exercise real available modules and kernels such as:

- Catalog canonicalizers.
- Binding canonicalizers.
- Binding duplicate-check paths.
- `evaluateLibraryRuntimeConflict`.
- `classifyLibraryBulkRuntimeConflictRows`.
- Bulk plan/chunking with injected executor.
- Cache-refresh shaping.
- Replay-defense kernels.

F16.3 proves bounded runtime behavior and catches algorithmic degradation in deterministic proof conditions. It does not certify real SQLite throughput, Tauri IPC latency, relay transport latency, or real UI rendering under 10k chats. Those belong to production and release-gate work.

## 4. Stress Data Model

F16.3 should use a seeded hash-only `StressDataSet`:

```js
{
  seed,
  tier,
  chats,
  catalogs,
  bindings,
  bulkRows,
  replayQueue,
  plantedAnomalies
}
```

Field requirements:

| Field | Contract |
|---|---|
| `seed` | Deterministic seed reported in output |
| `tier` | `lightweight` or `heavy` |
| `chats` | Hash-only chat subject fixtures |
| `catalogs` | Hash-only label/tag/category fixtures |
| `bindings` | Hash-only binding edge fixtures |
| `bulkRows` | Hash-only bulk bundle rows |
| `replayQueue` | Hash-only replay candidates |
| `plantedAnomalies` | Deterministic anomaly rows expected to be detected |

The generator should emit hashes directly. Raw human-readable names, titles, paths, and IDs should not exist in fixtures.

## 5. Scale Matrix

| Dimension | Lightweight default | Heavy env-gated |
|---|---:|---:|
| Chats | 1,000 | 10,000 |
| Labels/tags/categories | 40/40/40 | 100/100/100 |
| Bindings | 1,000 | 10,000 |
| Bulk rows | 500 | 5,000 |
| Cache refresh batch | 250 | 2,500 |
| Replay queue depth | 500 | 5,000 |
| Planted anomalies | About 2% | About 2% |

Heavy mode must be opt-in only, for example:

```text
F16_STRESS_HEAVY=1
```

F16.3.b validators should run the lightweight tier only. Heavy mode can be manual, nightly, or release-gate-adjacent after F16.3.c.

## 6. Metrics

The stress proof should collect:

- Total duration.
- Max phase duration.
- Catalog lookup duration.
- Binding duplicate-check duration.
- Runtime conflict gate duration.
- Bulk classification duration.
- Cache refresh shaping duration.
- Replay defense duration.
- Scaling ratio.
- Object counts and residual growth.
- Planted anomalies versus detected anomalies.
- Seed.
- Tier.
- Budget exceeded warnings.
- Hard ceiling violations.

Phase records should be compact and hash-safe. They should report counts, durations, anomaly results, and blocker/warning codes only.

## 7. Budget And Closure Rules

Soft budget overruns should warn only. Closure blockers should include:

- Anomaly misses.
- Scaling-ratio blowups, for example `>= 25x`.
- Hard-ceiling timing violations, for example `> 10x` soft budget.
- Residual object growth.
- Privacy leaks.
- Side-effect flag flips.

Budget examples for F16.3.b should be intentionally conservative and environment-tolerant. The goal is to catch severe algorithmic regressions without making normal validators flaky.

## 9. Validator Strategy

F16.3.b should create:

```text
tools/validation/sync/validate-f16-library-performance-stress.mjs
```

The validator should assert:

- Stress module exists.
- APIs and markers exist.
- Schema string exists.
- Structural output fields exist.
- Node smoke run uses lightweight tier only.
- Heavy mode exists but is not run by default.
- Real module API references exist.
- Planted anomalies are checked.
- Privacy guardrails exist.
- Side-effect guardrails exist.
- No 10k-scale stress runs in default validation.

F16.3.d should harden closure only after the proof is stable. Closure requires lightweight stress proof only. Heavy stress remains optional/manual/CI-nightly.

## 10. Privacy And Side Effects

Stress output must be hash-only.

Forbidden raw values:

- Raw chat IDs.
- Raw catalog IDs.
- Raw folder IDs.
- Names, titles, and folder names.
- Paths.
- URLs or tokens.
- Content, messages, or attachments.
- `category_id`.
- `chats.category_id`.
- `folder_id`.
- `chat_id`.

Side-effect requirements:

| Side effect | Required value |
|---|---|
| Real business table writes | `false` |
| Real bookkeeping writes | `false` |
| Native calls | `false` |
| F5 calls | `false` |
| Publication mutation | `false` |
| Relay mutation | `false` |
| Outbox mutation | `false` |

Injected executors are allowed only where needed to measure bulk plan/chunking behavior. They must be marked as injected and must not write real business tables.

## 11. Risks And Mitigations

| Risk | Mitigation |
|---|---|
| Slow validators | Keep default tier lightweight and reserve heavy mode for `F16_STRESS_HEAVY=1` |
| Memory bloat | Track residual object growth and object counts |
| Proof/runtime divergence | Call real modules and kernels where available |
| False confidence from synthetic data | Scope F16.3 as algorithmic stress, not production load certification |
| Timer noise | Use soft warnings for normal budget overruns and blockers only for hard ceilings |
| Bulk stress hiding conflict bugs | Plant deterministic anomalies and require detection counts |
| Operator report overload | Return compact summaries and phase rows, not full fixture dumps |

## 13. Explicit Deferrals

F16.3.a does not implement:

- Runtime stress proof module.
- Validator.
- Closure hardening.
- Loader or pack wiring.
- Heavy stress mode.
- SQLite trigger changes.
- Store mutations.
- Native/F5 execution.
- Publication, relay, or outbox behavior.
- Real UI rendering certification.
- Real SQLite throughput certification.

F16.3.b is the next phase and should implement the lightweight performance stress proof harness plus validator.

## 14. Authoritative Summary

F16.3 proves deterministic algorithmic performance and large-scale stress behavior using synthetic hash-only data and real available modules. It must catch severe scaling regressions, anomaly misses, residual object growth, privacy leaks, and side-effect boundary violations. It does not certify real production throughput or transport latency. Lightweight stress is the validator and closure target; heavy stress is opt-in only.
