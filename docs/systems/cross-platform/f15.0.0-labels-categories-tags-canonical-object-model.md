## 2. Recommended Model

### 2.1 Two F15 lanes

**`library.catalog` lane** owns the catalog entries themselves — Label, Tag, Category — and all operations on them (create, rename, recolor, archive, tombstone, restore). The lane is discriminated internally by a `catalogKind` field ∈ `{label, tag, category}`. One canonicalizer, one diagnostics module, one preflight, one set of proposal generators (one per operation type), one handoff preview, one receipt, one bookkeeping ledger, one proof, one UI.

**`library.binding` lane** owns binding edges — chat↔label, chat↔tag, chat↔category, tag↔category, and (after F15.5) chat↔folder. The lane is discriminated internally by a `bindingKind` field ∈ `{chat-label, chat-tag, chat-category, tag-category, chat-folder}`. One canonicalizer, one diagnostics module, one preflight, one set of proposal generators (bind, unbind), one handoff preview, one receipt, one bookkeeping ledger, one proof, one UI.

The two lanes are siblings, not nested. They cross-reference each other only via opaque `subjectId` hashes — never by raw catalog kind, raw name, or raw identifier.

### 2.2 Why hybrid beats the alternatives

The three-sibling-lanes option (label / tag / category lanes) would have produced ~30 modules with massive duplication — catalog operations have the same shape regardless of catalogKind (the diff is just a discriminator field and a few content fields). One-unified-binding-lane would have collapsed catalog operations into "bind to a null right-side" pseudo-bindings, which obscures the semantic difference between editing a label's name and connecting a label to a chat. The two-lane hybrid keeps the semantic clarity (catalog ≠ binding) while sharing infrastructure within each lane.

### 2.3 Subject identity rules

**Catalog subject** (one per Label / Tag / Category catalog entry):

```
subjectId = sha256(canonicalJSON({
  subjectType: 'library.catalog',
  catalogKind,                        // 'label' | 'tag' | 'category'
  rawId,                              // local SQLite primary key / uuid
  perEnvelopeSalt                     // per-subject random salt at creation
}))
```

**Binding subject** (one per edge):

```
subjectId = sha256(canonicalJSON({
  subjectType: 'library.binding',
  bindingKind,                        // 'chat-label' | 'chat-tag' |
                                      // 'chat-category' | 'tag-category' |
                                      // 'chat-folder' (post-F15.5)
  leftSubjectId,                      // sha256 of left endpoint
  rightSubjectId,                     // sha256 of right endpoint
  perEnvelopeSalt
}))
```

The endpoints are normalized — for symmetric M:N relationships (tag-category), the convention is `leftSubjectId` < `rightSubjectId` lexically. For asymmetric edges (chat-label, chat-tag, chat-category, chat-folder), the convention is `left = chat, right = catalog`.

**Critical implication:** renaming a Label changes the Label's `revisionHash` but NOT its `subjectId`. Bindings that reference the Label do so by the Label's `subjectId`. Rename does not invalidate bindings. This is what makes the lane work without a multi-subject primitive.

## 3. Alternatives Considered

### 3.1 Lane shape alternatives

| Alternative | Modules | Pros | Cons | Verdict |
|---|---|---|---|---|
| Three sibling lanes (label / tag / category) | ~30 | Most semantically explicit per domain | Heavy duplication — three near-identical canonicalizers, preflights, etc. Maintenance burden compounds with every F15.x change | **Rejected** |
| One unified library-binding lane | ~10 | Smallest surface | Collapses catalog ops into "null-right-side bindings", semantically unclear, forces single shape onto distinct concepts | **Rejected** |
| **Two-lane hybrid (catalog + binding)** | **~20** | Clean semantic boundary; catalog ops and binding ops have natural lane homes; intra-lane discriminators handle the three catalog kinds + five binding kinds without N-fold duplication | Slightly more modules than the unified option; cross-lane lineage queries cross two ledgers | **Recommended** |
| Five lanes (one per catalog × kind, plus binding) | ~40 | Maximum specificity | Disproportionate to scope — labels/tags/categories share 95% of their canonical operations | **Rejected** |

### 3.2 Subject identity alternatives

| Alternative | Pros | Cons | Verdict |
|---|---|---|---|
| Subject = catalog entity (Label is the subject; bindings are payload-side lists) | One proposal per logical "label state change" | Multi-subject payload (the bindings list); fights the F14.2 one-subject-per-proposal rule; rename would have to enumerate all binding lineages | **Rejected** |
| Subject = chat (chat metadata gains `labelIds`, `tagIds`, `categoryId`) | Mirror chat lane behaviorally | Cross-domain coupling; chat lane's privacy floor would have to expand; label rename touches N chats, which is multi-subject again | **Rejected** |
| **Subject = binding edge + catalog subject for catalog ops** | One subject per proposal; rename catalog is single-subject (bindings reference by id, unaffected) | Bindings are first-class subjects with their own lineage / watermark | **Recommended (final)** |
| Subject = batch envelope wrapping N edges | Batch UX is naturally one envelope | Requires new `shapeMultiSubjectOperation` primitive; F14 kernel does not support; breaks proposal=subject invariant | **Rejected** |

### 3.3 Lifecycle alternatives for catalog

| Alternative | Pros | Cons | Verdict |
|---|---|---|---|
| No lifecycle (delete is hard-delete only) | Simplest | No archive; no operator-recoverable delete; F5 review queue contract unused | **Rejected** |
| Chat-style lifecycle (active / archived / retained / expired / tombstoned) | Reuses F14.5.2 amendment patterns exactly; F5 review queue already exists | More state than minimal | **Recommended** |
| Snapshot-style lifecycle (adds 5-state with explicit retention) | Same as above | No additional value over chat-style for catalog entries; recovery contract from F14.4.12 doesn't apply (catalog entries don't have external source of truth) | **Rejected** |
| Binary lifecycle (active / deleted) | Minimal | No archive UX (operators want hidden-but-restorable labels); no F5 review for high-impact deletes | **Rejected** |

### 3.4 Lifecycle alternatives for binding

| Alternative | Pros | Cons | Verdict |
|---|---|---|---|
| **Binary (bound / unbound; rebind = new subjectId)** | Simplest; matches operator mental model (a binding either exists or it doesn't); no retention window needed | Old binding history is preserved only in the bookkeeping ledger, not as a reachable subject | **Recommended** |
| Full lifecycle (active / archived / retained / expired / tombstoned) | Symmetric with catalog | Disproportionate — bindings are throwaway edges; restore-a-binding = just rebind | **Rejected** |
| Soft-delete only (active / removed; restore is permitted) | Avoids re-creating duplicate bindings | Same subjectId for bound-then-unbound-then-rebound — confusing per-subject history | **Rejected** |

## 5. Canonical Object Schemas (Conceptual)

All shapes below are conceptual — actual field types/widths to be ratified during F15.0.1 schema-doc phase. Schema version literal: `'h2o.library.<kind>.v1'`.

### 5.1 Catalog canonical envelope (`library.catalog`)

```
{
  objectType:      'libraryCatalog',
  subjectType:     'library.catalog',
  subjectId:       sha256,
  revisionHash:    sha256(over §5.2),
  catalogKind:     'label' | 'tag' | 'category',
  nameHash:        sha256 of raw name,           // raw name forbidden in envelope
  colorHash:       sha256 of '#RRGGBB' or null,
  displayOrder:    integer (coarsened),
  iconHint:        enum-string or null,
  originAccountIdHash: sha256,
  schemaVersion:   'h2o.library.catalog.v1',
  lifecycleState:  'active' | 'archived' | 'retained' | 'expired' | 'tombstoned',
  archived:        boolean (derived),
  tombstoned:      boolean (derived),
  archivedAtIso:   ISO, hour-coarsened or null,
  lifecycleChangedAtIso: ISO, hour-coarsened,
  retentionExpiresAtIso: ISO, hour-coarsened, optional,
  sourceTag:       enum: 'desktop' | 'mv3-import' | 'bundle-import',
  sourceTagHash:   sha256,
  observedAtIso:   ISO, hour-coarsened
}
```

### 5.2 Catalog `revisionHash` composition

```
revisionHash = sha256(canonicalJSON({
  lifecycleState,
  archived, tombstoned,
  retentionExpiresAtIso,
  catalogKind,
  nameHash,
  colorHash,
  displayOrder,
  iconHint,
  originAccountIdHash,
  schemaVersion,
  sourceTagHash
}))
```

`subjectId` and `observedAtIso` are explicitly excluded (identity vs transient).

### 5.3 Binding canonical envelope (`library.binding`)

```
{
  objectType:      'libraryBinding',
  subjectType:     'library.binding',
  subjectId:       sha256,
  revisionHash:    sha256(over §5.4),
  bindingKind:     'chat-label' | 'chat-tag' | 'chat-category' |
                   'tag-category' | 'chat-folder',
  leftSubjectId:   sha256 (chat subjectId for chat-* edges; smaller-by-lex for symmetric edges),
  rightSubjectId:  sha256 (catalog subjectId for chat-* edges; larger-by-lex for symmetric edges),
  leftSubjectType: 'chat.metadata' | 'library.catalog',
  rightSubjectType:'library.catalog',
  originAccountIdHash: sha256,
  schemaVersion:   'h2o.library.binding.v1',
  bindingState:    'bound' | 'unbound',
  boundAtIso:      ISO, hour-coarsened,
  unboundAtIso:    ISO, hour-coarsened, optional,
  sourceTag:       enum,
  sourceTagHash:   sha256,
  observedAtIso:   ISO, hour-coarsened
}
```

### 5.4 Binding `revisionHash` composition

```
revisionHash = sha256(canonicalJSON({
  bindingState,
  bindingKind,
  leftSubjectId, rightSubjectId,
  leftSubjectType, rightSubjectType,
  originAccountIdHash,
  schemaVersion,
  sourceTagHash
}))
```

### 5.5 Catalog uniqueness invariants (enforced at preflight)

- **Per (originAccountIdHash, catalogKind, nameHash)**: at most one `active` catalog subject. Re-creating a label with the same name as an existing-active label blocks with `library-catalog-name-already-bound`. (Hashed-name check; raw name never crosses the boundary.)

### 5.6 Binding uniqueness invariants (enforced at preflight)

- **Per (subjectType, bindingKind, leftSubjectId, rightSubjectId)**: at most one `bound` binding subject. Re-bind blocks with `library-binding-already-bound`.
- **Chat-category special case**: per `leftSubjectId` (the chat), at most one `bound` binding with `bindingKind='chat-category'`. Adding a new category to a chat that already has one is `library-binding-chat-already-categorized` → operator must explicitly unbind the old one (UI may collapse this into one user-visible action, but the lane sees two proposals: unbind + bind).
- **Chat-folder special case** (post-F15.5): same as chat-category — at most one active chat-folder binding per chat.

## 6. Operation Model

### 6.1 Catalog operations and their proposals

| Operation | Proposal name | Operation intent | Lifecycle effect |
|---|---|---|---|
| `create` | `library-catalog-created` | `create` | (initial → `active`) |
| `rename` | `library-catalog-rename-proposed` | `update` | (no lifecycle change; nameHash changes; revisionHash changes; binding subjects unaffected because they reference by subjectId) |
| `recolor` | `library-catalog-recolor-proposed` | `update` | (no lifecycle change; colorHash changes; revisionHash changes) |
| `archive` | `library-catalog-archive-proposed` | `update` | `active → archived` |
| `restore-from-archived` | `library-catalog-restore-proposed` | `update` | `archived → active` |
| `tombstone` | `library-catalog-tombstone-proposed` | `update` | `active \| archived → retained` (enters F5 review queue) |
| `restore-from-retained` | `library-catalog-restore-proposed` | `update` | `retained → active` (must be within retention window) |
| (automatic) | `library-catalog-retention-expired` | (auto, no proposal) | `retained → expired` |
| (automatic) | `library-catalog-terminal-sealed` | (auto, no proposal) | `expired → tombstoned` |

The retention-window architecture is **byte-identical** to the F14.5.2 snapshot amendment: tombstone proposals land in `retained`, not directly in `tombstoned`. F5 review queue per F14.5.5 mediates operator approve-seal / approve-restore decisions.

### 6.2 Binding operations and their proposals

| Operation | Proposal name | Operation intent | Lifecycle effect |
|---|---|---|---|
| `bind` | `library-binding-bind-proposed` | `create` | (initial → `bound`) |
| `unbind` | `library-binding-unbind-proposed` | `update` | `bound → unbound` (terminal — rebind creates new subjectId) |

There is no `bind-reviewed` or `unbind-reviewed` distinction — bindings are not F5-gated (low-impact, easily reversible by rebind).

### 6.3 Per-operation preflight gates (mirror chat lane structure)

**Catalog preflight gates:**
- `operationAllowed` (operation ∈ catalog operations list)
- `canonicalizationOk`
- `crossAccountSafe` (originAccountIdHash == localAccountIdHash)
- `nativeOwnerReachable`
- `mirrorNotStale`
- `tombstoneClear` (for create / rename / recolor on a non-tombstoned catalog)
- `forbiddenFieldsClear` (kernel.scanDomainForbiddenFields('library.catalog', ...))
- `consumedSafe`
- `watermarkSafe`
- `replaySafe`
- `expectedTargetValid`
- `uniqueNameClear` (for create / rename — no other active subject shares the nameHash)
- `lifecycleTransitionAllowed` (kernel.validateLifecycleTransition; fromState → toState legal per §6.1)

**Binding preflight gates:**
- `operationAllowed`
- `canonicalizationOk`
- `crossAccountSafe`
- `nativeOwnerReachable`
- `mirrorNotStale`
- `forbiddenFieldsClear` (kernel.scanDomainForbiddenFields('library.binding', ...))
- `consumedSafe`
- `watermarkSafe`
- `replaySafe`
- `expectedTargetValid`
- `bindingUniqueClear` (per §5.6 uniqueness rules — for bind operations)
- `bindingExists` (for unbind operations — target binding must currently be in `bound` state)
- `endpointsResolvable` (both leftSubjectId and rightSubjectId exist; their canonical subjects are not tombstoned)

## 7. Lifecycle Model

### 7.1 Catalog lifecycle state machine (chat-style 5-state)

```
(none) ──create──► active ──archive──► archived ──restore──► active
                    │                     │
                    │ tombstone            │ tombstone
                    ▼                     ▼
                  retained ◄───────────── retained
                    │
                    │ restore (within retention window)
                    ▼
                  active
                    │
                    │ (auto, retention expiry)
                    ▼
                  expired
                    │
                    │ (auto, terminal seal)
                    ▼
                  tombstoned (terminal)
```

Transition matrix exactly mirrors F14.5.2 snapshot amendment §3.2. Restore-from-retained is gated by the F5 review queue; restore-from-archived is not gated (no time window).

### 7.2 Binding lifecycle state machine (binary)

```
(none) ──bind──► bound ──unbind──► unbound (terminal)
```

Unbinding does NOT delete the binding's bookkeeping ledger row; the subject remains addressable for audit. Rebinding the same endpoints produces a NEW subjectId (different `perEnvelopeSalt`). The audit chain via `originBindingSubjectIdHash` (lineage marker on the new binding) is optional and operator-driven — not part of canonical identity.

### 7.3 Catalog lifecycle interaction with bindings

When a catalog subject enters `retained` or `expired` or `tombstoned`:
- Bindings referencing it remain in their own state (`bound` or `unbound`).
- The binding preflight gate `endpointsResolvable` will block NEW bind proposals against a non-active catalog endpoint.
- Existing `bound` bindings whose right-side catalog enters `tombstoned` become **orphaned** — the binding still exists but its endpoint is terminal. F15.x diagnostics module surfaces orphaned bindings; cleanup is operator-driven via the bind/unbind UI.
- No cascade: tombstoning a catalog does NOT auto-unbind all its bindings. This is intentional — auto-cascade is high-blast-radius; explicit operator cleanup is safer.

## 8. Privacy Model

### 8.1 New kernel domain policies required

Two new policies registered in `kernel/privacy-scan.tauri.js` alongside `chat.metadata`, `capture.artifact`, `snapshot.conversation`:

**`library.catalog`** — structured like chat.metadata (ALWAYS + REDACTED tiers):

```
LIBRARY_CATALOG_ALWAYS_FORBIDDEN_FIELDS = [
  'content', 'body', 'text', 'description', 'rawDescription',
  'icon', 'iconBlob', 'iconBase64',  // raw icon data (if any)
  'attachments', 'files'
]

LIBRARY_CATALOG_REDACTED_FORBIDDEN_FIELDS = [
  'name', 'rawName', 'displayName', 'label', 'title',
  'color', 'rawColor',  // raw hex; only colorHash is allowed
  'rawId', 'labelId', 'tagId', 'categoryId',
  'accountId', 'rawAccountId', 'account_id',
  'userId', 'user_id', 'rawUserId',
  'createdBy', 'createdByRaw',
  'notes', 'rawNotes', 'operatorNotes',
  'customMetadata', 'rawMetadata'
]
```

**`library.binding`** — same two-tier shape:

```
LIBRARY_BINDING_ALWAYS_FORBIDDEN_FIELDS = [
  'content', 'body', 'text',
  'rawPayload', 'bindingPayload'
]

LIBRARY_BINDING_REDACTED_FORBIDDEN_FIELDS = [
  'name', 'rawName',                         // never carry catalog names in binding envelope
  'rawLeftId', 'rawRightId',                 // raw endpoint ids
  'chatId', 'chat_id',                       // raw chat ids
  'labelId', 'tagId', 'categoryId', 'folderId',
  'accountId', 'rawAccountId', 'account_id',
  'userId', 'user_id',
  'title', 'chatTitle', 'rawTitle',          // never carry chat titles
  'notes', 'rawNotes'
]
```

### 8.2 Hash-only identity rule

**No raw catalog name** ever crosses the envelope boundary. Only `nameHash` (sha256 of the raw name) is carried. The local SQLite store retains the raw name for display; envelopes never carry it. Operators see the raw name only via the local-only display layer.

**No raw color** ever crosses. Only `colorHash`. The display layer maps colorHash back to a display color via a local-only catalog read.

**No raw endpoint ids** in binding envelopes — only `leftSubjectId` and `rightSubjectId` (both sha256).

### 8.3 Defense-in-depth scan points

Same pattern as chat lane (F14.3) + snapshot lane (F14.4):

1. Input scan — raw catalog record from SQLite → canonicalizer
2. Canonical envelope scan — post-canonicalization
3. Diagnostic output scan
4. Preflight result scan
5. Proposal candidate envelope scan
6. Candidate ledger row scan
7. Handoff preview scan
8. ApplyEvent receipt scan
9. Bookkeeping row scan
10. F5 review queue ingest scan (catalog tombstone only)
11. UI render scan (operator-facing rows)

### 8.4 Cross-domain isolation

Library envelopes never reference chat metadata raw fields. Chat metadata envelopes never reference library raw fields. Cross-domain references happen exclusively through subjectId hashes (e.g., a chat-label binding carries `leftSubjectId = chat's subjectId`, never raw chat fields).

The existing F14.3.8 `scanDomainForbiddenFields(domainTag, target)` extraction is the single privacy primitive used by both library lanes — no library-side sibling scanner.

**No new kernel primitive is required.** The binding-as-subject model means every operation is single-subject; the F14.2 one-subject-per-proposal contract holds.

**Trim follow-up reaffirmed:** The F14.5.7 trim (`shapeAuditHistory`, `shapeProofSummary`, `shapeLifecycleHistory`, `shapeLifecycleResult`, `shapeOwnerHandoffResult`, `shapeAuditProofResult`) is unchanged — F15 does not need any of them either. History is computed from the F15 lane's own bookkeeping ledger; results use `createResult`.

**DEFER set from F14.5.0 — adoption test:** F15.x will exercise the remaining DEFER set (`shapeProofRecord`, `shapeProofMetadata`, `shapeLifecycleMetadata`, `shapePublication`, `shapePublicationMetadata`, `shapePublicationReceipt`). F14.6 execute lane already uses publication shapes; F15 adapters feed them. The library proof modules will adopt `shapeProofRecord` / `shapeProofMetadata` for the first time — this is the trial that decides whether those exports stay (ADOPT) or get trimmed in F15.7.

## 10. Execute Lane Integration Model

### 10.1 New execute adapters

Two new adapters land in `src-surfaces-base/studio/sync/execute/adapters/`:

**`library-catalog-execute-adapter.tauri.js`** (F15.6.a):
- Registers domain envelope shape for catalog candidates + receipts
- Maps catalog `revisionHash` + `lifecycleState` into `execute-envelope`'s adapter registry
- Routes catalog tombstone proposals through `execute-f5-broker` (F5 review path)
- Routes catalog create / rename / recolor / archive / restore-from-archived proposals through `execute-native-broker` (direct Native apply path)

**`library-binding-execute-adapter.tauri.js`** (F15.6.b):
- Registers binding envelope shape
- Routes ALL binding proposals (bind / unbind) through `execute-native-broker` only
- Does NOT register an F5 path (bindings are not F5-gated)

### 10.2 Operation flow through execute lane

```
operator action in F15 UI
  → F15 proposal candidate generator
  → F15 bookkeeping ledger (candidate row, status='generated')
  → F15 handoff preview
  → F15 applyEvent receipt builder
        ↓
        if catalog-tombstone:
          receipt's f5Handoff → execute-f5-broker
            → F5 review queue ingest (via F14.5.5.1 queue)
              → operator approve-seal / approve-restore (via F14.5.5.3 panel)
                → execute-f5-broker close
                  → execute-settlement-writer
                    → Native owner handoff (catalog tombstone apply)
                      → kernel.shapeTombstone emit
                        → consumed-op + watermark + bookkeeping writes
        else (everything else):
          receipt → execute-envelope
            → execute-preflight-gate
              → execute-native-broker
                → Native catalog/binding apply (SQLite mutation)
                  → execute-settlement-writer
                    → consumed-op + watermark + bookkeeping writes
                    → publication finalize (status='published')
```

### 10.3 Side-effect summary contract for F15 modules

All F15 lane modules outside the four execute mutation writers carry `sideEffectSummary` envelopes with **all eight flags `false`**:
- `publicationTouched: false`
- `relayTouched: false`
- `outboxTouched: false`
- `nativeCalled: false`
- `f5Touched: false` (or `f5Executed: false`)
- `watermarkWritten: false`
- `consumedOperationWritten: false`
- `applyExecuted: false`

The library lanes themselves never mutate Native, never write consumed-ops, never advance watermarks. All writes happen exclusively through `execute-settlement-writer` (F14.6.8), `execute-native-broker` (F14.6.6), `execute-f5-broker` (F14.6.7), and `execute-resume-on-boot` (F14.6.9). The F15 audit trail asserts this boundary; the F15 proof harness verifies it on every emit.

### 10.4 Watermark scope

- **Catalog watermarks**: scope = `(originAccountIdHash, subjectType='library.catalog')`. One watermark per catalog subject. Advances on every catalog state change.
- **Binding watermarks**: scope = `(originAccountIdHash, subjectType='library.binding')`. One watermark per binding subject. Advances on bind and unbind only.

These are separate watermark series from chat / snapshot / capture. The execute settlement writer routes by subjectType.

### 10.5 Consumed-op scope

Same per-subject scope as watermarks. Catalog tombstone uses `operation: 'library-catalog-tombstone-applied'`; binding bind uses `operation: 'library-binding-bind-applied'`; etc. The execute lane's existing consumed-op ledger handles all subject types uniformly.

## 11. F7 Migration Decision

**Decision: Coexist through F15.4. Absorb at F15.5. F7 modules quiesced (not deleted) for at least two release cycles post-F15.5.**

### 11.1 Why not refactor F7 onto F14.2 kernel shapes pre-F15?

The F14 review §4.6 risk noted that F7's 9 binding modules (5,280 LOC) pre-date F14.2 kernel extraction and would benefit from a kernel-shape rewire. The natural temptation is to do that rewire FIRST and only then build F15.

This phase rejects that path because:

1. F7 folder bindings have a stable user-visible contract that should not regress during library-feature evolution.
2. The kernel adoption pattern for bindings is itself novel — proving it on greenfield F15 library bindings is safer than retrofitting it onto the older folder-binding lane.
3. F7 already encodes the binding-as-subject model (subject = `sha256("folderBinding:" + chatSubjectId + ":" + folderSubjectId)`), so the conceptual migration target is clear — only the kernel-shape adoption needs rewiring.
4. Doing both at once doubles the risk surface and slows shipping of label/category/tag features that operators want now.

### 11.2 Migration timing

- **F15.0 → F15.4**: F7 unchanged. F15 builds the catalog + binding lanes greenfield. F7 folder bindings continue to operate on F13.0.x convention. F15 lanes and F7 lane coexist as separate convergence surfaces.
- **F15.5 — F7 absorption**: F7's `binding-*` modules are rewired to consume kernel shapes AND its `bindingKind` discriminator becomes a value (`'chat-folder'`) in `library.binding`'s already-shipped enum. F7 ceases to be a separate lane; folder-binding operations route through `library-binding-execute-adapter`. F7 modules become thin wrappers that delegate to `library.binding`.
- **F15.5+ deprecation window**: at least two release cycles, F7 modules remain as a fallback. Bug reports against F7 can still be fixed without forcing operators onto F15.5+.
- **F16 (post-F15)**: F7 modules deleted; only thin compat shims remain.

### 11.3 What F15.5 absorption requires

- F7 folder-binding subjectId formula changes from `sha256("folderBinding:" + chatSubjectId + ":" + folderSubjectId)` to the F15.0 binding formula. To avoid breaking existing folder-binding rows, F15.5 emits **migration shadow events** that bridge the two identity schemes. The F5 review queue is NOT involved (folder bindings are not F5-gated, same as other bindings).
- Privacy domain tag changes from F7's local-only forbidden field list to the kernel's `library.binding` policy.
- Watermark scope changes from per-folder-binding-subject to the unified library.binding watermark series.

This is significant work and intentionally pushed past the label / category / tag value-delivery phases.

### 11.4 What F15.5 does NOT do

- Does NOT delete F7 modules.
- Does NOT change the operator UX for folders.
- Does NOT change the folder catalog (folders are not Labels — they remain their own catalog. Only the binding side is unified.).

## 12. Risks and Mitigations

**12.1 Two-lane coupling complexity.** Catalog operations and binding operations interact (e.g., tombstoning a catalog orphans its bindings). The cross-lane interaction is real but bounded: only orphaning, not cascading. The diagnostics module in each lane surfaces cross-lane consequences via a `relatedSubjects` annotation.

*Mitigation*: F15.0.1 — explicit cross-lane interaction matrix doc enumerating every cross-lane consequence; F15.x proof harnesses verify the matrix.

**12.2 Existing `chats.category_id` column re-expression.** Moving category assignment from a SQL FK to a binding-edge model means existing row reads (`SELECT category_id FROM chats`) need a materialized cache derived from binding state. The cache is local-only, refreshed on binding apply. If the cache drifts from binding state, chat UI shows stale category assignment.

*Mitigation*: F15.0.2 — define the materialized cache contract (read-only mirror, invalidated and re-derived on every chat-category binding apply event). F15.x diagnostics module includes a "category cache drift" probe.

**12.3 F7 migration shadow events.** F15.5's identity bridging from F13.0 folder-binding subjects to F15.0 binding subjects emits shadow events that exist solely to translate between the two ID schemes. If those shadow events are mis-emitted, consumed-op ledger entries collide.

*Mitigation*: F15.5 ships with an isolated proof harness that exercises the bridge in both directions; F7 modules remain as fallback until the bridge is proved stable across two release cycles.

**12.4 Cross-install duplicate creates.** Two operators on different installs each create a `library.catalog` subject with the same `nameHash` and the same `originAccountIdHash`. Per the uniqueness invariant in §5.5, only one can be `active` — but both proposals may pass local preflight independently.

*Mitigation* (provisional, full contract in F15.6): execute-lane settlement detects the duplicate at apply time via the unique-name index in Native SQLite. The second-to-apply proposal blocks with `library-catalog-name-already-bound-at-apply`. The first wins; the second is surfaced to the operator who proposed it with a clear UI message and an option to rename-and-retry.

**12.5 Tag↔Category orphaning.** When a Category is tombstoned, its `tag-category` bindings become orphaned (per §7.3 no-cascade rule). If many such bindings accumulate, diagnostic noise grows.

*Mitigation*: F15.x diagnostics module surfaces orphan bindings with a cleanup affordance. UI hides orphaned bindings by default; operators see them only via an explicit "Show orphans" toggle.

**12.6 Cross-install conflict resolution full contract.** §11 is provisional. The full F15.6 contract must cover: rename collision (two installs rename the same catalog with different `nameHash` values), bind/unbind race (one install binds while another unbinds), recolor collision, tombstone vs restore race.

*Mitigation*: F15.6 is its own design phase. It produces a contract doc analogous to F14.5.5 (F5 queue contract), specifying exact blocker codes, retry semantics, and execute-lane behavior for each conflict shape. F15.x library lanes ship before F15.6 with the conservative behavior: replay-defense blocks all cross-install collisions, requiring operator-driven rebind / rename-retry.

**12.7 Color / display metadata stability.** Display order and color may change frequently as operators reorganize. Every change is a `recolor` or implicit display-order proposal, producing watermark advances. Watermark growth could become noisy.

*Mitigation*: F15.x diagnostics surfaces watermark growth per subject; UI batches successive recolor / display-order changes within a short window (e.g., 30 seconds) into a single proposal at emit time. Batching is a UI affordance, not a kernel concern.

**12.8 Auto-tagging vs operator-driven tagging.** Tags have a "suggestion pool" per the existing doc. If the suggestion pool auto-tags chats, those proposals must still flow through the F15.binding lane — not bypass it. Bypassing would create unobservable bindings.

*Mitigation*: F15.x explicitly forbids auto-tagging from writing to the tag bindings table directly. All auto-suggestions become candidate proposals in `library.binding` with `sourceTag = 'auto-suggested'`, requiring operator-driven publish-or-discard. The auto-suggestion engine never crosses the lane boundary.

**12.9 Existing labels store API (`bindChat`, `unbindChat`, `replaceForChat`) bypass.** The current `labels.tauri.js` store exposes direct write APIs that, if called, would write to SQLite without going through F15. This is a real bypass risk.

*Mitigation*: F15.0.3 — the existing store-layer write APIs are deprecated and rewired to be no-ops or to emit F15 binding proposals (operator-driven, not auto-apply). Read APIs (`listChats`, `listForChat`) remain functional and now read from the F15-maintained binding state plus the materialized cache.

F15.0.0 is **final** for §§1–11 and **provisional** only for §12.6 (cross-install conflict resolution, scoped to F15.6).
