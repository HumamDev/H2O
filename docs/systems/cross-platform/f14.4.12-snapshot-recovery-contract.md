# F14.4.12 - Snapshot recovery contract

> **Docs-only contract.** F14.4.12 introduces no runtime code, recovery
> executor, restore execution, publication, relay, apply, Native execution, F5
> execution, watermark write, consumed-operation write, or storage mutation.

F14.4 implemented the Snapshot convergence lane through canonicalization,
diagnostics, preflight, proposal candidates, owner-handoff previews,
applyEvent receipt previews, bookkeeping, proof, and UI. That lane is about
reviewing and proving lifecycle operations over an existing snapshot subject.

Recovery is intentionally different. Recovery reconstructs usable snapshot
state from external evidence when the original local subject is tombstoned,
corrupted, incomplete, or otherwise unsafe to restore in place. Because that
can cross source, install, and custody boundaries, recovery needs a separate
contract before any executor exists.

## 1. Recovery scope

Snapshot recovery is a future reviewed operation that may create a new local
snapshot subject from trusted external evidence.

Recovery may eventually cover:

- Rehydrating a snapshot from an imported full Studio bundle.
- Rehydrating a snapshot from an MV3 cache export.
- Rehydrating a snapshot from a future ChatGPT export ZIP flow.
- Preserving provenance and linkage to the original redacted subject.
- Keeping the original tombstone or corrupted marker as safety evidence.

Recovery does not repair the old subject in place. It does not bypass
retention, tombstone, privacy, account, or replay gates.

## 2. Out of F14.4 runtime implementation

Recovery is explicitly out of the F14.4 runtime implementation.

F14.4.12 does not add:

- recovery preflight
- recovery source ingestion
- recovery proposal generation
- recovery handoff execution
- recovery apply
- recovery applyEvent writing
- recovery bookkeeping writes
- recovery UI actions
- Native or F5 execution
- remote recovery

Any current F14.4 runtime module that mentions recovery must treat it as
deferred metadata or documentation only.

## 3. New subject creation rule

Recovery creates a **new snapshot subject**.

The recovered subject gets its own:

- `subjectId`
- `revisionHash`
- lifecycle state
- lineage
- operation IDs
- dedupe keys
- audit/proof chain

The original subject remains independently addressable as the prior
tombstoned, corrupted, or unsafe subject. The recovered subject may reference
the original only through redacted linkage fields.

## 4. Original marker persistence

The original subject tombstone or corrupted marker persists.

Recovery must not:

- remove the original tombstone
- overwrite the original row
- clear the original corruption marker silently
- mutate the original lifecycle row
- make the original subject appear restored

The original marker remains replay-defense and audit evidence. It explains why
recovery created a new subject instead of restoring the old one.

## 5. Recovery source priority

Future recovery implementation must evaluate sources in this priority order:

| Priority | Source | Contract |
|---:|---|---|
| 1 | `h2o.studio.fullBundle.v2` import | Preferred source because it carries the richest Studio-owned provenance and schema context. |
| 2 | MV3 cache export | Secondary source for browser-observed cache state when full bundle evidence is absent. |
| 3 | ChatGPT export ZIP | Deferred source. Requires a separate ingestion and provenance contract before use. |

Lower-priority sources must not override higher-priority valid evidence
without explicit operator review and a recorded blocker/warning explanation.

## 6. Reserved fields

The following fields are reserved for future recovery artifacts:

| Field | Meaning | Visibility |
|---|---|---|
| `recoveredFromSubjectIdHash` | Redacted hash of the original snapshot subject that motivated recovery. | Relay-safe only when hashed. |
| `recoveryProvenance` | Enum describing the recovery source class. | Enum only. |
| `recoveryAtIso` | Timestamp when the recovered subject was created. | Relay-safe timestamp. |

Allowed `recoveryProvenance` values should start narrow:

- `studio-full-bundle-v2`
- `mv3-cache-export`
- `chatgpt-export-zip`
- `manual-redacted-evidence`

Raw source paths, filenames, export IDs, ChatGPT conversation IDs, account IDs,
and content excerpts are not provenance values.

## 7. Recovery identity rule

Recovery identity is forward-only:

- Never reuse the original `subjectId`.
- Never silently overwrite the original subject.
- Never mutate the tombstoned, corrupted, or original row.
- Never derive the recovered `subjectId` from raw snapshot, chat, account, or
  install identifiers.
- Always generate new operation IDs, lineage, and dedupe keys for recovery.
- Always link back through `recoveredFromSubjectIdHash`, not through raw IDs.

If a recovered subject already exists for the same original subject and
provenance, future implementation must route to duplicate/replay review rather
than create another silent copy.

## 8. Privacy rules

Recovery artifacts and convergence envelopes must remain redacted.

Forbidden in convergence envelopes:

- raw snapshot id
- raw chat id
- raw account id
- content fields
- message text
- turn content
- attachment content
- model slug
- file paths
- URLs
- tokens
- local export filenames

Recovery provenance is enum-only. Recovery may use external evidence locally
for materialization, but relay-visible artifacts must contain only hashed
subjects, lifecycle summaries, bucketized metadata, and approved provenance
enums.

## 9. Replay rules

Recovery must not look like a reversal of old lifecycle evidence.

Rules:

- Recovery emits new operation IDs.
- Recovery emits new dedupe keys.
- Recovery does not reverse old consumed-operation rows.
- Recovery does not delete or rewrite consumed-operation evidence.
- Recovery must pass normal privacy gates.
- Recovery must pass normal tombstone gates.
- Recovery must pass normal account gates.
- Recovery must pass normal replay and watermark checks before any proposal or
  applyEvent receipt can be produced.

If old consumed evidence proves the same source was already recovered, the new
attempt is duplicate/replay, not a valid recovery.

## 10. Restore vs recovery

Restore and recovery are distinct operations.

| Operation | Subject identity | Source | Effect |
|---|---|---|---|
| Restore | Preserves the existing `subjectId`. | Existing local subject and tombstone/lifecycle evidence. | Reverses lifecycle from archived or tombstoned to captured when allowed. |
| Recovery | Creates a new `subjectId`. | External redacted evidence source. | Creates a replacement subject linked to the prior subject hash. |

Restore is a lifecycle operation. Recovery is a subject-creation operation with
provenance.

Restore must fail when the existing subject cannot safely be restored in place.
Recovery may later be considered when external evidence can prove a safe new
subject.

## 11. Future implementation gates

Future recovery implementation must be split into small phases:

| Gate | Purpose |
|---|---|
| Recovery preflight | Determine whether recovery is eligible without mutation. |
| Recovery source validation | Validate source type, schema, custody, account boundary, and privacy. |
| Recovery proposal candidate | Emit a redacted proposal for creating a new recovered snapshot subject. |
| Recovery handoff preview | Validate owner handoff without execution. |
| Recovery applyEvent receipt | Build redacted receipt only after a successful reviewed local recovery. |
| Recovery bookkeeping | Append proof/audit linkage without rewriting old evidence. |
| Recovery proof | Exercise positive and negative recovery lanes, including replay and raw-leak checks. |
| Recovery UI | Display recovery evidence and status without automatic recovery. |

No gate may skip the prior gate. No gate may apply automatically.

## 12. Explicit deferrals

The following are deferred:

- recovery executor
- ChatGPT ZIP ingestion
- cross-install transfer
- automatic recovery
- remote recovery
- silent overwrite repair
- batch recovery
- recovery from untrusted raw content
- recovery that clears old tombstones
- recovery that rewrites old consumed-operation rows

These deferrals are hard boundaries until a later review explicitly changes
the contract.

## 13. MUST / SHOULD / DEFER

MUST:

- Treat recovery as a new-subject creation path.
- Preserve the original subject tombstone or corrupted marker.
- Use `recoveredFromSubjectIdHash`, `recoveryProvenance`, and `recoveryAtIso`
  for redacted linkage.
- Prefer `h2o.studio.fullBundle.v2`, then MV3 cache export, then deferred
  ChatGPT export ZIP.
- Reject raw snapshot, chat, account, content, model, path, URL, and token
  fields in convergence envelopes.
- Generate new operation IDs, lineage, and dedupe keys.
- Pass normal privacy, tombstone, account, replay, watermark, and
  consumed-operation gates.
- Keep restore and recovery separate in code, UI, proof, and audit.

SHOULD:

- Make provenance enum values stable and versioned.
- Add negative proofs for duplicate recovery, raw field leaks, wrong account,
  expired retention, missing tombstone evidence, and corrupted source schema.
- Show the original subject only as a short hash in UI.
- Keep local raw recovery source details device-local and out of relay-visible
  artifacts.

DEFER:

- Recovery executor.
- ChatGPT export ZIP ingestion.
- Cross-install transfer.
- Automatic or remote recovery.
- Silent overwrite repair.
- Any recovery behavior that mutates the original subject.
