# F15.11 - Folder Binding Absorption Plan

## 1. Purpose

F15.11 absorbs existing F7 folder bindings into the F15 `library.binding` lane with:

```js
bindingKind: "chat-folder"
```

In shorthand, the target discriminator is `bindingKind = "chat-folder"`.

The absorption target is the binding edge only. F15.11 does not delete F7 modules in F15, does not remove the current folder UI behavior, and does not change folder color, rename, move, delete, parity, popup, menu, Native, Chrome, or Desktop behavior during the bridge phase.

F7 remains active initially. After the bridge and proof phases, F7 APIs may become thin wrappers over the F15 `library.binding` lane, but F7 fallback remains available for at least two release cycles after absorption.

## 2. Folder Endpoint Decision

The folder endpoint remains:

```txt
folder.metadata
```

Folders must not be modeled as `library.catalog`.

F15 `library.catalog` owns labels, tags, and categories. Folder metadata already has independent convergence ownership for color, rename, move, and delete. Recasting folders as `library.catalog` would duplicate identity, blur ownership, and risk regressions in existing folder parity and folder UI behavior.

F15.11 absorbs only this edge:

```txt
chat.metadata -> folder.metadata
```

into:

```txt
subjectType: "library.binding"
bindingKind: "chat-folder"
```

## 3. Identity Bridge

F15.11 must explicitly bridge three identity layers.

### 3.1 Legacy F10/F7 Preview Identity

The legacy folder sync preview lane has used `folderBinding` preview objects. Historical F10/F7 material contains mismatched ordering and raw-id based descriptions. F15.11 must not depend on raw identifiers in new artifacts.

The diagnostic may accept an existing legacy preview subject hash if already available:

```js
legacyF10SubjectId
```

It must not compute a raw-id preview subject from raw `chatId` or raw `folderId`.

### 3.2 F13 Bridge Identity

F13 established the redacted folder-binding bridge identity:

```js
legacyF13SubjectId =
  sha256("folderBinding:" + chatSubjectId + ":" + folderSubjectId)
```

where:

- `chatSubjectId` is a sha256 chat subject.
- `folderSubjectId` is a sha256 folder subject.
- raw chat IDs and raw folder IDs are forbidden.

### 3.3 F15 Target Identity

F15 target identity is the existing `library.binding` edge identity:

```js
libraryBindingSubjectId =
  sha256(canonicalJSON({
    subjectType: "library.binding",
    bindingKind: "chat-folder",
    leftSubjectId: chatSubjectId,
    rightSubjectId: folderSubjectId,
    perEnvelopeSalt
  }))
```

For `chat-folder`, `leftSubjectId` is always the chat subject and `rightSubjectId` is always the folder subject.

## 4. Migration Shadow Event Shape

F15.11 migration shadow events are append-only bridge evidence. They do not mutate F7 rows and do not create or delete folder bindings.

```js
{
  legacyF10SubjectId,
  legacyF13SubjectId,
  libraryBindingSubjectId,
  chatSubjectId,
  folderSubjectId,
  bindingKind: "chat-folder",
  migrationDigest
}
```

`legacyF10SubjectId` may be `null` when no redacted legacy preview hash is available. Raw IDs are never included.

`migrationDigest` is derived from the redacted shadow event fields with canonical JSON and sha256.

## 5. Compatibility Strategy

F15.11 compatibility rules:

- F7 remains active at the start of F15.11.
- F7 APIs are not deleted in F15.
- F7 fallback remains available for at least two release cycles after absorption.
- F7 APIs may later become thin wrappers over the F15 `library.binding` lane after proof closure.
- Existing folder color, rename, move, delete, and parity behavior must remain unchanged.
- Existing `desktopSetFolderBinding` return shape must be preserved.
- Existing `folder-binding-changed` event behavior must be preserved.
- Existing cache busting and library index refresh behavior must be preserved.
- Existing folder popup and menu behavior must remain compatible.

The current F7 direct store behavior uses `folder_bindings.PRIMARY KEY (chat_id)` and can atomically move a chat between folders through replacement semantics. The F15 lane must model that as two explicit edge operations when it takes ownership:

1. unbind old `chat-folder`
2. bind new `chat-folder`

There is no single replace operation at the `library.binding` lane level.

## 6. Implementation Split

### F15.11.a - Bridge Diagnostic

Add this contract and a read-only diagnostic that maps existing F7 folder bindings into the future F15 identity model.

No writes, no wrapper delegation, no F15 `chat-folder` enablement.

### F15.11.b - Enable `chat-folder` in F15 Binding Lane

Enable `bindingKind: "chat-folder"` in:

- library binding canonicalizer
- diagnostics
- preflight
- proposal candidates
- handoff preview
- applyEvent receipt
- bookkeeping

No execute or settlement support yet.

### F15.11.c - Execute and Settlement Support

Add `chat-folder` execute envelope support and settlement routing.

Rules:

- no F5 path
- no category cache refresh
- no direct `chats.category_id` writes
- no folder metadata mutation

### F15.11.d - Migration Shadow Events and Guarded F7 Wrapper Delegation

Emit migration shadow events and allow guarded F7 wrapper delegation to the F15 lane.

F7 fallback remains available.

### F15.11.e - Optional Trigger/Sentinel Protection

Consider SQLite trigger/sentinel protection for `folder_bindings` only after the F15 settlement writer path is proven.

Do not protect `folder_bindings` before wrapper delegation and recovery behavior are proven.

### F15.11.f - Proof, Validators, and UI Status

Extend F15 proof closure and operator status with folder-binding absorption cases.

Required proof areas:

- F7 parity remains green.
- Legacy F7 row maps to F15 `chat-folder`.
- Bind `chat-folder` through F15 pipeline.
- Second active `chat-folder` for same chat blocks.
- Unbind `chat-folder` through F15 pipeline.
- F7 wrapper delegates without changing return shape or events.
- No duplicate rows after migration.
- Folder color, rename, move, delete, and parity remain unchanged.
- F15.9 proof remains green.

## 7. Safety Gates

F15.11 must block or warn on:

- missing chat subject hash
- missing folder subject hash
- malformed subject hash
- legacy F13 identity mismatch
- F15 target identity mismatch
- more than one active `chat-folder` binding for a chat
- raw ID, name, path, URL, content, or token exposure
- F5 footprint in any `chat-folder` binding receipt or execute envelope
- category cache refresh metadata on `chat-folder`

## 8. Privacy Contract

F15.11 artifacts must be hash-only.

Forbidden:

- raw chat IDs
- raw folder IDs
- folder names
- folder colors
- paths
- URLs
- content
- tokens
- `folder_id`
- `chat_id`

Allowed:

- `chatSubjectId`
- `folderSubjectId`
- `legacyF10SubjectId`
- `legacyF13SubjectId`
- `libraryBindingSubjectId`
- `migrationDigest`
- blocker and warning codes
- counts

## 10. Non-Goals

F15.11.a does not:

- enable `chat-folder` in the F15 binding lane
- delegate F7 APIs
- delete F7 modules
- write folder stores
- write SQLite
- change execute or settlement behavior
- change folder parity behavior
- change folder UI behavior
- change Native, Chrome, or Desktop folder sync behavior
