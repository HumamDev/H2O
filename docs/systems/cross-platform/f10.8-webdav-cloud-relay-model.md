# F10.8.0 - WebDAV/cloud relay transport model

> **Contract document. F10.8.0 is docs-only.** No WebDAV code, no cloud
> adapter, no remote apply, no convergence logic, no automatic sync, and no
> mobile write-back is introduced here.

This document defines the first safe external transport layer for the F10
cross-platform mesh: read-only relay of validated envelopes between peers.

## 1. Scope and status

| | |
|---|---|
| Phase | **F10.8.0** |
| Deliverable | This document only |
| WebDAV/cloud runtime | **None** |
| Remote apply | **Forbidden** |
| Convergence | **None** |
| Automatic sync | **None** |
| Mobile write-back | **None** |
| Mutation behavior | **Unchanged** |
| Status | Pre-implementation transport safety contract |

F10.7 proved the local mutation boundary:

```
local preview approval -> local Desktop apply -> applyEvent receipt
```

F10.8 does not make that receipt executable by another peer. It only defines
how future peers may relay envelopes as inert data.

## 2. Architectural principle

**Transport is separated from mutation.**

WebDAV/cloud relay may move envelope bytes between peers, but relay delivery
never authorizes a write. A receiving platform treats remote envelopes as
advisory inputs for validation, diagnostics, preview, proposal, or review.

`applyEvent` remains evidence only:

```
producer: "I already applied this locally under my own gates."
receiver: "I can validate, display, dedupe, and compare this receipt."
receiver must not: "Apply this mutation locally."
```

All mutation remains locally authorized. A remote envelope can motivate a local
preview or review flow, but cannot skip local gates.

## 3. Transported envelope kinds

The relay may eventually carry these F10.2 envelope kinds as immutable blobs:

| Kind | Meaning in relay | Allowed emitters |
|---|---|---|
| `evidence` | Observation of state or peer health. | Native extension, Chrome Studio, Desktop Studio, mobile metadata-only. |
| `preview` | Dry-run diagnostic. | Chrome Studio and Desktop Studio; mobile read-only preview may display but not publish by default. |
| `proposal` | Explicit request for local review. | Desktop Studio first; future mobile/native only after separate proposal-source authorization. |
| `conflictCandidate` | Reportable divergence for review. | Desktop Studio only until a strong comparator is authorized elsewhere. |
| `applyEvent` | Past-tense local apply receipt. | Desktop Studio only. Never a remote command. |

`bundle` and `cacheMetadata` may be used as local packaging/status formats, but
they do not increase authority. A bundle wrapping an `applyEvent` is still just
a carrier for a past-tense receipt.

## 4. Transport topology

The relay topology has five distinct layers:

```
local producer -> local outbox -> relay storage -> local inbox -> local review/preview
```

| Layer | Role | Authority |
|---|---|---|
| Desktop peer | Strong local authority; can emit evidence/proposal/conflict/applyEvent after local gates. | Local only. |
| Chrome Studio peer | Preview coordinator and evidence surface. | No apply. |
| Relay storage | Dumb blob store and optional metadata index. | None. |
| Local outbox | Pending locally produced envelopes ready to upload. | None beyond producer's envelope validation. |
| Local inbox | Received envelopes after download and validation. | Display/review input only. |

Manual refresh is the first safe sync posture. Polling can be considered later
only if it writes to inbox state, not to domain state. Relay namespaces must be
per install/peer or per trust group and must never be a mutable shared database.

## 5. Relay storage model

The relay stores immutable envelope blobs and optional metadata indexes.

| Item | Requirement |
|---|---|
| Envelope blob | Canonical JSON envelope bytes, immutable after upload. |
| Envelope digest | SHA-256 over canonical bytes; used for integrity and dedupe. |
| Metadata index | Optional append-only list of digests, kinds, createdAt, source peer hashes, and redaction class. |
| Peer identity | Redacted peer envelope only; no raw peer IDs in shared relay state. |
| Timestamps | Advisory for ordering/freshness only; never authority. |
| Dedupe keys | Stored as hashes/strings from envelope, used only to detect repeats. |
| Retention policy | Bounded by age/count/quota. Retention cleanup cannot imply domain delete. |

Relay storage is not the source of truth. It is a mailbox.

## 6. WebDAV safety rules

| Rule | Requirement |
|---|---|
| Append-only preferred | Upload new digest-addressed blobs; avoid modifying existing objects. |
| Immutable envelopes | A digest-addressed blob must never be overwritten with different bytes. |
| No remote delete authority | Deleting relay files is mailbox cleanup only, never entity deletion. |
| No mutable shared state | Avoid a shared "current state" JSON document that peers overwrite. |
| No last-writer-wins | Relay upload time does not determine domain truth. |
| No lock-as-authority | WebDAV locks, if used, protect upload mechanics only, not apply decisions. |
| No apply-on-download | Download ends at local inbox validation. |

If a transport cannot provide immutable semantics, the implementation must
simulate them with digest-named files and refuse digest collisions.

## 7. Remote envelope acceptance gates

A downloaded envelope is accepted into local inbox only after all gates pass:

| Gate | Requirement |
|---|---|
| Schema validation | F10.2 base/kind/authority validators pass. |
| Peer identity validation | Source peer envelope is redacted, well-formed, and allowed for the namespace/trust group. |
| Digest validation | Blob digest matches metadata index and envelope `eventDigest`/`payloadHash` semantics. |
| Dedupe | Existing `eventDigest`, `dedupeKey`, or envelope ID is handled idempotently. |
| Replay protection | Replayed or stale envelopes do not reopen writes or re-trigger review automatically. |
| Expiration | Expired envelopes are displayed/archived as stale, not used for new writes. |
| Redaction validation | Default relay envelopes must be `redactionClass:"redacted"` or stricter. |
| Forbidden-field scan | Reject forever-no fields: content, body, text, messages, attachments, URL/path/secrets/tokens. |
| Capability check | Source platform may emit the envelope kind/capability under F10.1/F10.2. |

Acceptance into inbox is not apply approval. It only means "safe to store as
received evidence/report/proposal input."

## 8. Local inbox model

The local inbox is an append-only or append-mostly table/cache of received
envelopes and validation outcomes.

| State | Meaning |
|---|---|
| `received` | Downloaded and parsed, awaiting validation. |
| `validated` | Passed schema/digest/redaction/authority checks. |
| `pending-review` | Requires human or local-authority review before any next step. |
| `ignored` | Operator or policy ignored it. |
| `expired` | Too old for proposal/apply use; display only. |
| `deduped` | Duplicate of an already known envelope. |
| `blocked` | Failed validation or safety policy. |

The inbox can feed diagnostics, preview, proposal-review UI, and conflict
analysis. It cannot feed direct domain mutation.

## 9. Transport lifecycle

### Outbound

```
local event/proposal/evidence produced
-> validate envelope locally
-> write local outbox row/blob
-> upload immutable blob
-> append/update relay metadata index
```

### Inbound

```
manual refresh or future safe polling
-> list relay namespace
-> download unknown blobs
-> verify digest
-> validate envelope
-> store local inbox row
-> expose as display/review/preview input only
```

The lifecycle intentionally stops at local inbox. Any future mutation requires a
new local preview/proposal/review/apply sequence.

## 10. Explicitly forbidden behavior

These behaviors are forbidden in F10.8.x unless a later safety contract changes
them:

- Remote-triggered apply.
- Remote overwrite of local state.
- Remote delete authority.
- Automatic merge.
- Remote conflict resolution.
- Mutable shared cloud state as source of truth.
- Last-writer-wins sync.
- Applying from `applyEvent` receipt.
- Applying from relay metadata index.
- Applying from inbox insert.
- Mobile write-back.
- WebDAV as a lock manager for domain authority.
- Uploading raw local audit rows.
- Uploading raw names, raw IDs, raw capture text, attachments, paths, URLs, or secrets.

## 11. Privacy and redaction requirements

Relay data is externally stored and must default to redacted envelopes.

| Data class | Relay rule |
|---|---|
| Folder names | Hash only by default; no raw names. |
| Folder/chat IDs | Hashed `subjectId`/peer-safe IDs only; no raw local IDs. |
| Capture/chat text | Forbidden. |
| Attachments | Forbidden. |
| File paths / URLs | Forbidden. |
| Tokens/secrets | Forbidden. |
| Local audit rows | Do not upload raw rows. Upload redacted `applyEvent` receipts only. |
| Device-local diagnostics | Not relay-safe unless separately encrypted and authorized. |

Encryption can be added later, but encryption does not relax redaction. A
leaked relay blob should still be non-content and non-authoritative.

## 12. Recommended future order

| Phase | Scope | Mutation status |
|---|---|---|
| F10.8.1 | Local outbox | Store locally produced envelopes for relay; no network. |
| F10.8.2 | Local inbox | Store validated remote envelopes; no apply. |
| F10.8.3 | WebDAV adapter | Upload/download immutable blobs manually. |
| F10.8.4 | Relay dedupe/index | Digest-addressed metadata index and replay handling. |
| F10.8.5 | Manual sync UI | Operator-triggered send/refresh/status. |
| F10.8.6 | Optional convergence experiments | Still local-gated; no remote apply. |

No phase may turn delivery into mutation. Convergence experiments come after
outbox, inbox, validation, dedupe, and manual review are proven.

## 13. MUST / SHOULD / DEFER tables

### MUST

| Requirement | Reason |
|---|---|
| Preserve F10.2 validation strictness. | Keeps kind/authority/redaction gates enforceable. |
| Treat relay storage as transport only. | Prevents cloud state from becoming authority. |
| Keep `applyEvent` evidence-only. | Prevents remote apply by receipt. |
| Validate before inbox acceptance. | Prevents malformed/unsafe remote data from entering review flow. |
| Use digest-based dedupe. | Prevents replay and duplicate processing. |
| Keep relay envelopes redacted by default. | Reduces privacy risk outside the device. |
| Make first sync manual/operator-triggered. | Avoids hidden background behavior while safety is immature. |

### SHOULD

| Recommendation | Reason |
|---|---|
| Use digest-named immutable blobs. | Makes overwrite/collision detection straightforward. |
| Keep metadata indexes reconstructable from blobs. | Avoids index corruption becoming data loss. |
| Separate outbox and inbox stores. | Prevents received data from looking locally authored. |
| Record validation blockers and warnings. | Supports operator debugging without applying. |
| Cap retention by age/count/quota. | Keeps relay storage bounded. |
| Prefer code/count diagnostics over raw payload display. | Maintains privacy posture. |

### DEFER

| Item | Deferred until |
|---|---|
| Automatic polling | Manual relay flow is proven safe. |
| WebDAV conflict repair | Inbox/outbox/index model is proven. |
| Mobile relay publishing | Mobile proposal/write-back safety model exists. |
| Encrypted relay blobs | Redacted plaintext model and key lifecycle are designed. |
| Convergence | Local-gated apply and relay inbox/outbox are production-proven. |
| Delete propagation | F5-backed delete relay safety is separately designed. |

## 14. Done definition for F10.8.0

F10.8.0 is done when this document is committed and no runtime behavior has
changed.

The next implementation phase must cite this document and prove that WebDAV or
cloud delivery cannot trigger local mutation, remote overwrite, automatic merge,
mobile write-back, or applyEvent-as-command behavior.
