# F16.5 Production Release Gate Checklist

## Architecture Verdict

F16.5 is a thin production release-gate orchestrator for the Library Sync and folder binding hardening stack. It is not a new runtime feature phase.

The release gate should be modeled on `tools/validation/identity/run-identity-release-gate.mjs`:

- Execute existing F15/F16 validators as subprocesses.
- Do not re-implement validator logic.
- Run all default checks and aggregate results instead of failing fast.
- Emit a three-state release verdict.
- Preserve heavy checks as opt-in only.

F16.5.a is docs-only. Future implementation should add a runner after the checklist is stable.

## Default Release-Gate Groups

### 1. Repo State

The runner should inspect git state before and after validation.

- Working tree status.
- Staged state.
- Unstaged unrelated WIP is a warning.
- Staged unapproved files are a blocker unless `--approve-staged` is supplied.

### 2. F15 Closure

Default F15 closure validators:

- `node tools/validation/sync/validate-f15-library-closure.mjs`
- `node tools/validation/sync/validate-f15-library-sync-proof.mjs`
- `node tools/validation/sync/validate-f15-cutover.mjs`
- `node tools/validation/sync/validate-f15-bulk-migration.mjs`
- `node tools/validation/sync/validate-f15-library-conflict-contract.mjs`
- `node tools/validation/sync/validate-f15-folder-binding-absorption.mjs`
- `node tools/validation/sync/validate-f15-library-sync-ui.mjs`

### 3. F16 Hardening

Default F16 hardening validators:

- `node tools/validation/sync/validate-f16-library-conflict-runtime.mjs`
- `node tools/validation/sync/validate-f16-library-multipeer-soak.mjs`
- `node tools/validation/sync/validate-f16-library-performance-stress.mjs`
- `node tools/validation/sync/validate-f16-folder-bindings-trigger-decision.mjs`

### 4. Parity

Default parity validator:

- `node tools/validation/sync/validate-f7-folder-metadata-hash-parity.mjs`

### 5. Cross-Platform

Default cross-platform validators:

- `node tools/validation/cross-platform/run-cross-platform-repo-scan.mjs`
- `node tools/validation/cross-platform/validate-cross-platform-envelope.mjs`

### 6. Optional Checks

Optional checks must not run by default:

- Heavy soak with `F16_SOAK_HEAVY=1`.
- Heavy stress with `F16_STRESS_HEAVY=1`.
- Live Studio DevTools runtime proof attestation.
- `cargo check`.

## Decision States

| State | Meaning | Exit |
|---|---|---|
| `RELEASE READY` | All default checks pass and no warnings exist | `0` |
| `RELEASE CONDITIONALLY READY` | All default checks pass but warnings exist | `0` |
| `RELEASE NOT READY` | One or more blockers exist | `1` |

## Blocker Taxonomy

The release runner should use stable blocker codes so release reports can be compared across runs:

- `release-gate-default-validator-failed:<name>`
- `release-gate-privacy-leak`
- `release-gate-side-effect-flag-flip`
- `release-gate-direct-protected-write-bypass`
- `release-gate-conflict-runtime-unavailable`
- `release-gate-soak-proof-failed`
- `release-gate-performance-anomaly-miss`
- `release-gate-folder-trigger-proof-failed`
- `release-gate-f7-parity-regression`
- `release-gate-cross-platform-scan-failed`
- `release-gate-staged-files-not-approved`

## Warning Taxonomy

Warnings indicate release evidence is incomplete or non-blocking risk remains:

- `release-gate-unrelated-unstaged-wip`
- `release-gate-heavy-soak-not-run`
- `release-gate-heavy-stress-not-run`
- `release-gate-studio-runtime-not-attested`
- `release-gate-performance-soft-budget-warning`
- `release-gate-cargo-check-not-run`
- `release-gate-manual-checklist-incomplete`

## Manual Studio DevTools Runtime Checklist

The release gate cannot fully replace live Studio/Tauri runtime checks because Node validators do not exercise the live Tauri runtime, real module graph, or Studio loader state.

Manual DevTools checks:

```js
await H2O.Desktop.Sync.runLibraryEndToEndSyncProof()
await H2O.Desktop.Sync.runLibrarySyncClosureProof()
await H2O.Desktop.Sync.runLibraryMultiPeerSoakProof()
await H2O.Desktop.Sync.runLibraryPerformanceStressProof()
await H2O.Desktop.Sync.runLibraryFolderBindingTriggerProof?.()
```

If these are not attested for a release candidate, the runner should emit `release-gate-studio-runtime-not-attested`.

## Future Runner Behavior

The F16.5.b runner should:

- Use `spawnSync`.
- Mirror the identity release gate style.
- Run all checks, not fail fast.
- Aggregate command status, stdout/stderr summaries, blockers, and warnings.
- Support `--json`.
- Support `--approve-staged`.
- Keep heavy checks opt-in.
- Never duplicate validator logic.

## Risks And Mitigations

| Risk | Mitigation |
|---|---|
| Validators become slow | Default gate runs lightweight checks only |
| Heavy checks accidentally become default | Require explicit env/flag gating |
| False confidence without live Studio runtime | Emit runtime-attestation warning |
| Unrelated WIP pollutes release signal | Classify staged vs unstaged and in-scope vs out-of-scope |
| Gate duplicates validator logic | Orchestrate existing validators only |
| Fail-fast hides additional failures | Run all default checks and aggregate |
