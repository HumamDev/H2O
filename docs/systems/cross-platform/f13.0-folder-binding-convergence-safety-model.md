# F13.0 - Folder binding convergence safety model

> **Contract document. F13.0.0 is docs-only.** No folder binding
> implementation, apply change, mobile write-back, remote apply, automatic
> binding, or runtime behavior change exists in this phase.

This document defines the safety boundary for folder binding convergence after
color, rename, move, and delete convergence. A folder binding is not cosmetic
metadata: it changes the relationship between a chat and a folder, affects
library placement and delete eligibility, and can create membership conflicts.
F13.0 therefore starts with a narrow, reviewed, Desktop-local binding-add path
only.

## 1. Scope and status

| | |
|---|---|
| Phase | **F13.0.0** |
| Deliverable | This document only |
| Runtime code | **None** |
| Binding apply implementation | **None** |
| Mobile write-back | **None** |
| Remote apply | **Forbidden** |
| Automatic binding | **Forbidden** |
| Status | Authoritative pre-implementation safety contract |

F13.0 may eventually allow Desktop Studio to perform one locally approved
folder binding-add convergence action. It must not allow any platform to bind
or unbind a chat by receiving a remote envelope.

## 2. Binding convergence lifecycle

Binding convergence must follow this sequence without skipping stages:

```txt
remote observed evidence
-> canonical folderBinding identity
-> local chat/folder materialization
-> binding cardinality diagnostic
-> binding preflight
-> operator approval
-> local reviewed binding add
-> local applyEvent receipt
-> consumed-operation row
-> convergence watermark row
```

Every stage is fail-closed. Receipt of a remote envelope is never authority to
change membership. UI confirmation is not enough; the apply transaction must
repeat identity, cardinality, duplicate, tombstone, watermark, and replay checks
immediately before mutation.

## 3. Canonical binding identity model

F13.0 chooses the canonical binding subject order:

```txt
bindingSubjectId =
  sha256("folderBinding:" + chatSubjectId + ":" + folderSubjectId)
```

Where:

- `chatSubjectId` is the redacted sha256 subject for the chat.
- `folderSubjectId` is the redacted sha256 subject for the folder.
- Raw `chatId` and raw `folderId` never appear in relay-visible artifacts.
- `subjectType` is `"folderBinding"`.
- `revisionHash = sha256(canonicalJSON({
  chatSubjectId, folderSubjectId, state: "bound"
}))`.

This contract intentionally uses subject hashes, not raw IDs, as the binding
identity inputs. The resulting `bindingSubjectId` is stable across peers only
when both peers agree on the same redacted chat/folder subjects and hash
version.

## 4. Existing mismatch to resolve before implementation

There is an existing mismatch in the pre-F13 material:

| Source | Existing order |
|---|---|
| F10.6 object-model doc | `folderBinding:<chatId>:<folderId>` |
| Current canonicalizer implementation | hashes `folderBinding:<folderId>:<chatId>` |
| **F13 canonical contract** | `folderBinding:<chatSubjectId>:<folderSubjectId>` |

F13 implementation MUST add an identity/cardinality diagnostic before any apply
path. That diagnostic must report:

- the legacy doc order
- the current canonicalizer order
- the F13 canonical order
- whether observed local binding rows can be mapped unambiguously to the F13
  subject-hash identity

No binding proposal or apply may ship until this mismatch is resolved in code
or explicitly bridged by a compatibility rule.

## 5. Binding cardinality policy

Implementation must detect and report the current product cardinality policy
before apply:

- **Single-folder-per-chat** if current product behavior requires exactly one
  live folder binding per chat.
- **Explicit multi-folder policy** if current product behavior allows a chat to
  appear in multiple folders.

The diagnostic must fail closed if policy cannot be determined. It must report:

- number of live bindings for the chat
- whether adding the target binding would violate policy
- whether an equivalent binding already exists
- whether any duplicate or malformed binding row exists

If single-folder-per-chat is active, binding add requires the chat to have no
other live folder binding. A chat move between folders is a separate compound
operation and is deferred.

## 6. First safe operation

The first safe mutation is **binding add only**.

Binding add is allowed only when:

- the chat resolves locally
- the folder resolves locally
- the target folder is live
- the chat is live
- no equivalent binding already exists
- cardinality policy allows the add
- F5/F6 blockers are clear
- base, watermark, consumed-operation, and replay checks pass

The first implementation must bind exactly one chat to exactly one folder.
There is no batch binding and no automatic convergence.

## 7. Deferred operations

The following are explicitly deferred:

- unbind
- chat move between folders
- cascade binding delete
- binding tombstone apply
- automatic reconciliation
- batch membership repair
- mobile write-back
- remote apply

Unbind is more destructive than add because it removes organization state. A
chat move is not a primitive binding add; it is a reviewed remove-plus-add
workflow that needs its own safety model.

## 8. Ownership model

| Platform | Evidence / preview | Proposal | Apply | Notes |
|---|---:|---:|---:|---|
| Native extension | yes | no | no | May observe chat/folder context only. |
| Chrome Studio | yes | no | no | Preview/coordinator only. |
| Desktop Studio | yes | yes | yes, reviewed local only | First trusted apply authority. |
| Mobile | read-only / future proposal only | no for F13.0 | no | No mobile write-back. |

Desktop applies first. Chrome, native, and mobile produce evidence/previews
only for F13.0. A remote `applyEvent` is evidence of another peer's completed
local mutation; it is never interpreted as a remote binding command.

## 9. Conflict taxonomy

| Conflict | Rule |
|---|---|
| add vs add | Idempotent only if the same `bindingSubjectId`, lineage, base, and target are proven. Otherwise duplicate/conflict review. |
| add vs unbind | Hard conflict. Route to F6/F5-reviewed membership decision. |
| duplicate binding | Blocker. Equivalent live binding already exists. |
| orphan binding | Blocker. Chat or folder cannot be resolved, or folder/chat is deleted. |
| binding vs folder delete | Folder delete owns safety. Binding add to a tombstoned or delete-reviewed folder blocks. Folder delete remains blocked when membership exists. |
| binding vs folder move | Binding identity remains attached to the folder subject, not the folder's parent. Block if move changes the base/watermark needed to prove the binding target. |
| binding vs folder rename | Rename does not change binding identity. Block only if folder identity cannot be resolved or rename conflicts obscure target confirmation. |
| binding vs chat delete | Block. Chat tombstone/delete review owns the conflict. |
| binding vs unknown chat | Block unless the chat evidence is materialized under a future reviewed chat-create path. |

No binding conflict is auto-merged in F13.0.

## 10. Required preflight checks

All checks run during diagnostic/preflight and again immediately before apply:

| Check | Blocker on failure |
|---|---|
| F13 canonical identity can be derived | `binding-identity-unverified` |
| Chat resolves to exactly one live local chat | `chat-not-resolved` |
| Folder resolves to exactly one live local folder | `folder-not-resolved` |
| Chat is not deleted/tombstoned | `chat-tombstone-present` |
| Folder is not deleted/tombstoned/F5 pending | `folder-tombstone-present` |
| Equivalent binding is absent | `duplicate-folder-binding` |
| Cardinality policy known and satisfied | `binding-cardinality-violation` |
| No orphan binding condition | `orphan-binding` |
| No open F6 conflict for chat/folder/binding | `f6-blocker-present` |
| No F5 blocker for chat/folder/binding | `f5-blocker-present` |
| Base hash matches current binding set | `baseline-hash-not-verified` |
| Watermark safe | `watermark-not-safe` |
| Consumed-operation safe | `consumed-operation-not-safe` |
| Replay/oscillation safe | `replay-not-safe` |
| Operator approval present | `operator-approval-token-required` |
| Output redaction scan passes | `binding-output-contains-forbidden-field` |

If any local source cannot prove these conditions, the candidate is blocked.

## 11. Watermark interaction

Watermarks are not advanced by preview, proposal, conflictCandidate, UI review,
or preflight. A binding watermark may be recorded only after:

- local reviewed binding add committed successfully
- binding applyEvent receipt was built from committed audit evidence
- consumed-operation row was recorded
- `bindingSubjectId`, `lineageId`, and post-state hash are valid

Watermark key:

```txt
peerId + bindingSubjectId + lineageId + revisionHash
```

Below-watermark binding evidence is inert. A binding proposal with a stale base
must be regenerated from a fresh canonical snapshot.

## 12. Consumed-operation interaction

After successful local binding add finalization, record exactly one consumed
operation row:

```txt
envelopeKind: "applyEvent"
operationKind: "folderBinding.add"
consumedStatus: "consumed"
subjectId: bindingSubjectId
lineageId: <binding lineage>
eventDigest: <applyEvent digest>
dedupeKey: <applyEvent dedupe key>
```

The consumed-operation ledger blocks:

- duplicate event digest
- duplicate dedupe key
- replayed binding proposals
- inverse-lineage oscillation within the configured window
- reprocessing an already consumed binding add

Recording consumed-operation rows does not advance watermarks by itself.

## 13. Replay and dedupe protection

Binding replay protection must use all of:

- `bindingSubjectId`
- `chatSubjectId`
- `folderSubjectId`
- `baseHash`
- `targetHash`
- source peer identity
- lineage id
- event digest
- dedupe key
- watermark position
- consumed-operation ledger status

Recommended binding proposal dedupe:

```txt
dedupeKey = sha256(canonicalJSON({
  subjectType: "folderBinding",
  operation: "folderBinding.add",
  bindingSubjectId,
  baseHash,
  targetHash,
  actorPeerSyncHash
}))
```

Replayed add of an existing equivalent binding is idempotent only if the
binding already exists with the same identity and lineage has been consumed.
Otherwise it is blocked as duplicate/replay.

## 14. Rollback / compensating unbind model

Rollback is not a silent delete of the binding row.

For binding add, rollback is a **compensating reviewed unbind**:

```txt
binding add apply
-> applyEvent
-> operator discovers error
-> reviewed unbind proposal
-> unbind safety gates
-> local unbind apply
-> unbind applyEvent
-> consumed row + watermark
```

The compensating unbind is deferred until F13 proves binding add and a separate
unbind safety model is accepted. F13.0 does not implement unbind.

## 15. Privacy and redaction rules

Forbidden in relay-visible binding artifacts:

- raw `chatId`
- raw `folderId`
- chat title
- folder name
- paths
- URLs
- content
- message text
- snapshots
- attachments
- tokens or credentials

Allowed in redacted artifacts:

- `chatSubjectId`
- `folderSubjectId`
- `bindingSubjectId`
- `revisionHash`
- counts
- operation/status enums
- blocker/warning codes
- peer identity hashes
- audit ids where already allowed by the applyEvent contract

Device-local diagnostics may inspect raw IDs internally to resolve local rows,
but raw values must not be returned from default redacted APIs or stored in
relay-visible envelopes.

## 16. Binding vs other convergence lanes

Binding vs color:

- Color changes do not affect binding identity.
- Block only if folder identity or base cannot be verified.

Binding vs rename:

- Rename does not affect binding identity.
- Block if target folder cannot be confirmed after rename or if F6 reports a
  rename conflict for the folder.

Binding vs move:

- Move does not affect binding identity.
- Block if target folder is in a stale move lineage or the binding base was
  computed against a different folder state than the current move watermark.

Binding vs delete:

- Delete owns safety. A folder with live bindings is non-empty and delete is
  blocked in F12. Binding add to a folder under delete review blocks.

Binding vs create:

- Binding add to an unknown remote-created folder or chat blocks until the
  folder/chat exists locally through a reviewed create path.

## 17. MUST / SHOULD / DEFER

### MUST

- Resolve the binding ID order mismatch before implementation.
- Use `bindingSubjectId = sha256("folderBinding:" + chatSubjectId + ":" +
  folderSubjectId)`.
- Detect and report binding cardinality policy before apply.
- Start with binding add only.
- Apply only on Desktop, reviewed and operator-approved.
- Re-run all preflight checks inside the apply transaction.
- Require live chat, live folder, no duplicate binding, no orphan condition,
  no F5/F6 blocker, fresh base, safe watermark, safe consumed-operation ledger,
  and replay suppression.
- Emit applyEvent only after successful local commit.
- Record consumed-operation and watermark rows only after applyEvent
  validation.
- Keep outputs redacted and forever-no-field scanned.

### SHOULD

- Add a dedicated identity/cardinality diagnostic before materialization.
- Surface cardinality policy and duplicate binding count prominently in UI.
- Treat add-vs-add as idempotent only with matching identity, lineage, and
  hashes.
- Route add-vs-unbind to F6/F5 review.
- Keep binding evidence counts separate from folder metadata diffs.
- Add runtime proof harness before dogfood.

### DEFER

- Unbind.
- Chat move between folders.
- Cascade binding delete.
- Binding tombstone apply.
- Multi-binding batch repair.
- Automatic reconciliation.
- Mobile write-back.
- Remote apply.

## 19. No-go areas

F13.0 and its initial implementation phases must not:

- auto-bind or auto-unbind
- apply remote envelopes
- infer chat moves from binding diffs
- cascade folder delete into binding delete
- bypass F5/F6
- emit raw chat/folder identifiers
- mutate mobile state
- upload or download through WebDAV as part of apply
- batch apply binding changes

The only acceptable first mutation is a single, reviewed, Desktop-local
`folderBinding.add` operation after the identity/cardinality diagnostic proves
the local policy and the operator approves the action.
