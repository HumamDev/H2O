# F10.9.0 - Rename convergence safety model

> **Contract document. F10.9.0 is docs-only.** No rename implementation, apply
> change, mobile write-back, remote apply, automatic rename, or runtime behavior
> change exists in this phase.

This document defines the safety boundary for folder rename convergence after
F10.8 proved color-only local convergence. Rename is intentionally stricter
than color because folder names are user-visible, privacy-sensitive, and
ambiguous when sibling names collide.

## 1. Scope and status

| | |
|---|---|
| Phase | **F10.9.0** |
| Deliverable | This document only |
| Runtime code | **None** |
| Rename apply implementation | **None** |
| Mobile write-back | **None** |
| Remote apply | **Forbidden** |
| Automatic rename | **Forbidden** |
| Status | Authoritative pre-implementation safety contract |

F10.9 may eventually allow Desktop Studio to perform one locally approved
folder rename convergence action. It must not allow any platform to remotely
rename a folder by sending an envelope.

## 2. Rename convergence lifecycle

Rename convergence must follow this sequence without skipping stages:

```txt
remote observed evidence
-> redacted targetNameHash
-> local materialization of target name
-> hash verification
-> rename preflight
-> operator approval
-> local transactional rename
-> local applyEvent receipt
-> consumed-operation row
-> convergence watermark row
```

Each stage is fail-closed. A later stage cannot repair or waive a failed
earlier stage. Receipt of a remote envelope is never authority to rename.

## 3. Remote targetNameHash model

Remote peers may expose only a redacted name target:

```txt
targetNameHash = sha256(normalizedTargetName)
```

Remote envelopes and relay-visible artifacts must not contain raw folder names.
They may contain:

- `subjectId`
- `lineageId`
- `eventDigest`
- `dedupeKey`
- `baseHash`
- `targetHash`
- `targetNameHash`
- `changedFields: ["name"]`
- blocker and warning codes

They must not contain:

- raw target name
- raw previous name
- chat names or chat IDs
- local folder IDs
- file paths, URLs, tokens, or content

`targetNameHash` is evidence only. It is not enough to apply a rename.

## 4. Local materialization model

The cleartext target name must be materialized locally by the operator or by a
device-local trusted UI state that never leaves the device.

Valid sources:

- operator manually types the target name
- operator selects a device-local candidate already visible in local UI
- future device-local encrypted/private cache, if explicitly designed later

Invalid sources:

- raw name from relay envelope
- raw name from WebDAV/cloud blob
- raw name from remote applyEvent
- raw name inferred from another user's device state without local approval

The materialized name is local input to preflight and apply only. It must not
be written into proposal, conflictCandidate, applyEvent, relay index, or
transport artifacts.

## 5. Hash verification flow

Before preflight can become actionable:

1. Normalize the operator-provided target name using the F10.9 normalization
   rules.
2. Compute `sha256(normalizedTargetName)`.
3. Compare it to remote `targetNameHash`.
4. Refuse if either value is missing.
5. Refuse if the hashes differ.
6. Refuse if `changedFields` is anything other than exactly `["name"]`.

The local apply path must verify again immediately before mutation. Hash
verification in the UI is not sufficient.

## 6. Unicode normalization requirements

Rename convergence must define one canonical name normalization function before
any rename implementation is allowed.

Minimum requirements:

- Use Unicode NFC normalization.
- Trim leading and trailing whitespace.
- Collapse platform-invalid control characters to blockers, not replacements.
- Preserve case unless the platform's folder-name semantics are proven
  case-insensitive.
- Preserve internal spaces unless a later product decision changes local folder
  name semantics.
- Reject empty normalized names.
- Reject names that exceed the local platform's safe length limit.
- Hash the exact normalized string that the local apply path will persist.

The same normalization function must be used for:

- `targetNameHash` verification
- duplicate sibling detection
- preStateHash/postStateHash computation
- apply audit
- applyEvent payload hashes

If a peer used an incompatible normalization version, the rename is stale or
blocked and must be regenerated.

## 7. Duplicate sibling rules

Duplicate-name ambiguity is a hard blocker for rename.

Before local apply:

- Resolve the target folder subject to exactly one local folder.
- Resolve the target parent to exactly one local parent context.
- Normalize the proposed target name.
- Scan siblings under the same parent.
- Exclude the subject folder itself.
- Block if any sibling has the same normalized name or same `nameHash`.
- Block if sibling parent identity is missing or ambiguous.
- Block if local folder data cannot prove uniqueness.

Duplicate checks must run during preflight and again inside the transaction.
No "best effort" rename may proceed when sibling uniqueness is unknown.

## 8. Rename vs move

Rename and move remain separate operations.

F10.9 v1 only allows:

```txt
changedFields: ["name"]
```

It must block or route to conflict review when any of these are present:

- `changedFields: ["parent"]`
- `changedFields: ["name", "parent"]`
- missing parent
- parent changed since base
- parent tombstoned
- parent subject mismatch

`rename-vs-move` is a conflict, not a merge opportunity. Field-level merging
may be reconsidered only after rename and move are independently proven.

## 9. Rename vs delete

Delete always wins the safety gate, not the state outcome.

If the subject is tombstoned, deleted, pending cleanup, or involved in an F5
delete/review/apply flow, rename must block and route to F5/F6 review. Rename
must not resurrect a deleted folder. Rename must not clear a tombstone.

`delete-vs-update` remains a hard blocker. A future operator may create a new
reviewed action, but F10.9 rename must not bypass F5.

## 10. Rename conflict taxonomy

| Conflict kind | Meaning | F10.9 action |
|---|---|---|
| `both-renamed-same` | Both peers changed to the same `targetNameHash`. | May be treated as already equivalent after local verification. |
| `both-renamed-different` | Both peers changed the same folder to different `targetNameHash` values. | Hard conflict; manual review required. |
| `rename-vs-move` | One peer renamed while another moved/reparented. | Hard conflict; no field-level merge in v1. |
| `rename-vs-delete` | One peer renamed while another deleted/tombstoned. | F5/F6 blocker. |
| `duplicate-folder-name` | Target name collides with a sibling. | Hard blocker. |
| `orphan-parent` | Parent context cannot be resolved. | Hard blocker. |
| `stale-base` | Local current hash no longer matches proposal base. | Re-preview/regenerate required. |
| `normalization-mismatch` | Hash cannot be verified under current normalization rules. | Hard blocker. |
| `raw-name-leak` | Envelope or relay artifact contains raw name. | Reject/quarantine. |

ConflictCandidate emission, if added later, must remain separate from apply.
ConflictCandidate is review evidence, not authority to rename.

## 11. Watermark interaction

Watermarks are not advanced by preview, proposal, conflictCandidate, or UI
review. A rename watermark may be recorded only after:

- local transactional rename committed successfully
- applyEvent receipt was built from the committed audit row
- consumed-operation row was recorded
- subjectId, lineageId, and postStateHash are valid

Watermark key:

```txt
peerId + subjectId + lineageId + revisionHash
```

Below-watermark evidence is replay/stale and must not generate a new rename
action. A watermark mismatch blocks and requires a fresh plan.

## 12. Replay interaction

Rename must use all existing replay and dedupe surfaces:

- `eventDigest`
- `dedupeKey`
- consumed-operation ledger
- convergence watermarks
- relay index replay/stale markers
- lineage ID
- operation ID

The rename dedupe key should include:

```txt
subjectId
operation: folder-metadata-rename
baseHash
targetNameHash
actorPeer.syncPeerIdHash
normalizationVersion
```

If an equivalent rename was already consumed, applied, superseded, blocked, or
expired, the new attempt must not mutate state. It may report duplicate,
replay, stale, or already-converged status.

## 13. Consumed ledger interaction

After successful local rename convergence, F10.9 bookkeeping must record a
consumed-operation row with:

- `eventDigest`
- `dedupeKey`
- `lineageId`
- `subjectId`
- `sourcePeerId`
- `envelopeKind`
- `operationKind: "folder-metadata-rename"`
- `consumedStatus: "consumed"`
- `actorPeer`
- validation summary

Failed, blocked, duplicate, expired, or replayed rename attempts may be
recorded as non-mutating ledger rows only if the row remains redacted. Consumed
ledger rows do not themselves advance watermarks.

## 14. Rollback model

Rollback before commit is transactional rollback.

Rollback after commit is not raw restore and not remote overwrite. It must be a
new compensating rename operation with:

- new operator approval
- fresh base hash
- duplicate sibling check
- F5/F6 checks
- new audit row
- new applyEvent
- new consumed-operation row
- new watermark row

The previous rename remains in audit history. The rollback action must not
erase the earlier applyEvent or ledger evidence.

## 15. Required preflight checks

Rename preflight must report booleans and blocker codes only. It must not
mutate state.

Required checks:

- planner entry is rename-eligible
- `changedFields` exactly equals `["name"]`
- `subjectId` resolves to exactly one local folder
- parent context resolves and has not changed
- operator-supplied target name is present
- normalized target name is non-empty and valid
- `sha256(normalizedTargetName) === targetNameHash`
- local current hash matches base hash
- target postStateHash is computable
- duplicate sibling name absent
- F5 tombstone/delete blockers absent
- F6 conflict blockers absent
- relay replay markers absent
- consumed-operation duplicate absent
- watermark stale/below-watermark markers absent
- peer identity and authority valid
- proposal not expired
- no forever-no-field leak in any returned object

Any unknown result blocks.

## 16. MUST / SHOULD / DEFER

### MUST

- Keep raw names device-local.
- Use `targetNameHash` for remote evidence.
- Define and version Unicode normalization before apply.
- Verify name hash before preflight becomes actionable.
- Reverify name hash inside the transaction.
- Allow exactly one subject per action.
- Allow exactly `changedFields: ["name"]`.
- Block duplicate sibling names.
- Block rename-vs-move.
- Block rename-vs-delete through F5/F6.
- Use replay, dedupe, consumed-ledger, and watermark gates.
- Emit applyEvent only after successful local commit.
- Treat applyEvent as evidence, never a remote command.
- Record consumed-operation and watermark rows only after successful local
  apply.

### SHOULD

- Show explicit UI copy that raw target name stays local.
- Display only shortened hashes in diagnostics.
- Provide a device-local preview of current name to target name when the user
  already sees both locally.
- Include normalization version in proposal, audit, and applyEvent payload
  hashes.
- Keep rename UI separate from color convergence UI until proven.
- Provide a clear duplicate-name explanation.

### DEFER

- Automatic rename.
- Batch rename.
- Mobile rename/write-back.
- Chrome/native rename apply.
- Remote apply.
- Rename plus move merge.
- Rename plus color merge.
- Folder create, move, delete, and binding convergence.
- Encrypted raw-name transport.
- ConflictCandidate auto-ingest.
- One-click rollback.

## 17. F10.9.x roadmap

| Phase | Scope | Mutation? |
|---|---|---:|
| F10.9.1 | Rename materialization diagnostic: local-only target name entry, normalization, and `targetNameHash` verification. | No |
| F10.9.2 | Rename preflight: duplicate sibling, base, F5/F6, replay, consumed, watermark, and parent checks. | No |
| F10.9.3 | Rename proposal candidate: redacted proposal envelope with `targetNameHash`, no raw name. | No |
| F10.9.4 | Desktop-local rename apply: one subject, one normalized target name, transactional audit. | Yes, local only |
| F10.9.5 | Rename applyEvent: past-tense receipt only after commit. | No |
| F10.9.6 | Rename bookkeeping: consumed-operation and watermark rows after successful apply. | No domain mutation |
| F10.9.7 | Manual rename UI: materialize, preflight, apply, and bookkeeping controls. | Uses existing gated APIs only |
| F10.9.8 | Runtime proof and dogfood validation. | Validation only |

F10.9 must remain Desktop-local until rename has the same audit, rollback, and
operator validation confidence that color convergence has today.
