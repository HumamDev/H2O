# F19.3 Live Chrome/Desktop Two-Surface Parity Proof Contract

Status: F19.3 proof harness contract.

## Architecture Verdict

F19.3 is a live/operator proof harness, not a new propagation feature. It verifies that the already-implemented F19.2 local sync-folder bridge can make Chrome Studio and Desktop Studio converge for the currently supported premium-sync fields.

Node validators cannot fully automate this because the proof needs two live surfaces:

- Chrome Studio running inside the MV3 extension context with File System Access permission for the shared sync folder.
- Desktop Studio running inside the Tauri runtime with the same sync folder configured.
- Real `latest.json` and `chrome-latest.json` movement through the local folder transport.
- Live `H2O.LibraryIndex` and catalog/folder stores on both surfaces.

F19.3 therefore has two layers:

- A default Node validator that proves the contract, schema, existing APIs, and redaction rules.
- A live manual/operator harness that captures Chrome and Desktop snapshots, runs both propagation directions, and records a machine-readable result.

Premium Sync is still not fully closed by F19.3 unless the live proof returns `ok:true` for supported fields and F19.4 conflict/offline/restart hardening is either closed or explicitly scoped as post-premium.

## Required Existing APIs

Chrome and Desktop must both expose:

```js
await H2O.Studio.sync.libraryParity.captureSnapshot()
await H2O.Studio.sync.libraryParity.runDiagnostic({ chromeSnapshot, desktopSnapshot })
```

Desktop must expose the F19.2.b Chrome -> Desktop receiver:

```js
await H2O.Studio.sync.importChromeLatestFromFolder()
await H2O.Studio.sync.importChromeLatestFromFile("/path/to/chrome-latest.json")
await H2O.Studio.sync.importChromeLatestBundle(bundle)
```

Chrome must expose the F19.2.c Desktop -> Chrome receiver:

```js
await H2O.Studio.sync.folder.syncNow()
await H2O.Studio.sync.folder.importLatestBundle(bundle)
```

## Transport Setup

Use one shared local sync folder that both surfaces can access.

Expected transport files:

- `chrome-latest.json`: Chrome-origin export consumed by Desktop.
- `latest.json`: Desktop-origin export consumed by Chrome.

The sync folder is transport only. It is not a source of truth, not a backend, and not a conflict resolver.

## Live Proof Flow

1. Configure both surfaces to use the same sync folder.
2. Capture an initial Chrome parity snapshot.
3. Capture an initial Desktop parity snapshot.
4. Compare initial snapshots with `runDiagnostic`.
5. Run Chrome -> Desktop propagation:
   - Chrome writes or refreshes `chrome-latest.json`.
   - Desktop imports `chrome-latest.json` through `importChromeLatestFromFolder()` or `importChromeLatestFromFile()`.
   - Desktop captures a new parity snapshot.
6. Run Desktop -> Chrome propagation:
   - Desktop writes or refreshes `latest.json`.
   - Chrome imports `latest.json` through `H2O.Studio.sync.folder.syncNow()`.
   - Chrome captures a new parity snapshot.
7. Compare final Chrome and Desktop snapshots.
8. Record the proof result with the F19.3 schema.

## Chrome Studio Console Steps

Run in the Chrome Studio console:

```js
const chromeBefore = await H2O.Studio.sync.libraryParity.captureSnapshot()
await H2O.Studio.sync.autoImport?.exportNow?.({ reason: "f19.3-live-proof" })
```

After Desktop imports `chrome-latest.json` and exports `latest.json`, run:

```js
const desktopToChrome = await H2O.Studio.sync.folder.syncNow({ reason: "f19.3-live-proof" })
const chromeAfter = await H2O.Studio.sync.libraryParity.captureSnapshot()
```

## Desktop Studio Console Steps

Run in the Desktop Studio console:

```js
const desktopBefore = await H2O.Studio.sync.libraryParity.captureSnapshot()
const initialParity = await H2O.Studio.sync.libraryParity.runDiagnostic({ chromeSnapshot: chromeBefore, desktopSnapshot: desktopBefore })
const chromeToDesktop = await H2O.Studio.sync.importChromeLatestFromFolder(null, { reason: "f19.3-live-proof" })
const desktopAfterChromeImport = await H2O.Studio.sync.libraryParity.captureSnapshot()
await H2O.Studio.sync.autoExport?.flushNow?.("f19.3-live-proof")
const desktopAfter = await H2O.Studio.sync.libraryParity.captureSnapshot()
```

After Chrome imports `latest.json`, compare:

```js
const finalParity = await H2O.Studio.sync.libraryParity.runDiagnostic({ chromeSnapshot: chromeAfter, desktopSnapshot: desktopAfter })
```

## Supported Parity Fields

F19.3 supported parity covers:

- total Library row count
- saved count
- linked count
- pinned count
- archived count
- folder count and folder hash
- category count and category hash
- recents hash
- row hash for supported chat/category/folder projection

## Deferred Fields

F19.3 must explicitly report these as deferred:

- labels
- tags
- projects
- chat-folder bindings
- tombstones
- sync apply events
- unsupported storage/KV fields

Deferred fields must never be silently treated as success. If the final F19.1 parity diagnostic reports mismatches only for deferred fields, the F19.3 proof may be `supported-parity-ready` but not `premium-sync-closed`.

## Pass / Fail Criteria

F19.3 passes for supported parity when:

- Chrome snapshot exists and is redacted.
- Desktop snapshot exists and is redacted.
- Chrome -> Desktop propagation result is `ok:true` or idempotent `already-imported`.
- Desktop -> Chrome propagation result is `ok:true` or idempotent `already-imported`.
- Final supported fields match.
- Final supported mismatch count is `0`.
- Privacy leak count is `0`.
- `latest.json` and `chrome-latest.json` are both observed in the shared transport.

F19.3 blocks when:

- either surface snapshot is missing;
- either propagation direction fails for supported fields;
- supported final parity mismatches remain;
- raw IDs, names, titles, folder names, category names, URLs, tokens, or private content appear in proof output;
- the sync folder is not shared by both surfaces;
- deferred fields are hidden rather than listed.

## Validator Strategy

`tools/validation/sync/validate-f19-live-parity-proof.mjs` should:

- assert this contract exists;
- assert F19.1, F19.2.b, and F19.2.c validators exist;
- assert live proof schema and manual console steps are documented;
- run a synthetic redacted proof against the F19.1 parity diagnostic in Node/VM;
- assert supported vs deferred mismatch partitioning;
- optionally validate an operator-supplied proof JSON with `--proof <path>`.

The default validator must not mutate real user data.

## Why F19.3 Still Is Not Full Premium Closure

F19.3 proves live two-surface convergence for the F19.2 supported subset. It does not yet prove:

- label/tag/project propagation;
- chat-folder binding propagation;
- stale-base conflict replay across real Chrome/Desktop sessions;
- offline/restart/permission-loss resilience;
- repeated delayed bidirectional loops under real user activity.

Those belong to F19.4 hardening and F19.5 closure evidence.
