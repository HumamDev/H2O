# F17.1 Secret / Config Audit Contract

## Architecture Verdict

F17.1 is a read-only secret/config regression guard. It locks in an already-good posture; it is not a remediation phase.

F17.0 under-scoped the secret surface as mostly local-first. F17.1 corrects that scope and explicitly includes:

- Supabase backend.
- Chrome MV3 extension / OAuth identity surface.
- Mobile app / Expo / billing surface.
- `config/local` and built extension artifacts.

The scanner should fail future release gates when a known-good invariant regresses: committed env files, hardcoded service-role keys, Stripe secrets, OAuth client secrets, private-key material, tracked local config, tracked built artifacts, loosened gitignore rules, or high-entropy credential-shaped tokens.

## Current Posture

Current repo posture is already good and should be preserved:

- Built extension artifacts are gitignored.
- `config/local/*.local.json` is gitignored.
- `config/extension-keys.json` contains RSA public keys and documents them as non-secret.
- Supabase Edge Functions read service-role keys, Stripe keys, and webhook secrets via `Deno.env.get(...)`.
- No hardcoded secrets should appear in tracked shipped source.

## Secret / Config Surface Table

| Surface | Secret/config concern | Expected posture |
|---|---|---|
| Tauri Studio | Local SQLite; no cloud secret expected | No Supabase/Stripe secret literals |
| Chrome extension source | OAuth/Supabase config references | Publishable IDs allowed; secret literals forbidden |
| Chrome extension built artifacts | Generated JS may contain anon key/URL | Gitignored; service-role and private secrets forbidden |
| `config/extension-keys.json` | RSA public keys for stable extension IDs | Allowlisted non-secret public keys |
| `config/local/*.local.json` | Local-only operator config | Gitignored; must not be tracked |
| Supabase functions | Service-role, Stripe, webhook secrets | Env-only via `Deno.env.get(...)` |
| Supabase migrations | SQL schema/RLS | No secrets expected |
| Mobile app | Expo/EAS/Supabase/Stripe billing surface | Public/project config may warn; secret literals forbidden |

## Blocker Taxonomy

Blockers make the release not ready:

- `secret-audit-committed-env-file`
- `secret-audit-hardcoded-service-role`
- `secret-audit-hardcoded-stripe-secret`
- `secret-audit-hardcoded-oauth-client-secret`
- `secret-audit-private-key-material`
- `secret-audit-tracked-build-artifact`
- `secret-audit-tracked-local-config`
- `secret-audit-gitignore-rule-removed`
- `secret-audit-generic-high-entropy-secret`

## Warning Taxonomy

Warnings require review but do not block by default:

- `secret-audit-expo-project-config-present`
- `secret-audit-supabase-anon-key-inline`
- `secret-audit-supabase-url-inline`
- `secret-audit-dev-endpoint-in-shippable`
- `secret-audit-allowlist-stale`

## Allowlist Policy

Allowlisted/non-secret items:

- `config/extension-keys.json` public keys.
- Pinned OAuth-Google extension IDs in identity validators.
- Supabase anon key/URL only in designated config files.
- `Deno.env.get(...)` references as the correct env-secret pattern.

Every allowlist entry should include a path, pattern, and reason. Stale allowlist entries should warn with `secret-audit-allowlist-stale`.

## Include / Exclude Plan

Default source scanning should include tracked files from `git ls-files`, especially:

- `src-runtime-base/`
- `src-surfaces-base/`
- `apps/`
- `tools/`
- `supabase/`
- `config/`
- root config and dotfiles

Default scanning should exclude:

- `node_modules/`
- gitignored build outputs
- binary/media files
- `target/`
- sibling archive/meta/changelog folders
- lockfiles unless the scanner design later chooses otherwise

## Scanner Strategy For F17.1.b

F17.1.b should implement three scanner passes:

1. Source pass:
   - Default and blocking.
   - Scans tracked files only.
   - Detects blocker and warning taxonomy patterns.
   - Consults the allowlist.
2. Built-artifact pass:
   - Opt-in with `--artifacts`.
   - Scans generated extension/artifact outputs when present.
   - Confirms shipped artifacts do not contain service-role, Stripe secret, OAuth client secret, private key, or local config material.
3. Gitignore-integrity pass:
   - Verifies env, local config, and build-output ignore rules remain present.
   - Blocks on removed ignore rules with `secret-audit-gitignore-rule-removed`.

The scanner should reuse existing validation patterns where practical, including `tools/validation/cross-platform/allowlist.json` conventions and cross-platform scan utilities such as `scan-helper-forbidden-patterns.mjs`.

## Risks And Mitigations

| Risk | Mitigation |
|---|---|
| False positives on public keys / anon keys | Curated allowlist with explicit reasons |
| Scanner misses built-artifact leaks | Opt-in `--artifacts` pass for generated outputs |
| Gitignore loosened later | Gitignore-integrity pass blocks removed rules |
| Entropy heuristic noise | Prefer credential-shaped patterns; keep entropy allowlist-backed |
| Scanner too slow | Default to tracked text files only |
| Allowlist becomes a dumping ground | Require reasons and warn on stale entries |
| Mobile/Supabase surfaces overlooked | Surface table explicitly includes both |
| Real secret already in git history | Manual history scan and rotation before production |

Git history scanning and secret rotation are manual pre-production operator steps. F17.1 automates HEAD/source/artifact regression checks only.
