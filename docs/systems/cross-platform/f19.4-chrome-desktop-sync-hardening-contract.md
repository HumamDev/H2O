# F19.4 Chrome/Desktop Sync Hardening Contract

Status: F19.4 hardening contract.

## Architecture Verdict

F19.4 hardens the existing local sync-folder transport between Chrome Studio and Desktop Studio. It does not introduce a backend, identity UI, direct native bridge, signing work, or a new source of truth.

The sync folder remains transport only:

- Chrome-origin payload: `chrome-latest.json`.
- Desktop-origin payload: `latest.json`.
- Desktop guarded import: `H2O.Studio.sync.importChromeLatestFromFile()` / `importChromeLatestFromFolder()`.
- Chrome guarded import: `H2O.Studio.sync.folder.syncNow()` / `importLatestBundle()`.

The goal is realistic premium behavior under offline, restart, permission-loss, stale replay, malformed file, and simultaneous update conditions. F19.4 does not declare Premium Sync complete; closure still requires live two-surface parity proof evidence for supported fields.

## Failure Taxonomy

| Code | Meaning | Required behavior |
|---|---|---|
| `sync-folder-missing` | Operator has not connected a sync folder or Desktop folder path is missing. | Do not mutate; report reconnect/setup action. |
| `permission-denied` | Browser File System Access permission is missing or revoked. | Do not mutate; ask operator to reconnect folder. |
| `transport-file-missing` | Expected `latest.json` or `chrome-latest.json` cannot be read. | Do not mutate; retry after peer exports. |
| `transport-file-malformed` | Transport JSON cannot be parsed. | Fail closed before import. |
| `transport-schema-unsupported` | Transport schema is unsupported. | Fail closed before import. |
| `transport-stale` | Incoming export is older than the last applied export for that direction. | Block unless it is the exact duplicate already applied. |
| `duplicate-import-idempotent` | Same content/export is imported again. | Treat as safe no-op. |
| `local-newer-conflict` | Local state is newer than incoming transport. | Block or require refresh/re-export before apply. |
| `simultaneous-update-conflict` | Incoming `previousExportId` does not match the last applied export. | Block; do not silently merge divergent branches. |
| `deferred-field-present` | Payload contains a recognized field not supported in F19.2 propagation. | Warn explicitly; do not count as synced. |
| `unsupported-field-present` | Payload contains unsupported storage/KV fields. | Warn explicitly; strip from guarded import. |
| `source-metadata-missing` | Payload lacks peer/workspace/source metadata. | Warn; Premium Sync closure must require known source metadata. |
| `parity-peer-snapshot-required` | Live parity proof lacks one peer snapshot. | Fail proof; do not claim convergence. |

The older `library-propagation-*` codes remain for compatibility. F19.4 appends the normalized codes above so validators and operator reports have stable cross-direction taxonomy.

## Offline and Restart Behavior

| Scenario | Expected behavior |
|---|---|
| Chrome offline while Desktop exports | `latest.json` remains pending. Chrome imports it on next manual/focus/boot-triggered sync if permission is still granted. |
| Desktop offline while Chrome exports | `chrome-latest.json` remains pending. Desktop imports it after restart or manual import. |
| Repeated import after restart | Same fingerprint/checksum is `duplicate-import-idempotent`; no duplicate rows should be created. |
| Peer exports while other side closed | Latest file persists; no background daemon is required for F19.4. |
| Transport write crash | `.tmp` and partial download suffixes are ignored where applicable; malformed final JSON fails closed. |

## Stale Replay and Conflict Behavior

Each side records the last applied export metadata for its inbound direction:

- content fingerprint or checksum,
- `exportId`,
- `previousExportId`,
- `sequenceNumber` when present,
- `exportedAt`,
- import timestamp.

Incoming transport is blocked when:

- `exportedAt` is older than the last applied export and the file is not an exact duplicate;
- `previousExportId` is present and does not match the last applied `exportId`;
- the schema is unsupported or the JSON is malformed.

F19.4 does not invent a separate protected-domain conflict model. Desktop protected catalog and binding domains continue to depend on F15/F16 runtime conflict and guarded settlement paths. Chrome-side guarded import remains merge-only and strips unsupported domains until those domains are explicitly implemented.

## Permission-Loss Behavior

Chrome uses the File System Access API. If the stored directory handle loses permission, `syncNow()` must return `permission-denied` / `sync-folder-reconnect-required`, preserve the stored handle state, and avoid import mutation.

Desktop path reads use Tauri file APIs. Missing folder path or unreadable `chrome-latest.json` must return `sync-folder-missing` or `transport-file-missing` without mutation.

## Deferred Fields

These fields remain explicit warnings and are not silently counted as synced:

- labels,
- tags,
- projects,
- chat-folder bindings,
- tombstones,
- sync apply events,
- unsupported Chrome storage / KV fields.

Legacy per-field warning codes remain:

- `library-propagation-labels-deferred`,
- `library-propagation-tags-deferred`,
- `library-propagation-projects-deferred`,
- `library-propagation-chat-folder-bindings-deferred`,
- `library-propagation-tombstones-deferred`,
- `library-propagation-apply-events-deferred`,
- `library-propagation-unsupported-storage-deferred`.

Supported F19.2 fields remain:

- saved chat records,
- linked chat records,
- folder metadata,
- category metadata,
- chat-category bindings where the existing bundle shape supports them.

## Privacy and Proof Safety

F19.4 diagnostics, validators, and evidence must remain hash-only/redacted. They must not emit raw:

- chat IDs,
- snapshot IDs,
- catalog IDs,
- folder IDs,
- project IDs,
- names,
- titles,
- folder names/colors,
- paths,
- URLs/tokens,
- content/messages/attachments,
- `category_id`,
- `chats.category_id`,
- `folder_id`,
- `chat_id`.

Validator/proof mode must use VM fixtures and synthetic payloads only. It must not mutate real user data or real sync folders.

## Validator Strategy

`tools/validation/sync/validate-f19-sync-hardening.mjs` proves:

- the contract exists;
- both propagation directions expose the normalized F19.4 taxonomy;
- malformed, unsupported-schema, missing-folder, permission-loss, stale, conflict, duplicate, deferred-field, unsupported-field, source-metadata, and missing-peer-snapshot cases are represented;
- Desktop `chrome-latest.json` import records export metadata in its ledger;
- Chrome `latest.json` import records `lastAppliedExportedAt`;
- VM fixture imports remain privacy-safe and no real data paths are touched.

Existing validators remain required:

```bash
node tools/validation/sync/validate-f19-live-parity-proof.mjs
node tools/validation/sync/validate-f19-desktop-chrome-propagation.mjs
node tools/validation/sync/validate-f19-chrome-desktop-propagation.mjs
node tools/validation/sync/validate-f19-chrome-desktop-library-parity.mjs
npm run gate:library
npm run validate:build
npm run audit:secrets
npm run validate:migration -- --restore-drill
```

## Remaining Before Premium Sync Closure

F19.5 can close Premium Sync only after live Chrome and Desktop surfaces demonstrate:

- supported fields converge through `latest.json` and `chrome-latest.json`;
- deferred fields are still reported as deferred or are implemented in a later phase;
- offline/restart/idempotent/stale/conflict/permission-loss behavior is proven;
- F19 live parity proof returns `ok:true` for supported fields;
- no privacy leaks or unintended proof-mode writes occur.
