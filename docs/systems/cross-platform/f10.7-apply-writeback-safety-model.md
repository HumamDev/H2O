# F10.7.0 - Apply and write-back safety model

> **Contract document. F10.7.0 is docs-only.** No apply implementation, no
> WebDAV implementation, no mobile write-back, no automatic merge, and no
> runtime behavior change exists in this phase.

This document defines the safety boundary that must exist before any
cross-platform mutation or write-back code is authorized.

## 1. Scope and status

| | |
|---|---|
| Phase | **F10.7.0** |
| Deliverable | This document only |
| Runtime code | **None** |
| Apply implementation | **None** |
| WebDAV/cloud implementation | **None** |
| Mobile write-back | **None** |
| Automatic merge | **None** |
| Remote apply | **Forbidden** |
| Status | Authoritative pre-implementation safety contract |

F10.1 through F10.6 established the platform-capability model and the
read-only preview/report pipeline:

```
canonical snapshot -> diff -> proposal-preview -> conflict report
```

F10.7 is the first phase that will eventually authorize mutations. This
document exists before mutation code so the apply surface cannot grow by
accident.

## 2. Architectural principle

**`applyEvent` is evidence of a local apply. It is not a remote apply command.**

An `applyEvent` says:

```
this platform already committed a local, audited mutation
```

It never means:

```
remote platform, perform this mutation now
```

Remote envelopes are observations, previews, proposals, conflict candidates, or
past-tense apply receipts. They do not directly mutate local state. Convergence
is locally authorized: a receiving platform may use remote evidence to build a
new preview, proposal, or conflict-review item, but local apply requires local
gates and local approval.

Transport is not authority. WebDAV, file bundles, Chrome extension messages,
native runtime mirrors, and mobile picker files can carry envelopes, but they
cannot make an envelope executable.

## 3. Authority transitions

Every mutation must advance through this sequence without skipping steps:

```
preview -> proposal -> conflict review -> apply approval -> apply -> applyEvent emission
```

| Stage | Meaning | Can mutate? | Required output |
|---|---|---:|---|
| `preview` | Dry-run diagnostic of a possible state change. | No | `kind:"preview"`, `dryRun:true` |
| `proposal` | Explicit request to consider a change. | No | `kind:"proposal"` |
| `conflict review` | Human or local-authority review of conflict/gate status. | No | F6-compatible review decision or refusal |
| `apply approval` | Local decision that all predicates pass and mutation may run. | No | audit-ready operation plan |
| `apply` | Transactional local mutation under audited gates. | Yes, local only | local durable state update |
| `applyEvent emission` | Past-tense receipt after successful local commit. | No | `kind:"applyEvent"` |

No platform may interpret receipt of `proposal`, `conflictCandidate`, or
`applyEvent` as permission to apply.

## 4. Platform authority table

| Platform | Preview | Propose | Approve | Apply | Emit `applyEvent` |
|---|---|---|---|---|---|
| Chrome Studio | Yes, as `preview-coordinator`. | Future only, after proposal-source authorization. | No. | No. | No. |
| Desktop Studio | Yes. | Yes. | Yes, through local gates. | Yes, gated only. | Yes, after successful local commit only. |
| Native extension | Limited evidence-derived preview only. | No. | No. | No. | No. |
| Mobile | Yes, read-only preview. | Future proposal-only, not enabled now. | No. | No. | No. |

Desktop Studio is the first strong local authority because it owns the durable
local database, audited apply paths, transaction boundaries, and rollback
surface. Chrome Studio coordinates previews and diagnostics. The native
extension produces evidence. Mobile remains read-first and may become a
proposal source only after a later safety phase.

## 5. First safe mutation scope

The first safe F10.7 mutation scope is **folder color only**.

| Scope | F10.7 treatment | Reason |
|---|---|---|
| `folders.color` / canonical `iconColor || color` | First apply target. | Already proven by F7 as exact-gated, low-content, local-only metadata. |
| Folder rename | Later. | Names carry user meaning and duplicate-name ambiguity; needs stronger review. |
| Folder create | Later. | Requires identity allocation, parent validation, duplicate checks, and ordering semantics. |
| Folder move/reparent | Later. | Parent/orphan/cycle risk; can change library structure substantially. |
| Folder delete | Deferred. | Destructive and tombstone-owned; must pass F5 gates. |
| `folderBinding` changes | Deferred. | Membership semantics can move or detach chats; v1 remains diff/preview-only. |
| Labels/categories/chats/snapshots | Out of scope. | Broader identity and content/privacy risks. |

Color-only is intentionally narrow. It allows the first end-to-end apply model
to prove ledger, audit, stale-base, dedupe, and event emission without mixing
destructive or membership semantics.

## 6. Required safety gates

An apply path must fail closed if any required gate cannot be evaluated.

| Gate | Requirement |
|---|---|
| Stale-base verification | Local current `revisionHash` or local parity hash must match the proposal base. If not, re-preview is required. |
| Exact destructive gate | Destructive operations require exact subject, exact base, exact predicate, and explicit local approval. |
| Tombstone verification (F5) | Deletes, delete-vs-update cases, and cleanup operations must pass F5 tombstone/review/apply rules. |
| Conflict verification (F6) | Existing or newly detected conflicts must be reviewed through F6-compatible review flow before apply. |
| Duplicate-name verification | Rename/create must block if sibling name ambiguity exists. |
| Orphan-parent verification | Move/create must block when parent is missing, tombstoned, or ambiguous. |
| Proposal expiration | Expired proposals cannot apply; they must be regenerated from fresh evidence. |
| Replay protection | Previously consumed operation IDs, dedupe keys, or event digests must not apply again. |
| Dedupe | Idempotent repeats must be detected and reported as already applied or duplicate, not re-run. |
| Peer identity verification | Actor peer envelope must validate against F2/F10.1 identity and capability manifest. |
| Audit logging | Every attempted apply must record enough audit detail to prove why it passed or failed. |
| Transaction boundaries | State mutation, audit ledger write, and local result recording must commit or roll back together. |

These gates are cumulative. Passing one gate never waives another.

## 7. Apply ledger model

The apply ledger is append-only. It records attempted and successful local
applies; it is not a queue of remote commands.

Minimum ledger fields:

| Field | Meaning |
|---|---|
| `operationId` | Stable unique ID for the reviewed local operation. |
| `actorPeer` | Redacted peer envelope for the local approving/applying actor. |
| `sourceProposalId` | Optional envelope ID or preview/proposal reference that motivated the local apply. |
| `subjectType` / `subjectId` | Redacted subject being changed. |
| `operation` / `operationIntent` | Specific operation and categorical intent. |
| `baseHash` | Hash the proposal was evaluated against. |
| `preStateHash` | Local state hash immediately before mutation. |
| `postStateHash` | Local state hash immediately after mutation. |
| `predicateVersion` | Version of the gate predicate used. |
| `transactionId` | Local transaction identifier tying mutation and audit together. |
| `dedupeKey` | Deterministic idempotency key. |
| `resultState` | `committed`, `rejected`, `rolled-back`, `duplicate`, or `failed`. |
| `warnings` / `blockers` | Code strings only; no raw content. |
| `createdAtIso` / `completedAtIso` | Advisory timestamps, never authority by themselves. |

The ledger must preserve failed and rejected attempts when doing so is safe and
redacted. Successful commits must be durable before an `applyEvent` is emitted.

## 8. `applyEvent` rules

An `applyEvent` is emitted only after a successful local commit.

| Rule | Requirement |
|---|---|
| Past tense only | The event reports a completed local mutation. |
| Never a command | Consumers must not dispatch mutations from `applyEvent`. |
| No propagation loops | Receiving an `applyEvent` may update diagnostics or dedupe state, but must not emit another applyEvent unless a separate local apply occurred. |
| Dedupe | `dedupeKey` and `eventDigest` must prevent duplicate interpretation. |
| Event digest | `eventDigest` is a digest of the canonical envelope, excluding warnings/blockers/eventDigest per F10.2 rules. |
| Redaction | No raw folder names by default; no raw chat IDs; no content fields; no file paths; no URLs; no tokens. |
| Subject scope | `subjectType`, `subjectId`, and operation intent must match the local ledger entry. |
| Transaction proof | Payload must cite local `operationId`, `transactionId`, `preStateHash`, and `postStateHash`. |

If a platform receives an `applyEvent` for an entity whose local state differs,
it must treat that as evidence for preview/conflict analysis, not as a command
to overwrite local state.

## 9. Rollback and recovery model

Rollback and recovery are local authority functions.

| Mechanism | Rule |
|---|---|
| Transaction rollback | If mutation or audit write fails before commit, the full transaction rolls back. |
| Compensating inverse operation | If an already-committed change must be reversed, create a new reviewed operation with its own ledger row. |
| No raw restore overwrite | Do not restore by blindly replacing a whole row/table/bundle from remote data. |
| Audit consistency checks | Ledger, state hash, and emitted `applyEvent` must agree. Mismatch blocks propagation and requires manual review. |
| Partial failure handling | A partial or ambiguous commit state must block event emission until resolved. |

Rollback is not "last writer wins." Recovery must preserve audit history and
avoid hiding the failed operation that caused it.

## 10. WebDAV and cloud timing

Before apply exists, transport may safely relay only:

- evidence
- preview envelopes
- proposal-preview diagnostics
- report-only conflict summaries
- metadata-only cache/status summaries
- bundles that carry the above as inert data

Transport must wait for a later apply phase before relaying:

- `proposal` envelopes intended for local review
- `conflictCandidate` envelopes intended for F6 ingest
- `applyEvent` receipts

Even after these are allowed, relay does not imply execution. WebDAV/cloud is
a carrier, not a coordinator and not an authority. Mutation and transport must
stay decoupled so network delivery cannot bypass local safety gates.

## 11. Forbidden behaviors

These are forbidden for all F10.7.x work unless a later document explicitly
changes the contract:

- Remote-triggered apply.
- Automatic merge.
- Silent overwrite.
- Last-writer-wins shortcuts.
- Mobile write-back.
- Direct mobile apply.
- Proposal bypass.
- Applying from `preview` or `conflict report`.
- Applying from `applyEvent` receipt.
- Skipping F5 tombstone/delete checks.
- Skipping F6 conflict/review checks.
- Skipping stale-base checks.
- Skipping duplicate-name or orphan-parent checks for structure changes.
- Treating transport delivery as approval.
- Treating cache metadata as source state.
- Full bundle restore as a sync strategy.
- Raw content, raw chat IDs, raw paths, URLs, or secrets in apply payloads.

| Phase | Scope | Mutation status |
|---|---|---|
| F10.7.1 | Color apply | First local gated apply path. |
| F10.7.2 | `applyEvent` emission | Past-tense receipt after local commit. |
| F10.7.3 | Rename apply | Requires duplicate-name and stale-base hardening. |

No phase may skip the apply ledger, transaction, redaction, stale-base, dedupe,
and audit requirements in this document.

## 13. MUST / SHOULD / DEFER tables

### MUST

| Requirement | Reason |
|---|---|
| Keep `applyEvent` as past-tense local evidence only. | Prevents remote command execution. |
| Apply only through local audited gates. | Preserves authority boundary. |
| Start with folder color only. | Keeps first mutation low-risk and F7-aligned. |
| Verify stale base before apply. | Prevents overwriting newer local state. |
| Use append-only apply ledger. | Enables audit, dedupe, rollback, and event proof. |
| Emit `applyEvent` only after successful commit. | Prevents false receipts. |
| Fail closed on missing gate data. | Avoids unsafe best-effort writes. |
| Keep cache and transport non-authoritative. | Prevents hidden sync authority. |
| Preserve F10.2 validator strictness. | Keeps envelope privacy and capability checks enforceable. |

### SHOULD

| Recommendation | Reason |
|---|---|
| Keep proposal, review, apply, and event emission in separate modules. | Clear rollback and audit boundaries. |
| Maintain deterministic operation IDs/dedupe keys. | Idempotent retry and replay protection. |
| Record rejected attempts when safe. | Improves diagnostics without mutating state. |
| Add operator-visible blockers before write controls. | Keeps UX aligned with safety model. |
| Validate with real data before broadening scope. | Prevents model drift from synthetic-only tests. |

### DEFER

| Item | Deferred until |
|---|---|
| Mobile proposal source | A mobile proposal safety model exists. |
| Mobile write-back | Separate authorization after proposal and review are proven. |
| WebDAV/cloud apply relay | Transport hardening and local apply are proven independently. |
| Folder rename/create/move/delete apply | Color apply, ledger, and applyEvent are proven. |
| Folder binding apply | Membership semantics and delete/move gates are designed. |
| Automatic convergence | Manual/local gated apply model is production-proven. |

## 14. Done definition for F10.7.0

F10.7.0 is done when this document is committed and no runtime behavior has
changed.

The next implementation phase must cite this document and prove that it does
not turn remote envelopes, transport delivery, cache data, or mobile previews
into write authority.
