# F16.0 - Production Hardening / Runtime Conflict Resolution / Soak Plan

**Status:** Architecture plan. F16.0 is docs-only.
**Predecessor closure:** F15 closed at `df556373d484f500736fbffd3cff54a8175f351e` (`test(sync): add library conflict resolution proof`).
**Scope:** Production hardening, runtime conflict-resolution planning, multi-peer soak, performance stress, and release gating for the F15 library sync system.
**Non-scope:** Runtime implementation, validators, store changes, SQLite trigger changes, F16.1 implementation, F7 deletion, and product feature expansion.

## 1. Architecture Verdict

F16 is a production-hardening phase, not a new product-feature phase.

F15 is structurally closed. It delivered the `library.catalog` and `library.binding` sync pipeline, protected SQLite cutover, bulk migration path, end-to-end proof closure, operator status UI, F7 folder binding absorption, and the F15.12 cross-install conflict contract plus proof.

F15.12 conflict handling is currently contract/proof-driven. F16 should turn those proof fixtures into runtime gates, then prove the resulting system under real multi-peer, offline/replay, and soak conditions.

F16 must preserve the F14/F15 architectural constraints:

- Fail closed on unsafe conflict state.
- Refresh and re-propose from current state.
- No field-level auto-merge.
- No multi-subject proposal primitive.
- No direct store write re-enable.
- No bypass of the execute lane.
- Cache remains a materialized read view, never conflict authority.
- F7 folder binding fallback remains available until a later trigger/fallback decision proves safe.

## 3. Runtime Conflict Implementation Plan

F16.1 should introduce a runtime conflict gate, likely:

`src-surfaces-base/studio/sync/library/library-conflict-runtime.tauri.js`

The gate should be a shared runtime validator/composer, not an automatic merge engine. It should return structured blocker/warning results consumed by catalog and binding preflight/settlement paths.

### 3.1 Catalog gates

| Conflict | Runtime behavior |
|---|---|
| Catalog stale base | Compare expected current state/base hash against settled state; block stale loser with `library-catalog-cross-install-stale-base` |
| Duplicate `nameHash` | Enforce active uniqueness by account, catalogKind, and nameHash; block collision with `library-catalog-cross-install-name-collision` |
| Rename/recolor conflict | No field-level merge; stale base blocks and operator refreshes/re-proposes |
| Lifecycle conflict | Enforce legal lifecycle transition from current state; block with `library-catalog-cross-install-lifecycle-conflict` |
| F5 terminal conflict | First terminal closure wins; same closure idempotent; conflicting closure blocks with `library-catalog-f5-review-conflict` |
| Tombstone vs new binding | New bind to non-active catalog blocks; existing bindings surface warnings only |

### 3.2 Binding gates

| Conflict | Runtime behavior |
|---|---|
| Duplicate binding edge | Enforce uniqueness by bindingKind and endpoints; exact duplicate idempotent, conflicting duplicate blocks |
| Bind vs unbind race | Compare expected edge state; stale loser blocks or noops according to settled state |
| Chat-category one-active edge | Enforce one active chat-category binding per chat; replacement remains unbind plus bind |
| Chat-folder one-active edge | Enforce one active chat-folder binding per chat using `folder.metadata` endpoint |
| Chat-folder replacement race | Surface partial unbind/bind state; do not emit a single replace proposal |
| F7/F15 identity bridge mismatch | Use migration shadow identity; warning if consistent, blocker if active folder state conflicts |

### 3.3 Bulk and cache gates

| Conflict | Runtime behavior |
|---|---|
| Bulk per-row conflict | Classify each row independently through normal catalog/binding gates |
| Bulk partial failure | Return partial status with retry-safe row identity and no duplicates |
| Cache drift | Warning only; `library.binding` remains source of truth |
| Cache refresh after replay | Refresh from binding state; never infer binding conflict from `chats.category_id` |

### 3.4 Runtime gate contract

The runtime conflict gate must:

- Fail closed on unknown or unsafe states.
- Preserve F15.12 blocker/warning vocabulary.
- Return operator-visible refresh/retry guidance.
- Never mutate storage directly.
- Never bypass execute settlement.
- Never publish, relay, apply, or call Native/F5 directly.
- Never re-enable direct legacy store writes.
- Never introduce field-level auto-merge.
- Never introduce a multi-subject kernel primitive.

## 4. Multi-Peer Soak Matrix

F16.2 should add a multi-peer simulator / soak harness that models at least two installs with separate peer identities, offline queues, delayed replay, and deterministic clocks.

| Scenario | Setup | Expected result |
|---|---|---|
| Two installs create same label/tag/category | Peer A and peer B create same catalogKind/nameHash while offline | One valid settlement wins; loser blocks with name collision |
| Offline rename vs online rename | Peer A renames offline; peer B renames and settles first | Peer A stale base blocks and refreshes |
| Recolor vs archive | One peer recolors while another archives | First valid settlement wins; stale loser blocks |
| Duplicate chat-label/chat-tag binds | Two peers bind same edge | Exact replay/idempotent result or duplicate-edge block |
| Bind vs unbind same edge | Peer A binds; peer B unbinds stale edge | Stale loser blocks or noops from settled state |
| Chat-category replacement race | Two peers replace same chat category | Unbind/bind partial state visible; no single replace proposal |
| Chat-folder replacement race | Two peers move same chat folder | Uses `folder.metadata`; F7/F15 bridge warning/blocker as needed |
| Delayed F5 approve-seal vs approve-restore | Peer decisions arrive in opposite order | First terminal closure wins; conflicting closure blocks |
| Bulk import while peer edits catalog | Peer A imports bundle; peer B edits catalog/bindings | Per-row conflict classification and partial retry-safe result |
| Cache drift after reconnect | Peer A settles binding; peer B cache is stale | Cache drift warning and refresh; cache does not decide conflict |
| Folder fallback vs F15 delegation | Flag-off legacy fallback and flag-on delegated path both exercised | Fallback remains compatible; delegated path stays F15-owned |

## 5. Performance Stress Plan

F16.3 should separate production-scale stress from the normal closure proof so day-to-day validation remains fast while release validation proves scale.

| Target | Stress expectation |
|---|---|
| 1k chats | Runtime proof and operator summaries remain responsive |
| 10k chats | Store reads, cache drift scan, and binding lookups remain bounded/indexed |
| 100+ labels/tags/categories | Catalog duplicate-name and lifecycle gates stay hash/index based |
| 1k+ bindings | Binding duplicate-edge checks and settlement remain chunkable |
| Large bundle import | Bulk import chunks, reports partial failures, and retries without duplicates |
| Proof runtime budget | Normal F15/F16 closure proof remains operator-practical; heavy stress runs separately |
| UI render budget | Operator UI renders summaries first and lazy-loads detail rows |
| SQLite transaction budget | Writer identity scope is short; chunked transactions avoid long locks |

Suggested budgets for F16.3 planning:

- Normal closure proof: target under 30 seconds in Desktop runtime.
- Multi-peer soak smoke: target under 2 minutes.
- Heavy performance stress: target under 10 minutes, release-gate only.
- Operator UI initial render: target under 1 second for summary state.
- Bulk chunk size: preserve bounded chunking and expose per-chunk timing.

## 6. `folder_bindings` Trigger Decision Path

Do not protect `folder_bindings` immediately in F16.0 or F16.1.

Revisit after F16.2 multi-peer soak proves both:

- F7 fallback remains stable and compatible.
- F15 delegated chat-folder path remains stable and compatible.

Decision options:

| Option | When to choose |
|---|---|
| Keep deferred | F7 fallback remains necessary or soak reveals divergence |
| Add scoped fallback identity | F7 fallback must continue writing after protection; identity could be `f16.folder-legacy-fallback` |
| Make F15 delegation default-on | Soak proves delegated path is safer than fallback for normal use |
| Protect `folder_bindings` | F15 path is proven and fallback either authorized or retired |
| Retire fallback later | F16+ / F17 after telemetry/proofs show no dependency |

Protection must not ship until the fallback behavior is explicitly decided. Triggering protection too early can break the compatibility rule from F15.11.

## 9. Risk Table

| Risk | Impact | Mitigation |
|---|---|---|
| Protected SQLite failures in normal use | Store writes fail for legitimate legacy callers | Soak store shims with real call paths before adding more triggers |
| Store shim timeout under load | Operators see pending or partial state too often | Add timeout telemetry, bounded pending states, and retry guidance |
| Cache drift after offline replay | UI appears inconsistent across installs | Keep cache warning-only and add refresh proof |
| F5 race across peers | Catalog tombstone decision can appear inconsistent | Enforce first-terminal-wins runtime gate and duplicate closure idempotency |
| F7 fallback divergence | Folder UI and F15 chat-folder state disagree | Keep delegation guarded; compare migration shadow identities during soak |
| Proof/UI hiding runtime failures | Static proof passes while runtime path fails | Add Desktop runtime proof runner and stale-proof UI state |
| Bulk partial failure confusion | Operators cannot distinguish settled/skipped/failed rows | Return per-row partial result, deterministic retry identity, and copyable report |
| Long SQLite writer identity scope | Lock contention or accidental authorization leakage | Keep identity scoped to short transactions and prove clear-on-error |
| Runtime conflict gate overreach | Gate becomes auto-resolver or new merge primitive | Keep gate validation-only and preserve fail-closed semantics |
