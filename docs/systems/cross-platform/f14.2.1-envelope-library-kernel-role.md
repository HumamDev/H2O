# F14.2.1 - Envelope library kernel role

> **Docs-only kernel role definition.** F14.2.1 introduces no runtime code, no
> envelope refactor, no kernel implementation, no domain refactor, no
> Chat/Snapshot/Capture implementation, no remote apply, no automatic merge,
> and no mobile write-back.

F14.2.0 defined the convergence kernel contract. The first L0 primitive
candidate is the existing F10.2 cross-platform envelope library because it is
already the shared narrow waist between domain-specific state and transport.

This document formalizes the envelope library's kernel role without changing
its behavior.

## 1. Why the envelope library is already kernel-shaped

The F10.2 envelope contract is already kernel-shaped for five reasons:

- It defines one shared outer shape for evidence, preview, proposal,
  conflictCandidate, applyEvent, bundle, and cacheMetadata.
- It separates envelope structure from domain payload semantics.
- It enforces platform identity, authority, capability, lifecycle kind, and
  redaction posture before consumers interpret payloads.
- It treats transports as opaque carriers that add no authority.
- It states that envelopes are records or requests, never commands.

Those are L0 properties. They belong below folder, Chat, Snapshot, Capture,
label, category, and future domain policy.

The envelope library should therefore become the canonical L0 entry point for
cross-platform message shape and authority validation. It must not become a
convergence engine.

## 2. Current envelope responsibilities

The current envelope contract already owns these responsibilities:

| Responsibility | Current role |
|---|---|
| Schema identity | Locks `h2o.crossPlatform.envelope.v1`, wrapper version, and kind version. |
| Envelope kind | Defines `evidence`, `preview`, `proposal`, `conflictCandidate`, `applyEvent`, `bundle`, and `cacheMetadata`. |
| Platform identity | Carries `sourcePlatform.platformId`, `surfaceKind`, and redacted peer envelope. |
| Authority split | Separates `declaredAuthority` from `effectiveAuthority`. |
| Capability attribution | Records `capabilityUsed` and `capabilitySnapshotHash`. |
| Subject attribution | Carries redacted `subjectType` and `subjectId`. |
| Operation posture | Carries `operation`, optional `operationIntent`, `dryRun`, and `transactional`. |
| Idempotency | Carries `dedupeKey`, `payloadHash`, and `eventDigest`. |
| Redaction posture | Carries `redactionClass` and rejects forbidden payload fields. |
| Diagnostics | Carries blocker and warning code arrays. |

These responsibilities are mechanical and cross-domain. They do not decide how
a folder is renamed, how a chat body is materialized, how a snapshot restores,
or how a capture is retained.

## 3. Kernel responsibilities

As an L0 kernel primitive, the envelope library is responsible for:

- Validating the envelope base shape.
- Validating kind-specific required fields.
- Validating kind posture, including `dryRun`, `transactional`, and
  `operationIntent`.
- Validating source platform, surface kind, declared authority, effective
  authority, and capability compatibility.
- Checking platform-kind allow rules before any owner handoff or publication.
- Validating redaction class and forbidden fields at the envelope boundary.
- Validating digest fields and deterministic identity fields.
- Returning explicit blockers and warnings without silent correction.
- Providing stable constants and predicates that L1 and L2 can depend on.

The envelope library may say:

```txt
this envelope is structurally valid
this envelope is authorized for this producing platform and kind
this envelope is blocked because it violates authority or privacy rules
```

The envelope library must not say:

```txt
this proposal should apply
this applyEvent should mutate local state
this conflict is resolved
this domain object should be repaired
```

## 4. Relationship to domains

### Chat

Chat convergence must use the envelope library for source identity, authority,
redaction class, replay-facing digest fields, and kind validation. Chat policy
still owns canonical chat body shape, message/edit materialization, content
privacy, conflict taxonomy, recovery, and apply.

The envelope library must not authorize raw chat content in cross-platform
envelopes.

### Snapshot

Snapshot convergence must use the envelope library for restore proposal,
applyEvent receipt, custody, expiration, and digest structure. Snapshot policy
still owns restore semantics, retention windows, recovery evidence, body
hashing, and rollback constraints.

The envelope library must not treat a snapshot envelope as restore authority.

### Capture

Capture convergence must use the envelope library for evidence identity,
origin/custody attribution, redaction posture, publication eligibility, and
eventDigest/dedupeKey handling. Capture policy still owns what observed
metadata can be materialized and what content remains evidence-only.

The envelope library must not turn capture evidence into a proposal.

### Folder domains

Folder color, rename, move, delete, and binding already emit and consume
envelope-shaped artifacts. Existing folder lanes may adopt the envelope library
opportunistically, but no shipped folder lane should be force-rewritten as part
of F14.2.1.

Folder policy remains L2:

- color apply
- rename materialization and duplicate checks
- move tree/cycle/orphan checks
- delete F5 handoff and tombstone ordering
- binding identity/cardinality checks

The envelope library validates the wrapper. Folder domains validate the
operation.

## 5. Allowed usage rules

Allowed uses:

- Constructing envelope base wrappers from already-approved domain inputs.
- Validating envelope base and kind shape before publication, inbox storage,
  proposal generation, conflictCandidate generation, or applyEvent emission.
- Validating that a platform is allowed to produce a specific envelope kind.
- Validating that `operationIntent` is present or absent for the kind.
- Computing or verifying payload hashes and event digests when the caller
  supplies canonical payload input.
- Checking redaction class and forbidden payload fields before return or store.
- Producing blocker and warning lists.
- Supporting read-only diagnostics, proof harnesses, and publication gates.

Allowed consumers:

- L1 publication kit.
- L1 replay-defense composer.
- L1 owner-handoff primitive.
- L1 lifecycle reducers.
- L2 domain proposal generators.
- L2 domain applyEvent builders.
- Relay inbox/outbox validators.
- Manual review and proof surfaces.

## 6. Forbidden usage rules

Forbidden uses:

- Calling any domain apply function.
- Writing domain rows.
- Creating tombstones.
- Creating conflict review rows.
- Advancing watermarks.
- Recording consumed operations.
- Enqueueing relay outbox rows by itself.
- Uploading or downloading relay blobs.
- Interpreting `applyEvent` as a remote command.
- Converting evidence into preview, proposal, or conflictCandidate
  automatically.
- Converting proposal or conflictCandidate into apply automatically.
- Repairing malformed envelopes by silently rewriting fields.
- Downgrading privacy by stripping fields and then accepting the envelope as if
  it had arrived valid.
- Granting authority from payload content, transport location, or relay
  custody.

Invalid envelopes are rejected, blocked, or quarantined by the caller. They are
not transformed into valid commands.

## 7. Required invariants

The envelope library must preserve these invariants:

| Invariant | Requirement |
|---|---|
| Envelope is not command | Receipt of any kind never directly mutates state. |
| Kind is lifecycle posture | Kind determines validation and display posture, not execution. |
| Authority is explicit | Consumers act on validated effective authority, never on declared authority alone. |
| Capability is checked | `capabilityUsed` must be compatible with platform, surface, and kind. |
| Transport adds no authority | Relay, WebDAV, file picker, Chrome storage, and local bundles carry envelopes opaquely. |
| Redaction is fail-closed | Raw identifiers, content, paths, URLs, credentials, and forbidden fields block or quarantine. |
| Digest fields are stable | `dedupeKey`, `payloadHash`, and `eventDigest` must be deterministic and verifiable. |
| Expired is stale | Expired envelopes may be displayed but cannot seed apply or review without revalidation. |
| applyEvent is past tense | `applyEvent` only records a completed local apply and never triggers remote apply. |
| Cache is off-pipeline | `cacheMetadata` cannot feed proposal, review, or apply. |

## 8. Authority validation role

The envelope library is the first authority gate, not the final domain gate.

It validates:

- `sourcePlatform.platformId`
- `sourcePlatform.surfaceKind`
- redacted peer envelope shape
- `declaredAuthority`
- computed or supplied `effectiveAuthority`
- `capabilityUsed`
- `capabilitySnapshotHash`
- platform-kind allow rules
- kind-specific authority requirements

It does not validate:

- whether a folder has duplicate siblings
- whether a chat edit is safe
- whether a snapshot can be restored
- whether capture evidence can be materialized
- whether an operator approved a local apply
- whether a transaction committed

Those are L2 checks.

The authority result from the envelope library is necessary but never
sufficient for mutation.

## 9. Relationship to PLATFORM_KIND_ALLOW

`PLATFORM_KIND_ALLOW` is the conceptual platform-kind production matrix. It
must answer this question:

```txt
May this platform produce this envelope kind at all?
```

The current helper constants already express this through the platform allowed
kinds table. F14.2.x may standardize naming later, but F14.2.1 does not rename
or change code.

The matrix must remain deny-by-default:

| Platform | Current envelope production posture |
|---|---|
| Desktop Studio | May produce evidence, preview, proposal, conflictCandidate, applyEvent, bundle, cacheMetadata when domain gates permit. |
| Chrome Studio | May produce evidence and preview only unless a later phase explicitly upgrades authority. |
| Native extension | May produce evidence/cacheMetadata within native evidence scope only. |
| Mobile | May produce evidence/cacheMetadata only; no proposal or applyEvent. |

Owner handoff must check this matrix before accepting proposal,
conflictCandidate, or applyEvent-producing paths. A valid payload cannot bypass
a platform-kind denial.

## 10. Relationship to publication infrastructure

Publication turns a local generated artifact into a relay-visible artifact. It
does not apply anything.

The envelope library supports publication by validating:

- schema and kind
- source platform and peer envelope
- capability and authority posture
- required kind payload fields
- digest fields
- redaction class
- forever-no fields
- expiration posture

Publication infrastructure owns:

- generated-to-published lifecycle
- publication ledger rows
- outbox enqueue calls
- duplicate publication checks
- operator publication approval

The envelope library must not append publication ledger rows or enqueue outbox
entries by itself.

## 11. Relationship to replay defense

The envelope library provides the fields replay defense needs:

- `id`
- `lineageId`
- `sequence`
- `exportSequence`
- `expiresAt`
- `dedupeKey`
- `payloadHash`
- `eventDigest`
- source peer envelope
- source platform identity

Replay defense composes those fields with:

- relay index
- publication ledger
- consumed-operation ledger
- convergence watermarks
- lineage ledgers
- domain-specific stale-base checks

The envelope library validates the field shape. L1 replay-defense composer
decides whether an event is duplicate, replay, stale, expired, below watermark,
or oscillating. L2 domains decide whether the still-valid event is actionable.

## 12. Relationship to privacy scanning

The envelope library is the first privacy boundary for cross-platform artifacts.

It must reject or block:

- raw chat IDs
- raw folder IDs
- raw snapshot IDs
- raw capture IDs
- chat titles
- folder names at `redacted`
- content bodies
- messages
- attachment data
- paths
- URLs
- credentials
- tokens, except explicitly allowed envelope fields such as preview token
  patterns already gated by the envelope contract

F14.2.2 will define the dedicated forever-no scanner and redaction-class
enforcer. Until then, the envelope library remains the place where envelope
privacy requirements are documented and validated.

The envelope library may validate that a payload is redacted. It must not
silently redact an invalid payload and accept it as valid.

## 13. Migration strategy

F14.2.1 is documentation only. Migration happens later and must be incremental:

1. Keep the current F10.2 envelope helper package and validation scripts as the
   reference behavior.
2. Define the envelope library's L0 role before moving any helper.
3. Add or expose kernel wrappers only in a later implementation phase.
4. Compare wrapper output to existing validators before adoption.
5. Adopt the envelope helper in new Chat/Snapshot/Capture code before forcing
   folder lane migrations.
6. Migrate existing folder domain emitters opportunistically, one domain and one
   envelope kind at a time.
7. Preserve public output shapes and blocker codes during migration.

No migration may change which platforms can produce proposal, conflictCandidate,
or applyEvent envelopes.

## 14. Acceptance criteria

For F14.2.1:

- One docs-only file is added.
- No runtime code changes.
- No helper package changes.
- No domain refactor.
- No Chat/Snapshot/Capture implementation.
- No label/category implementation.
- `git diff --check` passes.
- `git diff --cached --check` passes.

For later envelope-library kernel implementation phases:

- Existing envelope fixtures remain valid.
- Existing blocker codes remain stable unless a versioned spec change is
  explicitly approved.
- `validateEnvelopeBase`, `validateEnvelopeKind`, and authority validation
  behavior remain parity-checked against current output.
- `PLATFORM_KIND_ALLOW` behavior remains deny-by-default.
- The cross-platform envelope validator passes.
- The cross-platform repo scan passes.
- Runtime consumers do not gain new write paths.
- No domain apply path is added or widened by envelope helper adoption.

## 15. MUST / SHOULD / DEFER

### MUST

- Keep F14.2.1 docs-only.
- Treat the envelope library as L0, not L1 or L2.
- Preserve the envelope as a record/request contract, not a command protocol.
- Validate platform-kind authority before owner handoff.
- Keep `applyEvent` evidence-only and past-tense.
- Keep publication and transport separate from mutation.
- Keep privacy checks fail-closed.
- Preserve existing platform authority boundaries.

### SHOULD

- Use envelope helpers as the first shared kernel wrapper once implementation is
  authorized.
- Add parity tests before replacing copied envelope validation in domain
  modules.
- Prefer new Chat/Snapshot/Capture code as the first adopters.
- Keep wrapper names explicit enough that callers cannot confuse validation
  with authorization to apply.
- Surface envelope validation blockers directly in manual review UI.

### DEFER

- Runtime envelope refactor.
- Generic envelope builder for every domain.
- Automatic proposal or conflictCandidate generation.
- Automatic publication.
- Automatic apply.
- Remote apply.
- Mobile write-back.
- Chat/Snapshot/Capture mutation.
- Folder lane rewrites.
- Ledger schema migrations.

## 16. F14.2.1 status

F14.2.1 defines the envelope library's L0 kernel role only. No code is changed,
no helper is moved, no runtime behavior changes, and no domain behavior is
modified.
