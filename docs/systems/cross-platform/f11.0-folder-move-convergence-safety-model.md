# F11.0.0 - Folder move convergence safety model

> **Contract document. F11.0.0 is docs-only.** No folder move implementation,
> apply change, mobile write-back, remote apply, automatic move, or runtime
> behavior change exists in this phase.

This document defines the safety boundary for folder move/reparent convergence
after color and rename convergence. Move is higher risk than rename because it
changes tree structure, can orphan descendants, can create cycles, and can
silently change chat/folder context for every descendant.

## 1. Scope and status

| | |
|---|---|
| Phase | **F11.0.0** |
| Deliverable | This document only |
| Runtime code | **None** |
| Move apply implementation | **None** |
| Mobile write-back | **None** |
| Remote apply | **Forbidden** |
| Automatic move | **Forbidden** |
| Status | Authoritative pre-implementation safety contract |

F11.0 may eventually allow Desktop Studio to perform one locally approved
folder move convergence action. It must not allow any platform to remotely
reparent a folder by sending an envelope.

## 2. Move/reparent convergence lifecycle

Move convergence must follow this sequence without skipping stages:

```txt
remote observed evidence
-> redacted targetParentSubjectId
-> local parent materialization/resolution
-> tree safety validation
-> move preflight
-> operator approval
-> local transactional reparent
-> local applyEvent receipt
-> consumed-operation row
-> convergence watermark row
```

Each stage is fail-closed. Receipt of a remote envelope is never authority to
move a folder. UI confirmation is not enough; the transaction must repeat all
tree safety checks immediately before mutation.

## 3. Folder tree model

Move convergence must reason over the local folder tree, not isolated rows.

Required concepts:

- `folderId`: device-local durable folder row ID. It never leaves the device in
  relay-visible artifacts.
- `subjectId`: redacted hash for a folder subject.
- `parentId`: device-local parent folder ID, or `null`/empty for root.
- `parentSubjectId`: redacted hash for the parent folder subject, or `null` for
  root.
- `descendants`: all folders reachable below the subject through parent links.
- `ancestors`: all folders reachable upward from the subject through parent
  links.

Remote artifacts may carry `targetParentSubjectId` and structural hashes. They
must not carry raw local `folderId` or `parentId`.

## 4. Parent validation

Before move preflight can become actionable:

- Resolve the subject folder to exactly one local folder.
- Resolve the target parent subject to exactly one local folder when moving
  under a parent.
- Allow move-to-root only when `targetParentSubjectId` is explicitly `null`.
- Verify the target parent is live, not tombstoned, not pending cleanup, and
  not involved in an unresolved delete/review path.
- Verify the target parent has not changed since the base snapshot.
- Verify the source parent has not changed since the base snapshot unless the
  planner explicitly classified that exact move lineage as current.
- Refuse if parent identity is missing, ambiguous, stale, or untrusted.

Parent validation must run during diagnostic, preflight, and inside the local
transaction.

## 5. Orphan prevention

Move convergence must never leave the tree in an orphaned state.

Hard blockers:

- Target parent does not exist.
- Target parent is deleted or tombstoned.
- Target parent cannot be resolved from `targetParentSubjectId`.
- Current parent cannot be resolved when required for base verification.
- Any ancestor walk encounters a missing parent before root.
- Any child/descendant scan cannot prove the moved subtree remains connected.

If tree integrity cannot be proven locally, the move must block and route to
manual review. No repair, parent guess, or best-effort reparent is allowed in
F11.0.

## 6. Cycle prevention

Cycle prevention is mandatory.

Required blockers:

- `self-parent`: subject folder cannot become its own parent.
- `descendant-parent`: subject folder cannot move under any descendant.
- `cycle-detected`: any local ancestor walk repeats a folder ID or subject.
- `ancestor-walk-incomplete`: local ancestry cannot be fully resolved.

The apply transaction must compute the descendant set for the subject and the
ancestor walk for the target parent using current local state. It must fail
closed if either walk is incomplete.

## 7. Depth bound

Move convergence must define a maximum folder-tree depth before implementation.

Minimum contract:

- Define `MAX_FOLDER_TREE_DEPTH` in the implementation phase.
- Count depth after the proposed move.
- Count the moved subtree's deepest descendant depth relative to the new parent.
- Block if the resulting depth is unknown.
- Block if the resulting depth exceeds `MAX_FOLDER_TREE_DEPTH`.
- Block if current tree depth already exceeds the bound and the move cannot
  prove it reduces risk.

Depth checks prevent pathological trees and accidental cycles that appear as
very deep chains.

## 8. Duplicate sibling name rule under new parent

Move may create a duplicate sibling name even when the folder name itself is
unchanged.

Before local apply:

- Resolve the moving folder's current normalized name or `nameHash`.
- Resolve the target parent context.
- Scan siblings under the target parent.
- Exclude the moving folder itself.
- Block if any target sibling has the same normalized name or same `nameHash`.
- Block if sibling identity, parent identity, or normalized name hashing is
  unavailable.

Duplicate checks must run during preflight and again inside the transaction.
The rule applies to move-to-root as well as move-under-parent.

## 9. Move vs rename

Move and rename remain separate operations.

F11.0 v1 only allows:

```txt
changedFields: ["parent"]
```

It must block or route to conflict review when any of these are present:

- `changedFields: ["name"]`
- `changedFields: ["name", "parent"]`
- target parent has a duplicate sibling name conflict
- local name differs from the name used in the remote move evidence
- name normalization version mismatch

`rename-vs-move` is a conflict, not a merge opportunity. A future combined
rename+move flow may be designed only after standalone move is proven.

## 10. Move vs delete / F5 tombstone ownership

Delete safety owns move safety.

Move must block if:

- the subject is tombstoned, deleted, pending cleanup, or inside an F5
  delete/review/apply flow
- the source parent is tombstoned or deleted
- the target parent is tombstoned or deleted
- any required ancestor is tombstoned or deleted
- the move would place a live folder under a deleted/tombstoned parent

Move must not resurrect a deleted folder, clear tombstones, or bypass F5. Any
move/delete overlap routes to F5/F6 review.

## 11. Move vs binding

FolderBinding changes are not move changes.

F11.0 v1 must not:

- move chats between folders
- create or remove folder bindings
- infer binding changes from a folder move
- treat a binding-only remote event as a folder move
- apply a folder move when binding state is concurrently conflicting

If bindings reference the moving folder, they remain attached to the same
folder subject. Binding convergence remains a separate future lane.

## 12. Watermark interaction

Watermarks are not advanced by preview, proposal, conflictCandidate, UI review,
or preflight. A move watermark may be recorded only after:

- local transactional reparent committed successfully
- applyEvent receipt was built from the committed audit row
- consumed-operation row was recorded
- subjectId, lineageId, and postStateHash are valid
- tree integrity was rechecked after commit

Watermark key:

```txt
peerId + subjectId + lineageId + revisionHash
```

Below-watermark evidence is replay/stale and must not generate a new move
action. A watermark mismatch blocks and requires a fresh plan.

## 13. Replay / dedupe / oscillation guards

Move must use all existing replay and dedupe surfaces:

- `eventDigest`
- `dedupeKey`
- consumed-operation ledger
- convergence watermarks
- relay index replay/stale markers
- lineage ID
- operation ID
- source parent hash
- target parent hash

The move dedupe key should include:

```txt
subjectId
operation: folder.move
baseHash
sourceParentSubjectId
targetParentSubjectId
actorPeer.syncPeerIdHash
treePredicateVersion
```

Oscillation guard:

- Block immediate move-back if it reverses a recently consumed applyEvent from
  the same lineage window.
- Block A -> B -> A ping-pong unless a fresh operator-approved lineage and
  base hash proves intent.
- Treat repeated inverse lineage as conflict/review, not auto-convergence.

## 14. Rollback / repair model

Rollback before commit is transaction rollback.

Rollback after commit is not raw restore and not remote overwrite. It must be a
new compensating move operation with:

- new operator approval
- fresh base hash
- full parent validation
- full cycle/depth checks
- duplicate sibling check under the rollback parent
- F5/F6 checks
- new audit row
- new applyEvent
- new consumed-operation row
- new watermark row

Repair model:

- Orphan/cycle repair is not part of F11.0 v1.
- If a pre-existing broken tree is detected, move convergence blocks.
- A separate operator-guided repair tool may be designed later.

## 15. Required preflight checks

Move preflight must report booleans and blocker codes only. It must not mutate
state.

Required checks:

- planner entry is move-eligible
- `changedFields` exactly equals `["parent"]`
- `subjectId` resolves to exactly one local folder
- source parent context resolves and matches base
- target parent is either explicitly root/null or resolves to exactly one live
  local folder
- target parent stable since base
- target parent not the subject itself
- target parent not a descendant of the subject
- ancestor walk complete and cycle-free
- descendant walk complete and cycle-free
- resulting depth known and within `MAX_FOLDER_TREE_DEPTH`
- duplicate sibling name absent under target parent/root
- local current hash matches base hash
- target postStateHash is computable
- F5 tombstone/delete blockers absent for subject, source parent, target parent,
  and required ancestors
- F6 conflict blockers absent
- relay replay markers absent
- consumed-operation duplicate absent
- watermark stale/below-watermark markers absent
- peer identity and authority valid
- proposal not expired
- no forever-no-field leak in any returned object

Any unknown result blocks.

## 16. MUST / SHOULD / DEFER

### MUST

- Keep raw folder IDs and parent IDs device-local.
- Use `targetParentSubjectId` for remote evidence.
- Allow exactly one subject per action.
- Allow exactly `changedFields: ["parent"]`.
- Revalidate parent existence, liveness, and stability inside the transaction.
- Block move-to-self.
- Block move-under-descendant.
- Block unknown ancestry, unknown descendants, and any detected cycle.
- Define and enforce maximum tree depth before move apply.
- Block duplicate sibling names under the target parent/root.
- Block move-vs-rename in v1.
- Block move-vs-delete through F5/F6.
- Keep folderBinding changes out of move convergence.
- Use replay, dedupe, consumed-ledger, watermark, and oscillation gates.
- Emit applyEvent only after successful local commit.
- Treat applyEvent as evidence, never a remote command.
- Record consumed-operation and watermark rows only after successful local
  apply and post-commit tree verification.

### SHOULD

- Show explicit UI copy that move is local and operator-approved.
- Display only shortened subject/parent hashes in diagnostics.
- Provide a tree preview with counts and hashes, not raw IDs.
- Include `treePredicateVersion` in proposal, audit, and applyEvent payload
  hashes.
- Keep move UI separate from color and rename convergence UI until proven.
- Provide clear duplicate-name, cycle, depth, orphan, and tombstone explanations.
- Require manual refresh after a blocked move rather than automatic retry.

### DEFER

- Automatic move.
- Batch move.
- Mobile move/write-back.
- Chrome/native move apply.
- Remote apply.
- Move plus rename merge.
- Move plus binding merge.
- Move plus delete resolution.
- Folder create, delete, and binding convergence.
- Automatic orphan/cycle repair.
- Encrypted raw-tree transport.
- ConflictCandidate auto-ingest.
- One-click rollback.

## 17. F11.0.x roadmap

| Phase | Scope | Mutation? |
|---|---|---:|
| F11.0.1 | Move materialization diagnostic: resolve target parent/root, validate liveness, duplicate sibling, cycle/depth risk, and redacted parent hashes. | No |
| F11.0.2 | Move convergence preflight: base, parent stability, F5/F6, replay, consumed, watermark, orphan, cycle, and depth checks. | No |
| F11.0.3 | Move proposal candidate generation: redacted proposal envelope with `targetParentSubjectId`, no raw IDs. | No |
| F11.0.4 | Move convergence action + applyEvent: one Desktop-local transactional reparent and past-tense receipt. | Yes, local only |
| F11.0.5 | Move bookkeeping + proof: consumed-operation, watermark, post-commit tree verification, and dogfood proof harness. | No domain mutation beyond existing apply |
| F11.0.6 | Manual move convergence UI: materialize target parent, preflight, apply, and bookkeeping controls. | Uses existing gated APIs only |

F11.0 must remain Desktop-local until move convergence has the same audit,
rollback, runtime proof, and operator validation confidence that color and
rename convergence have today.
