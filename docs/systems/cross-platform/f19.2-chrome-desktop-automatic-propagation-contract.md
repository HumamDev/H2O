# F19.2 Chrome/Desktop Automatic Propagation Contract

Status: F19.2.a design contract.

## Architecture Verdict

F19.2 should use the existing local sync-folder bridge as the first automatic Chrome Studio <-> Desktop Studio propagation transport. This is the smallest safe path because the repo already has both directions:

- Chrome Studio can write `chrome-latest.json` through `src-surfaces-base/studio/sync/auto-import.mv3.js`.
- Desktop Studio can read `chrome-latest.json` through `src-surfaces-base/studio/sync/folder-sync.tauri.js` and `src-surfaces-base/studio/sync/focus-import.tauri.js`.
- Desktop Studio can write `latest.json` through `src-surfaces-base/studio/sync/auto-export.tauri.js` and `H2O.Studio.ingestion.exportLatestSyncBundle()`.
- Chrome Studio can read `latest.json` through `src-surfaces-base/studio/sync/folder-import.mv3.js`.

The sync folder is transport only. It is not a source of truth, not a backend, and not a conflict resolver.

F19.2 should not require a new backend or identity UI before the first premium-sync propagation path. Backend identity can become a later production-grade relay, but requiring it now would mix a product/account lane into the sync architecture lane. Direct extension-to-Desktop native messaging is also deferred because no complete installed bridge currently exists and it adds packaging, permission, and lifecycle risk.

The chosen source-of-truth strategy is local-first bidirectional replication with domain-specific authority:

- Chrome Studio remains the local authority for Chrome-origin Library writes until those writes are exported.
- Desktop Studio remains the local authority for SQLite-backed Desktop writes until those writes are exported.
- Desktop SQLite is the guarded settlement authority when applying Chrome-origin changes to protected F15/F16 Library catalog and binding domains.
- Chrome storage/archive is the Chrome-side local authority when applying Desktop-origin changes.
- The F19.1 parity diagnostic is the acceptance proof for convergence, not the apply mechanism.

Premium Sync is not complete after F19.2.a. Completion requires automatic propagation in both directions plus a live Chrome/Desktop parity proof that returns `ok:true`.

## Propagation Strategy

### Transport

Use the operator-selected local sync folder for F19.2. The folder contains two top-level latest files:

- `chrome-latest.json`: Chrome-origin export for Desktop to consume.
- `latest.json`: Desktop-origin export for Chrome to consume.

Both writers must keep atomic tmp-then-rename behavior. Both readers must keep SHA-256/file-fingerprint dedupe. Neither side should delete peer files as part of F19.2.

### Propagation Envelope

F19.2.b/c should formalize a propagation envelope, either as an additive section in the existing `h2o.studio.fullBundle.v2` bundle or as a new sidecar payload. The recommended schema is:

`h2o.studio.sync.chrome-desktop-propagation.v1`

Required fields:

- `schema`
- `version`
- `direction`: `chrome-to-desktop` or `desktop-to-chrome`
- `transport`: `local-sync-folder`
- `sourceSurface`
- `targetSurface`
- `sourcePeerEnvelope`
- `workspaceScopeHash`
- `exportId`
- `sequenceNumber`
- `previousExportId`
- `contentSha256`
- `createdAtIso`
- `librarySummary`
- `catalogDeltas`
- `bindingDeltas`
- `chatSnapshotDeltas`
- `folderStateDeltas`
- `paritySnapshotHash`

The propagation file may contain product data needed to sync the user's Library because it is the local sync payload. Validators, diagnostics, release evidence, and parity reports must remain hash-only/redacted and must never copy raw payload values into logs or evidence bundles.

## Chrome -> Desktop Flow

1. Chrome detects a completed Library change from a safe post-write event, manual export action, focus/visibility import result, or explicit F19 trigger.
2. Chrome produces `chrome-latest.json` using the existing sync-folder handle and atomic `chrome-latest.json.tmp` staging.
3. The payload carries source peer metadata, workspace scope, content hash, export id, sequence, and a parity snapshot hash.
4. Desktop detects the file through manual scan, focus import, or later watcher mode.
5. Desktop verifies schema, source peer, workspace scope, size, hash, and replay ledger before any write.
6. Desktop applies chats/snapshots with merge-only import semantics.
7. Desktop applies labels/tags/categories and chat-label/chat-tag/chat-category/chat-folder bindings through F15/F16 protected paths:
   - use catalog/binding canonicalizers;
   - call `evaluateLibraryRuntimeConflict`;
   - require conflict context when settlement is required;
   - use bulk row classification for bundle-sized imports;
   - fail closed before consumed-op, watermark, bookkeeping, cache, publication, journal, relay/outbox, Native/F5, or SQL side effects when a blocker exists.
8. Desktop refreshes `H2O.LibraryIndex` after a successful apply.
9. Desktop captures an F19.1 parity snapshot and reports any mismatch without auto-repair loops.

F19.2.b should be limited to this Chrome -> Desktop lane.

## Desktop -> Chrome Flow

1. Desktop detects a completed SQLite-backed Library change from store subscribers or explicit operator action.
2. Desktop produces `latest.json` via `H2O.Studio.ingestion.exportLatestSyncBundle()` and `sync/auto-export.tauri.js`.
3. The payload carries source peer metadata, workspace scope, content hash, export id, sequence, and a parity snapshot hash.
4. Chrome reads `latest.json` through `folder-import.mv3.js` manual or opt-in focus/visibility import.
5. Chrome verifies schema, source peer, workspace scope, size, hash, and replay ledger before any write.
6. Chrome applies import data through its existing archive/Library import path and preserves linked-only vs saved semantics.
7. Chrome refreshes `H2O.LibraryIndex`.
8. Chrome captures an F19.1 parity snapshot and reports any mismatch.

F19.2.c should be limited to this Desktop -> Chrome lane.

## Conflict Safety

Protected Library domains must reuse F15/F16. New F19 propagation code must not introduce a separate conflict model.

Required Desktop apply gates:

- `H2O.Desktop.Sync.evaluateLibraryRuntimeConflict`
- `H2O.Desktop.Sync.evaluateLibraryCatalogRuntimeConflict`
- `H2O.Desktop.Sync.evaluateLibraryBindingRuntimeConflict`
- `H2O.Desktop.Sync.classifyLibraryBulkRuntimeConflictRows`
- F15 catalog and binding canonicalizers
- F15/F16 settlement writer and side-effect guards where settlement is required

Required outcomes:

- Exact replay is idempotent.
- Stale-base replay blocks or requests refresh/re-propose.
- Same-name cross-install catalog creation blocks with the existing taxonomy.
- Duplicate bindings are idempotent or block with the existing duplicate-edge taxonomy.
- Bind/unbind and replacement races fail closed.
- Cache drift remains warning-only.
- Conflict runtime unavailable during required apply blocks.
- Blocked rows do not write SQL, consumed-op, watermark, bookkeeping, cache, publication, journal, relay/outbox, Native, or F5 side effects.

## Identity, Workspace, and Session Scope

F19.2 should not implement identity UI, but propagation must still be scoped.

Minimum required metadata:

- source surface kind (`chrome-studio` or `desktop-studio`)
- source install hash
- source device hash
- source sync peer hash when available
- workspace scope hash
- export id and content hash
- sequence number and previous export id when available

If peer identity or workspace scope is unknown:

- read-only parity diagnostics may warn;
- automatic mutation should be disabled or require explicit operator opt-in;
- premium-complete proof must require known peer/workspace scope.

This prevents a shared folder containing unrelated exports from silently mutating the wrong local Library.

## Offline, Restart, and Reload Behavior

F19.2 propagation is local-first and file-backed:

- If Chrome is offline or closed, Desktop-origin `latest.json` remains available until Chrome next opens and imports it.
- If Desktop is offline or closed, Chrome-origin `chrome-latest.json` remains available until Desktop next opens and imports it.
- Repeated imports are deduped by content SHA-256/export id.
- Out-of-order imports compare sequence/previous export metadata and should warn or block if the prior export has not been consumed.
- Stale-base protected catalog/binding changes use F16 runtime conflict blockers.
- Crash during write is handled by tmp suffix filtering and atomic rename.
- Restart should never require clearing ledgers to converge.

## Privacy and Evidence Boundary

Propagation files are local user data and may contain the Library payload required for sync. Proofs and reports must not.

Hash-only/redacted outputs are required for:

- F19.1 parity diagnostics
- F19.2 propagation diagnostics
- validators
- release evidence
- runtime proof summaries

Forbidden in diagnostics/evidence:

- raw chat IDs
- raw snapshot IDs
- raw catalog IDs
- raw folder IDs
- raw project IDs
- names
- titles
- folder names/colors
- paths
- URLs/tokens
- content/messages/attachments
- `category_id`
- `chats.category_id`
- `folder_id`
- `chat_id`

## Acceptance Proof

F19 premium sync acceptance must use the F19.1 parity APIs as the final gate:

```js
await H2O.Studio.sync.libraryParity.captureSnapshot()
H2O.Studio.sync.libraryParity.compareSnapshots(chromeSnapshot, desktopSnapshot)
await H2O.Studio.sync.libraryParity.runDiagnostic({ chromeSnapshot, desktopSnapshot })
```

Acceptance requires:

- Chrome snapshot source available.
- Desktop snapshot source available.
- Same workspace scope.
- Same peer pair.
- Total/saved/linked/pinned/archived counts match.
- Folder, label, category, project, and recents hashes match.
- No privacy leaks in parity output.
- No protected-domain side-effect violation in proof mode.

| F19.2.b | Minimal Chrome -> Desktop propagation | Chrome change exports `chrome-latest.json`; Desktop imports it automatically when opted in; Desktop protected catalog/binding changes route through F15/F16 conflict gates; Desktop parity snapshot updates. |

## F19.2.b Minimal Chrome -> Desktop Scope

F19.2.b implements the first Chrome -> Desktop receiver as a deliberately narrow bridge:

- Desktop accepts `chrome-latest.json` as the only F19 Chrome-origin transport filename.
- Desktop treats the file as an existing `h2o.studio.fullBundle.v2` payload and normalizes it into a supported subset before import.
- Supported fields are saved chat records, linked chat records, folder metadata, category metadata, and chat-category bindings where carried on the chat organization shape.
- Folder metadata may create missing folders but must not overwrite existing Desktop folder metadata in this phase.
- Labels, tags, projects, and chat-folder bindings are explicitly deferred and must surface warning taxonomy instead of being silently ignored.
- The public propagation result uses `h2o.studio.sync.chrome-desktop-propagation.v1` and must be hash-only/redacted.
- Desktop import must pass `allowLibraryShimFallback: false` and preserve F15/F16 guarded bulk/conflict behavior where available.
- Repeated identical files are idempotent through the existing SHA-256 ledger.

F19.2.b still does not close Premium Sync. It only moves supported Chrome-origin rows toward Desktop. Full closure requires Desktop -> Chrome propagation and live two-surface parity proof.

F19.2.b warning taxonomy:

- `library-propagation-labels-deferred`
- `library-propagation-tags-deferred`
- `library-propagation-projects-deferred`
- `library-propagation-chat-folder-bindings-deferred`
- `library-propagation-unsupported-storage-deferred`
- `library-propagation-source-metadata-missing`

## Blockers and Dependencies

| Dependency | Why it matters | F19.2 posture |
|---|---|---|
| Sync folder configured on both surfaces | Existing transport requires user-selected folder access | Required for local transport phase. |
| Chrome File System Access permission | Chrome cannot write/read sync-folder files without a page-context permission grant | Required; no background-only sync in F19.2. |
| Desktop folder path configured | Desktop needs a known folder path for `scanFolderOnce()` | Required for Desktop receiver. |
| Peer/workspace scope | Prevents applying unrelated user/workspace exports | Required before default automatic mutation. |
| F15/F16 runtime conflict gates | Protected Library domains must not bypass settled conflict safety | Required for Desktop protected-domain apply. |
| Live two-surface runner | Node validators cannot prove real Chrome storage + Desktop SQLite parity | Required before premium closure. |
| Backend identity/relay | Needed for cloud multi-device sync beyond local folder | Deferred; not required for F19.2 local premium path. |

## Why Prior Proofs Still Do Not Close Premium Sync

F15/F16/F17 prove local kernels, Desktop runtime health, simulated multi-peer safety, migration safety, packaging readiness, and release evidence. They do not prove that Chrome Studio and Desktop Studio automatically exchange live user Library changes and converge after reload/restart.

F19.2.a only defines the propagation contract. Premium Sync remains open until F19.2.b/c implement both directions and F19.3 proves live parity.
