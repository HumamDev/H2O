# F15.0.2 — Materialized Cache Contract for `chats.category_id`

**Status:** Architecture design — final for §§1–11, provisional only for §6.4 (cross-install settlement reordering edge cases; full contract in F15.6).
**Predecessors:** F15.0.0 at `cf681fc`, F15.0.1 at `e8e7db6`.
**Scope:** Doc only. No runtime code. No store-layer changes. No Labels/Categories/Tags implementation. Does not begin F15.0.3.
**Authoritative position:** F15.0.2 settles the cache contract deferred by F15.0.1 §7.5. It does not supersede F15.0.0 or F15.0.1 on any other point.

---

## 1. Executive Summary

This doc defines exactly how `chats.category_id` stops being source-of-truth and becomes a materialized read cache derived from `library.binding` state.

Top-level decisions:

| Decision | Verdict | Status |
|---|---|---|
| Source of truth for chat-category assignment | **`library.binding` lane** — the active `chat-category` binding for that chat | **final** |
| `chats.category_id` SQL column | **Materialized read cache only**, never source of truth | **final** |
| Allowed writers to `chats.category_id` | **Exactly one path** — F14.6 `execute-settlement-writer` after a `library.binding` chat-category apply event lands | **final** |
| Forbidden writers | chat.metadata canonicalizer / proposals / receipts; `store/chats.tauri.js` direct writes; any module other than the settlement writer | **final** |
| Cache rebuild trigger algorithm | **Event-driven with bounded fallbacks** — settlement-writer synchronous refresh + boot/resume backfill + periodic drift sweep + manual operator refresh. **No general polling.** | **final** |
| Read-your-own-write semantics | **YES** for local applies — settlement writer refreshes cache **synchronously** before returning the settlement result | **final** |
| Cross-install consistency | **Eventually consistent** — best-effort via execute-lane sync event arrival; drift diagnostic surfaces lag | **final** |
| Stale reads | **Permitted**; UI may render from stale cache; drift diagnostic gives operator explicit refresh affordance | **final** |
| Drift behavior | **Diagnostic-only.** No auto-correction. No proposal blocking. No apply blocking. (Per F15.0.1 §7.3.) | **final** |
| First-deployment migration | One-shot backfill on first boot post-F15.x deployment — read existing `chats.category_id` rows, generate corresponding `library.binding` rows in `bound` state with deterministic identity, mark migration with sentinel, skip on subsequent boots | **final** |
| Migration idempotency | **Guaranteed** by the binding-subject formula + uniqueness invariant — replay produces same subjectId → re-bind blocks with `binding-already-bound` (silent noop) | **final** |
| Cache row schema | `chats.category_id` retains its existing SQL shape (FK or NULL) for backward compatibility; SQL views may add a `category_subject_id_hash` derived column for joins | **final** |
| Cache invalidation primitive | Per-chat **monotonic** `cache_revision_number` (local-only integer); incremented on every cache write; UI compares versions to detect read-during-write inconsistency | **final** |
| Diagnostic codes | Reuses F15.0.1 §5 codes (`materialized-cache-drift-detected`, `category-cache-stale`); F15.0.2 adds **no new codes** | **final** |
| Cross-install settlement reordering edge cases | Provisional outline only; full contract → F15.6 | **provisional** |

**Bottom line:** the cache is an event-driven derivative of binding state. Local applies update the cache synchronously (read-your-own-write); cross-install applies update asynchronously via execute-lane sync arrival. Drift is observable but never auto-corrected. Migration is one-shot, deterministic, and idempotent under replay. No new privacy domain, no new diagnostic codes, no new proposal/apply gates beyond what F15.0.1 already specifies. The cache contract is small, mechanical, and operator-observable.

## 2. Source-of-Truth Boundary

### 2.1 The single rule

> **`library.binding` owns chat-category assignment. `chats.category_id` is a derived read cache. The settlement writer is the only mutator.**

### 2.2 Roles and authority

| Concern | Module / lane | Authority |
|---|---|---|
| **Owns** chat-category state | `library.binding` lane (F15.x) | Authoritative — the bound `chat-category` binding's `bindingState ∈ {bound, unbound}` IS the truth |
| **Writes** `chats.category_id` (cache) | `execute-settlement-writer` (F14.6.8) only | Sole mutator — called by execute lane after binding apply lands |
| **Reads** `chats.category_id` (cache) | chat.metadata UI, chat list views, category filter UI, store/chats.tauri.js read paths | Read-only; may render from stale cache |
| **May NEVER write** `chats.category_id` | chat.metadata canonicalizer / proposals / receipts; `store/chats.tauri.js` write paths; `store/categories.tauri.js`; any module not explicitly named above | Enforced by SQL-layer write-protection (SQLite trigger) at F15.0.3 phase + by audit grep at F15.x deployment |
| **Generates cache rebuild requests** | execute-settlement-writer (on apply), execute-resume-on-boot (on boot), F15.x drift sweep (low-frequency), F15.10 UI refresh button | Bounded set; no other module |

### 2.3 Chat.metadata coupling boundary (reaffirmed)

Per F15.0.0 §1 and F15.0.1 §7.4:

- Chat.metadata domain has **no chat-category mutation operation**.
- Chat.metadata canonical envelope MAY carry a `categoryIdHash` field for display purposes (sha256 of the bound category's subjectId, present iff the chat-category binding for that chat is `bound`), but this field is DERIVED at canonicalize time from the cache and is NEVER a write source.
- The chat.metadata canonicalizer (F14.3.1) MAY read the cache; the chat.metadata proposal generators (archive / rename) MUST NOT touch the cache.

### 2.4 Existing `store/chats.tauri.js` write paths

The existing `store/chats.tauri.js` SQLite store currently writes `category_id` via the `assignChat` / `clearChat` APIs (per the labels/categories/tags doc and §11.4 of the F14 audit). These write paths are categorized in F15.0.3 (the next doc phase) as one of:

- **Deprecate-and-rewire**: convert `assignChat` / `clearChat` to emit F15 binding proposals via the operator-driven path — they cease to write `category_id` directly.
- **Remove**: delete the API entirely if no operator-visible caller remains after F15.10 UI ships.

F15.0.2 commits to the deprecation outcome only; the precise rewire plan is F15.0.3 scope.

## 3. Cache Data Model

### 3.1 Existing schema (preserved for backward compatibility)

```
chats table:
  chat_id          TEXT PRIMARY KEY     -- raw chat UUID
  ...
  category_id      TEXT NULLABLE        -- existing column; FK or NULL
                                        -- F15.0.2: REPURPOSED as cache
```

The `category_id` column **retains its existing nullability and type**. No SQL schema migration is required at column-shape level. The semantic change (source-of-truth → read cache) is enforced by behavior and policy, not by column rename.

### 3.2 Optional derived helper columns (additive, non-breaking)

F15.0.3 implementation may add (NOT required by F15.0.2):

```
  category_subject_id_hash  TEXT NULLABLE  -- sha256 of bound category's
                                            -- library.catalog subjectId;
                                            -- enables hash-only joins
  cache_revision_number     INTEGER NOT NULL DEFAULT 0
                                            -- monotonic per-chat counter;
                                            -- incremented on every cache write
  last_cache_refresh_iso    TEXT NULLABLE   -- ISO timestamp, hour-coarsened;
                                            -- last successful cache write
```

These are additive. Existing readers that don't know about them continue to function. F15.0.2 commits to **introducing `cache_revision_number`** (mandatory for read-consistency contract §9) and treats the other two as F15.0.3 optimization choices.

### 3.3 Cache write payload

When `execute-settlement-writer` writes the cache after a `library.binding` apply event lands:

| Write field | Value source | Behavior |
|---|---|---|
| `chats.category_id` | If apply event is `bind chat-category`: raw `category_id` resolved from the bound `right` catalog's local store. If `unbind chat-category`: `NULL`. | Single UPDATE per chat |
| `chats.category_subject_id_hash` (if present) | If `bind`: the bound binding's `rightSubjectId` (already a sha256). If `unbind`: `NULL`. | Same UPDATE |
| `chats.cache_revision_number` | `cache_revision_number + 1` | Monotonically advancing |
| `chats.last_cache_refresh_iso` | `nowIsoSeconds()`, hour-coarsened | Set on every successful write |

The raw `category_id` field is sourced from the local categories store at write time — the apply event itself never carries the raw value (binding envelopes carry only hashes per F15.0.0 §8 / F15.0.1 §6).

### 3.4 What the cache does NOT store

- Catalog display name, color, icon — read from `categories.tauri.js` at render time
- Catalog lifecycle state — read from `library.catalog` lane snapshot
- Binding subjectId of the binding row — write time only; not persisted in `chats` table (already in `library.binding` ledger)
- Cross-install peer identity — never relevant to cache row
- Operator who triggered the binding — audit trail in `library.binding` bookkeeping, not in cache

## 4. Rebuild Trigger Matrix

The cache is event-driven with bounded fallbacks. **There is no general periodic refresh.**

### 4.1 Trigger taxonomy

| Trigger | When fired | Who fires it | Synchronous? | Scope |
|---|---|---|---|---|
| **Local apply** | After `library.binding` chat-category apply event lands at execute-settlement-writer | `execute-settlement-writer` (F14.6.8) | **Synchronous** — cache write happens **before** settlement returns | Single chat row |
| **Boot backfill** | On first boot post-F15.x deployment | F15.x migration module (F15.0.3 scope) | Synchronous (blocks app readiness) | All `chats` rows once; sentinel-gated |
| **Resume on boot** | Subsequent boots, if execute journal has incomplete chat-category apply events | `execute-resume-on-boot` (F14.6.9) | Asynchronous during resume | Affected chats only |
| **Cross-install sync event** | Execute lane receives a published chat-category apply event from a peer install | `execute-settlement-writer` invoked by the cross-install ingest path | Asynchronous | Single chat row |
| **Drift sweep** | Low-frequency periodic check for drift across all chat rows | F15.x drift sweep (F15.0.2 §8) | Asynchronous | All chats; throttled |
| **Manual refresh** | Operator clicks "Refresh categories" in chat list UI or `materialized-cache-drift-detected` banner | F15.10 UI | Synchronous (blocks UI button) | Single chat or all (operator choice) |

### 4.2 What is NOT a trigger

- **General polling** — explicitly rejected; wasteful and creates incorrect "live data" expectations.
- **Cache reads** — read paths never trigger rebuild; they read whatever the cache currently holds (stale-tolerant per F15.0.1 §7).
- **Drift detection alone** — detecting drift triggers a diagnostic emission, not a cache rewrite. Operator action is required.
- **Chat.metadata canonicalize** — read-only consumer; never triggers refresh.

### 4.3 Drift sweep cadence

| Setting | Value | Note |
|---|---|---|
| Default sweep interval | **5 minutes** | Background timer; pauses when app is unfocused |
| Per-sweep scope | Up to 1,000 chats per pass | Larger libraries paginated across multiple sweep cycles |
| Throttle on high-orphan-detect | 2x interval (10 min) | Backoff when many drift cases found in single pass |
| Sweep on resume | Single immediate pass | Catches any drift accumulated during sleep / offline period |

The sweep is a **diagnostic emitter**, not a cache writer. It surfaces `materialized-cache-drift-detected` warnings; it does not rewrite cache values.

## 5. Invalidation and Staleness Model

The cache row for a chat is in one of four well-defined states:

| State | Definition | Readable? | UI surfaces? |
|---|---|---|---|
| **Fresh** | Most recent cache write reflects the most recent successful local binding apply for that chat (and `cache_revision_number` matches the per-chat binding write counter) | Yes | No warning |
| **Stale** | A binding apply is **pending** in the execute journal but cache has not yet been refreshed (i.e., apply landed but settlement writer hasn't yet completed cache write — narrow window, <500ms target) | Yes — caller may read previous value | UI shows brief "Updating…" indicator on affected rows |
| **Diverged** | Cache value differs from the currently-bound binding's right subjectId (drift detected — F15.0.1 `materialized-cache-drift-detected`) | Yes — caller reads cached value with explicit drift annotation | UI shows refresh affordance on affected rows |
| **Unknown / boot-loading** | Cache state during pre-resume boot; cache rebuild has not completed | Yes — but no `cache_revision_number` advance has happened post-boot, so reads carry "boot" flag | UI shows "Loading categories…" until boot rebuild signals ready |

### 5.1 State transitions

- **Fresh → Stale**: binding apply lands at execute-settlement-writer; settlement writer hasn't completed cache write yet
- **Stale → Fresh**: settlement writer completes cache write (sets `last_cache_refresh_iso` + advances `cache_revision_number`)
- **Fresh → Diverged**: drift sweep detects cache value ≠ active binding's rightSubjectId
- **Diverged → Fresh**: operator manual refresh OR next local apply triggers settlement writer
- **Unknown → Fresh**: boot backfill / resume sweep completes for that chat

### 5.2 No "Invalid" state

Per F15.0.1: there is no "invalid" state that prevents read. The cache always carries a value (or `NULL`) that is **safe to display**. Worst case: an operator sees an old category assignment until refresh — never a crash, never a missing chat row.

### 5.3 Read protocol

The standard cache read protocol for UI / store consumers:

```
1. Read chats.category_id, chats.cache_revision_number, chats.last_cache_refresh_iso
2. Compare cache_revision_number against the in-memory "last observed" version
3. If versions match → cache state is consistent for this read
4. If versions differ → cache was updated between reads; refresh the in-memory view
5. UI render uses category_id verbatim; drift diagnostic (if any) is surfaced separately
```

This is the F15.0.2 read consistency contract (§9) operationalized.

## 6. Race Resolution Matrix

### 6.1 Local apply race scenarios

| Scenario | Resolution |
|---|---|
| Two concurrent `bind chat-category` proposals on the same chat (different categories) | First-to-pass-preflight wins via F14 replay defense; second receives `chat-category-conflict` (per F15.0.1 §4.3); cache reflects the winner |
| Concurrent `bind` + `unbind` for the same chat-category pair | F14 replay defense + uniqueness invariant; ordering at execute-settlement-writer determines order; cache reflects final state after both events apply |
| Replace-category half-completes (unbind succeeds, bind fails) | Cache reflects the actual binding state: `NULL` (no active chat-category binding). Per F15.0.1 §10.3, no rollback; chat is "uncategorized" until operator chooses a new category. UI surfaces this |
| Two concurrent `unbind` proposals on the same binding | Replay defense; first unbind succeeds; second is `binding-not-bound` (idempotent noop per F15.0.1 §4.2) |
| Local apply + cache rebuild trigger fire concurrently | Settlement writer serializes its writes; cache rebuild reads a consistent point-in-time snapshot; if a new apply lands during rebuild, the rebuild result is discarded and the apply's own synchronous refresh wins |

### 6.2 Cross-install race scenarios

| Scenario | Resolution |
|---|---|
| Local apply happens, then a cross-install apply for the same chat arrives via execute lane sync | Cross-install apply event is processed by `execute-settlement-writer`; if its `revisionHash` represents a strictly later state, cache updates to the cross-install value; if it's an earlier or concurrent state, replay defense rejects (no cache change) |
| Cross-install apply arrives **out of order** (i.e., second apply lands before first) | Per-chat monotonic version on the binding's `cache_revision_number` ensures convergence to latest; older apply events are rejected by replay defense even after they arrive |
| Cross-install apply arrives **during** local migration (boot backfill) | Local migration completes first (sentinel-gated); cross-install events processed normally afterward |
| Local apply lands, cross-install apply arrives **before** local cache refresh completes | Settlement writer's per-chat serialization ensures both writes apply in order; the second one wins (later `bindingState`) |
| Same-time cross-install applies from two peer installs | Replay defense at execute-settlement-writer applies; whichever apply event lands first wins; the second receives `binding-already-bound` blocker at the cross-install settlement boundary |

### 6.3 Cache-vs-binding race

| Scenario | Resolution |
|---|---|
| Read cache `category_id` while a write is in progress | SQLite single-writer model: read sees the pre-write value until the write transaction commits. No torn reads |
| Cache rebuild during pending apply | Rebuild scope is point-in-time; it does not block pending applies. Apply lands → settlement writer refreshes → cache rebuild result is superseded |
| Drift sweep + manual refresh racing | Each is independent; both produce the same final result (cache matches binding state); diagnostic events are deduped at emit time |

### 6.4 Provisional — cross-install settlement reordering edge cases

The full contract for handling cross-install settlement events arriving **before** the local install has finished local apply (e.g., laptop in airplane mode for a week, then reconnects) is deferred to F15.6 along with the F15.0.0 §20 provisional. F15.0.2 commits to the conservative behavior: **replay defense blocks late-arriving events whose `revisionHash` represents a strictly earlier state than the current local state.** This may discard valid cross-install evidence; F15.6 will refine.

## 7. Existing Data Migration Plan

First-deployment migration runs **exactly once** on the first boot following F15.x runtime deployment.

### 7.1 Migration steps

```
1. Check migration sentinel:
   SELECT value FROM h2o_meta WHERE key = 'f15.0.x.chat-category-migration-completed';
   if value = 'YES' → skip migration entirely

2. Read existing chats.category_id values:
   SELECT chat_id, category_id FROM chats WHERE category_id IS NOT NULL;

3. For each row:
   a. Resolve chat's subjectId via existing F14.3 chat canonicalizer
      (or compute from chat_id if canonicalizer not yet integrated)
   b. Resolve category's subjectId — generate the library.catalog subjectId
      using the F15.0.0 formula:
        subjectId = sha256(canonicalJSON({
          subjectType: 'library.catalog',
          catalogKind: 'category',
          rawId: category_id,
          perEnvelopeSalt: deriveSaltFromCategoryRow(category_id)
                           // deterministic; see §7.2
        }))
   c. Compute binding subjectId via F15.0.0 binding formula:
        bindingSubjectId = sha256(canonicalJSON({
          subjectType: 'library.binding',
          bindingKind: 'chat-category',
          leftSubjectId: chat's subjectId,
          rightSubjectId: category's subjectId,
          perEnvelopeSalt: deriveSaltFromMigration()
                           // deterministic per chat; see §7.2
        }))
   d. Insert library.binding row with state = 'bound', originAccountIdHash =
      local account hash, sourceTag = 'desktop-migration-f15'
   e. Use INSERT … ON CONFLICT DO NOTHING — idempotent on replay

4. Write migration sentinel:
   INSERT INTO h2o_meta (key, value) VALUES
     ('f15.0.x.chat-category-migration-completed', 'YES');

5. Emit migration audit event into snapshot bookkeeping (or a dedicated
   migration ledger; F15.0.3 specifies).
```

### 7.2 Deterministic salts for migration

To guarantee replay idempotency:

- `deriveSaltFromCategoryRow(category_id)` = `sha256("f15.0.x.category-migration-salt:" + category_id)`. Deterministic per category.
- `deriveSaltFromMigration()` = `sha256("f15.0.x.binding-migration-salt:" + chat_id + ":" + category_id)`. Deterministic per (chat, category) pair.

Same input data → same subjectIds. Repeated migration produces no new binding rows.

### 7.3 Handling malformed source data

| Source row condition | Migration behavior |
|---|---|
| `chat_id IS NULL` | Skipped; emit warning `chat-row-without-id-skipped` |
| `category_id IS NULL` | Skipped (no binding to create); cache will read as `NULL` correctly |
| `category_id` references a missing categories row | Skipped; emit warning `category-id-orphan-during-migration`. The chat's cache reverts to `NULL` after migration (consistent with binding state) |
| `category_id` references a tombstoned catalog | Per F15.0.1 §3.4, new binds against non-active catalogs are blocked. Migration is a special "back-fill" case — emit info `category-id-tombstoned-during-migration`, create the binding nonetheless, and surface to operator after migration as an orphan |
| Same `chat_id` appears twice (data corruption) | Process first occurrence only; warn `chat-row-duplicate-skipped` |

### 7.4 Migration audit trail

A single migration audit event is emitted to record the migration result:

```
{
  schema: 'h2o.desktop.sync.library-cache-migration.v1',
  migratedAtIso: ISO,
  chatsWithCategoryAtMigrationStart: N,
  bindingRowsCreated: M,
  bindingRowsSkippedAsDuplicate: K,
  bindingRowsSkippedAsOrphan: L,
  warnings: [...],
  migrationSentinelSet: true
}
```

This event is written to the snapshot bookkeeping ledger or a dedicated migration ledger (F15.0.3 decides). Operators / debuggers can verify migration outcome.

### 7.5 Roll-forward semantics

If the migration runs partially (e.g., crash during step 3) and the sentinel is NOT set:

- On next boot, migration re-runs from step 1
- Bindings already created have stable identities (deterministic salt) → re-insert is `ON CONFLICT DO NOTHING`
- New (not-yet-created) bindings get created
- After all chats processed, sentinel is set

This guarantees migration always completes within a bounded number of boots, even in the presence of mid-migration failures.

### 7.6 Downgrade safety

If an operator downgrades from F15.x runtime to pre-F15 runtime, the migration sentinel remains in `h2o_meta`. The pre-F15 runtime ignores it. The `chats.category_id` values remain valid as source-of-truth for the pre-F15 lane (no harm). On re-upgrade, the sentinel is detected and migration is skipped.

## 8. Drift Detection Contract

Per F15.0.1: drift is diagnostic only. No auto-correction. No proposal blocking. No apply blocking.

### 8.1 Detection mechanism

Drift is detected by comparing two values:

```
cacheValue       = chats.category_id (from cache row)
sourceOfTruthValue = the rightSubjectId of the active chat-category binding
                    for that chat (derived:
                    SELECT rightSubjectId FROM library_bindings
                    WHERE leftSubjectId = <chat's subjectId>
                      AND bindingKind = 'chat-category'
                      AND bindingState = 'bound'
                    LIMIT 1)
```

Drift is present when:

```
hash(cacheValue's catalog subjectId via the F15.0.0 formula) ≠ sourceOfTruthValue
  OR
(cacheValue IS NULL) ≠ (no active chat-category binding exists)
```

### 8.2 Diagnostic emission

| Condition | Emitted code | Severity | First detection threshold | Persistent threshold |
|---|---|---|---|---|
| First-time drift detected on a chat | `category-cache-stale` | info | Immediate on detection | (auto-promotes if persists) |
| Drift persists > 30 seconds | `materialized-cache-drift-detected` | warning | n/a (escalation only) | 30 seconds |
| Drift across > 10% of chats in a single sweep | `cross-lane-orphan-count-high` | warning | (aggregates from F15.0.1 §5 cross-lane code) | n/a |

The 30-second persistence threshold gives time for transient drift (e.g., apply pending, settlement writer about to refresh) to clear on its own without triggering operator-visible warnings.

### 8.3 Evidence digest

Per drift event emission, the diagnostic carries an `evidenceDigest`:

```
evidenceDigest = sha256(canonicalJSON({
  chatSubjectIdHash,
  observedCacheValueHash,        // null if cache was NULL
  observedSourceOfTruthValueHash, // null if no binding
  observedAtIso                    // hour-coarsened
}))
```

Privacy-safe — no raw chat IDs, no raw category names.

### 8.4 What drift does NOT do

- Does NOT update the cache (operator action required)
- Does NOT block any proposal at preflight (preflight ignores cache)
- Does NOT block any apply at settlement (settlement writes binding state, cache catches up async)
- Does NOT mutate source-of-truth (binding state is authoritative; cache lagging is the only condition)
- Does NOT cascade to chat.metadata (chat.metadata UI may also show drift annotation if it reads the cache directly; same diagnostic surfaces)

### 8.5 Operator clearing of drift

The only paths from `Diverged` → `Fresh`:

1. **Manual refresh** — operator clicks refresh button; F15.x UI invokes a synchronous cache refresh for the affected chat(s); cache is rewritten from binding state; drift cleared.
2. **Next local apply** — any subsequent `bind` or `unbind` chat-category on that chat triggers settlement writer to refresh cache.
3. **Drift sweep emission of `materialized-cache-drift-detected`** — does NOT auto-clear; only emits diagnostic for operator visibility.

### 8.6 Stricter conditions (F15.0.2 deliberate)

F15.0.1 said "drift is diagnostic only unless F15.0.2 deliberately defines a stricter condition with evidence." F15.0.2 commits:

- **No stricter conditions** introduced. Drift remains diagnostic-only.
- The cache is **never authoritative** in any code path.

This is final.

## 9. Read Consistency Contract

### 9.1 What readers may assume

| Reader scenario | Allowed assumption |
|---|---|
| Read after local apply succeeds (read-your-own-write) | **YES, fresh** — settlement writer refreshes cache synchronously before returning. Subsequent reads from the same install always see the just-applied state. |
| Read while another install's apply is in flight (no local indication) | Eventually consistent — within bounded latency from cross-install sync arrival (target < 5 seconds), cache will reflect the remote state. Until then, reads may return previous value. |
| Read during boot/resume | "Loading" indicator; cache may be stale until rebuild completes. UI shows boot-loading state explicitly; readers should not assume freshness. |
| Read during drift sweep emission | The cache value is whatever it is; readers see the actual cache state. Drift sweep diagnostic surfaces independently — reader can decide to show "may be stale" UI annotation. |
| Two consecutive reads of the same chat row | Either same `cache_revision_number` (consistent) or different (cache was updated between reads → re-read recommended) |
| Read by chat.metadata canonicalizer | May read cache to populate optional `categoryIdHash` field; must not write |
| Read by chat list UI | Best-effort; render whatever cache holds; drift diagnostic surfaces refresh affordance |

### 9.2 What readers may NOT assume

- **Reads do not guarantee cross-install consistency**: a chat may have been re-categorized on another install; the local cache will catch up via execute lane sync but not synchronously.
- **Reads do not see in-flight applies**: if an apply event is queued for settlement but hasn't yet written cache, reads see the pre-apply value.
- **Reads do not block**: never block a read waiting for cache rebuild. UI shows "Loading…" state if rebuild is in progress; never spin or freeze.

### 9.3 Read-your-own-write guarantee

**For local applies**, the contract is strict: after `settlement-writer` returns a successful settlement result for a chat-category apply, the cache for that chat reflects the applied state. UI code should read cache **after** the apply completes; this read is guaranteed fresh.

This is enforced by `execute-settlement-writer` performing the cache write **synchronously** within the same transaction as the binding apply write to Native. If the cache write fails, the settlement fails (treated as a hard apply failure; transaction rolls back). This narrow window — where binding state and cache state could drift — is closed by transactional coupling.

### 9.4 Cross-install latency budget

| Path | Target latency cache-reflects-truth |
|---|---|
| Local apply → settlement → cache write → read | < 100ms (synchronous; bounded by SQLite transaction commit) |
| Cross-install apply lands → execute lane ingests → settlement writes → cache reflects | < 5 seconds (target; depends on sync transport) |
| Drift sweep detects → diagnostic emitted → operator refresh → cache reflects | Operator-driven; bounded by operator action |
| Boot backfill → cache reflects all chat-category bindings | < 2 seconds for libraries with < 10k chats; throttled for larger |

These are targets; the contract does not bind these as SLAs. They serve as design budgets for F15.x implementation phases.

## 10. Privacy Model

### 10.1 Domain inheritance

Cache sync envelopes (cache-write events emitted by settlement writer, cache-rebuild events emitted by resume-on-boot, drift diagnostic events emitted by sweep) all use the existing **`library.binding` privacy domain** policy from F15.0.0 §8. **No new privacy domain is required.**

### 10.2 Forbidden in cache-related envelopes

Per inheritance from `library.binding`:

- Raw category name, raw color, raw description (would be a leak from the catalog domain into the binding/cache domain)
- Raw chat title, raw chat content, raw turn data
- Raw account identifiers
- Raw catalog ID, raw chat ID, raw binding ID — only sha256 hashes allowed
- Raw timestamps at sub-hour precision (all ISO values are hour-coarsened)

### 10.3 What the cache row may contain

The `chats` SQL row is local-only persistent state — it MAY contain raw values for backward-compatibility (the raw `category_id` is what was always there pre-F15). However:

- Cache **sync envelopes** (data that crosses the install boundary or appears in audit/diagnostic envelopes) NEVER contain raw values; they always carry hashes.
- The `chats.category_id` column is **device-local only**. It is never serialized into proposal candidates, applyEvent receipts, publication envelopes, or relay outbox rows.

### 10.4 Drift diagnostic privacy

Per §8.3, drift evidence digests are hash-only. No raw chat IDs, no raw category names, no raw account fields. Privacy scan via `kernel.scanDomainForbiddenFields('library.binding', diagnosticEntry)` at every emit time.

### 10.5 Migration audit trail privacy

The migration audit event (per §7.4) carries only counts and warnings — no raw IDs, no per-chat detail. Counts are coarsened (e.g., bucketed as "0-9", "10-99", "100-999", "1000+") to prevent fingerprinting library size at envelope level if the event ever crosses install boundary.

### 10.6 Read protocol privacy

Reads of `chats.category_id` (raw value) by chat.metadata UI / chat list / categorize filter are LOCAL only. The read result is rendered to the operator on this device. It does NOT propagate into envelopes that cross the install boundary — for that, the consumer (e.g., chat.metadata canonicalizer) hashes the value into `categoryIdHash` per F15.0.0 §1.

## 11. Proof Expectations

The F15.9 binding proof harness, the F15.x catalog proof harness, and a future F15.0.3-defined migration proof must directly assert:

### 11.1 Local apply updates cache

- `bind chat-category(chat, category)` succeeds → assert `chats.category_id` for that chat equals the raw category_id of `category` (via local categories store lookup).
- Assert `cache_revision_number` incremented by exactly 1.
- Assert `last_cache_refresh_iso` matches the apply's `appliedAtIso` (hour-coarsened).

### 11.2 Unbind clears cache

- `unbind chat-category(chat, category)` succeeds → assert `chats.category_id` for that chat IS NULL.
- Assert `cache_revision_number` incremented by exactly 1.

### 11.3 Replace-category updates cache after two proposals

- Bind chat to category A → cache shows A.
- Emit unbind chat-category(chat, A) → cache shows NULL.
- Emit bind chat-category(chat, B) → cache shows B.
- Assert intermediate state at unbind point: cache = NULL.
- Assert final state after both proposals: cache = B.

### 11.4 Drift is detected (diagnostic only)

- Force a drift state by directly writing `chats.category_id` to a wrong value while a binding remains bound to a different category. (This is a test-only invariant violation; in production, this code path is write-protected.)
- Run drift sweep.
- Assert `materialized-cache-drift-detected` warning emitted (after persistence threshold).
- Assert binding state UNCHANGED.
- Assert `chats.category_id` UNCHANGED (not auto-corrected).

### 11.5 Drift does not mutate source-of-truth

- Set up drift state per §11.4.
- Run drift sweep.
- Read `library.binding` rows.
- Assert binding `bindingState` and `rightSubjectId` UNCHANGED from pre-drift-sweep values.
- Source-of-truth is inviolable.

### 11.6 Migration is idempotent

- Insert N rows into `chats` with `category_id` populated.
- Run F15.0.x migration once → assert N binding rows in `library.binding` (state = bound).
- Run F15.0.x migration AGAIN → assert N binding rows (no duplicates; replay defense triggered or `ON CONFLICT DO NOTHING` did nothing).
- Assert migration sentinel is set after each run.

### 11.7 Cross-install replay does not create duplicate bindings

- Local install creates a chat-category binding via local apply.
- Simulate cross-install sync delivering the SAME apply event (replay).
- Assert no new binding row is created (replay defense or uniqueness invariant).
- Assert `cache_revision_number` does not advance (no re-write).

### 11.8 Chat.metadata cannot mutate cache

- Attempt to invoke `chat.metadata` canonicalizer / proposal generators with the intent to mutate `category_id`.
- Assert: no F14.3 module exposes a chat-category mutation API.
- Assert: any attempted SQL write to `chats.category_id` outside the execute-settlement-writer code path is rejected by the SQLite trigger (F15.0.3 will introduce; F15.0.2 proof reserves the assertion).

### 11.9 Read-your-own-write after local apply

- Local apply succeeds.
- Immediately read `chats.category_id`.
- Assert value matches the just-applied state (no race window).
- Assert this holds across thousands of apply-then-read cycles (no transient inconsistency).

### 11.10 Side-effect summary

- All F15.0.2-scoped diagnostic emits (drift sweep, migration audit) carry `sideEffectSummary` with all 8 flags `false`.
- Cache writes by settlement-writer are recorded in the settlement-writer's existing `sideEffectSummary` (which already ships `true` flags per F14.6.8). No new side-effect tracking required.

## 12. Risks and Mitigations

**12.1 Stale cache misleading operators.** During cross-install lag, operators see old category assignment. They may make decisions based on stale data.

*Mitigation*: `materialized-cache-drift-detected` warning surfaces refresh affordance; UI shows "may be out of date" banner with refresh button. Operators always have an explicit clear path. F15.10 UI defines exact banner shape.

**12.2 Migration data loss if salt logic changes.** If the F15.0.x migration uses deterministic salts and a later F15.x phase changes the salt formula, re-running migration would generate different subjectIds and create duplicate bindings (now under different IDs).

*Mitigation*: F15.0.2 commits the salt formulas in §7.2 as **stable and immutable**. Future F15.x phases must NOT change these. F15.0.3 implementation ratifies this by hard-coding the formulas in the migration module.

**12.3 Cache rebuild storm on bulk operations.** Bulk operator action ("tag all 1000 chats with label X") produces 1000 binding applies, each triggering a synchronous cache write. SQLite contention possible.

*Mitigation*: F15.0.3 may add batch path — settlement writer may write cache rows in a transaction batch (e.g., 100 rows per commit). The synchronous-write guarantee holds per-batch, not per-row.

**12.4 Drift sweep overhead on large libraries.** Sweeping 100k chats every 5 minutes is wasteful.

*Mitigation*: Sweep is paginated (1k chats per pass, throttled to spread across cycles). Operators can disable sweep entirely via settings if desired. Drift is also detected opportunistically on chat reads (cheap per-read check).

**12.5 Wider chat list UI degradation.** Adding cache_revision_number columns and drift checks to every read may slow down chat list rendering.

*Mitigation*: cache_revision_number is a single INTEGER column; reads are O(1). Drift check on read is optional (chat list may render without it; drift sweep surfaces drift separately).

**12.6 Boot backfill takes too long for large libraries.** Library with 50k chats with categories → 50k binding INSERTs at boot.

*Mitigation*: Backfill is bounded — uses `INSERT … ON CONFLICT DO NOTHING` for idempotency. Operator UI shows progress indicator. Boot completes when first 1k chats are migrated; remaining chats migrate progressively. F15.0.3 specifies exact pagination.

**12.7 Salt collision risk.** If two operators happen to migrate at exactly the same time on shared accounts, deterministic salts could in principle produce duplicate binding subjectIds.

*Mitigation*: salt formula is deterministic per (chat_id, category_id) pair — same input → same output. Cross-install replay of the same migration produces the same binding subjectIds; uniqueness invariant prevents duplicates. The "salt collision" risk is the same as the F14 binding-as-subject identity collision risk, which is bounded by sha256 collision probability.

**12.8 Backward compatibility break.** Existing readers of `chats.category_id` continue to function — but they now read from a cache that may be stale. Some readers (e.g., the chat list filter) may show wrong category counts during cross-install lag.

*Mitigation*: F15.x release notes explicitly call out the migration to source-of-truth = library.binding. Existing readers receive an "eventually consistent" guarantee. Drift diagnostic surfaces lag.

**12.9 SQLite trigger to enforce write-protection.** F15.0.3 will introduce a SQLite trigger that rejects writes to `chats.category_id` from any path other than execute-settlement-writer. If the trigger is buggy, it could lock out legitimate writes.

*Mitigation*: F15.0.3 specifies the trigger contract precisely; trigger is testable in isolation; if needed, can be bypassed via a one-off PRAGMA call by debugger.

**12.10 Cross-install settlement reordering during long offline periods.** A laptop offline for a week reconnects and receives a flood of cross-install apply events. Settlement writer must process them in order and converge cache to the latest state.

*Mitigation*: per F15.0.2 §6.4 (provisional, full contract in F15.6) — settlement writer processes events in `revisionHash`-defined order via replay defense. Events that represent earlier states are rejected. Cache converges to the latest. Operator may see a brief reorganization of category assignments after reconnection.

**12.11 Migration sentinel collision with downgrade/upgrade.** If a user downgrades F15 → pre-F15 → F15.x in sequence, the sentinel remains set, but the pre-F15 install may have made changes to `chats.category_id`. On re-upgrade, those changes are not migrated.

*Mitigation*: Re-upgrade detects sentinel; if pre-F15 has written `chats.category_id` since migration, the next chat-category binding apply will resolve via the cache as-found (post-pre-F15 writes). This is acceptable — F15.x treats pre-F15 cache writes as "stale data that needs binding reconciliation," not as authoritative.

## 14. Authoritative Summary

F15.0.2 is **final** for §§1–11 and **provisional** only for §6.4 (cross-install settlement reordering edge cases; defers to F15.6).

It defines:

1. The source-of-truth boundary: `library.binding` owns; `chats.category_id` is read cache; settlement writer is sole mutator.
2. The cache data model: existing `category_id` column repurposed; `cache_revision_number` mandatory; optional helper columns at F15.0.3 discretion.
3. The rebuild trigger matrix: event-driven via settlement writer, boot backfill, resume-on-boot, cross-install sync, drift sweep, manual operator refresh; no general polling.
4. The four-state invalidation model: Fresh / Stale / Diverged / Unknown; all states are readable; no "Invalid" state.
5. The race resolution matrix: F14 replay defense + uniqueness invariant + per-chat monotonic version; cross-install convergence to latest.
6. The migration plan: one-shot backfill with deterministic salts and `ON CONFLICT DO NOTHING` for idempotency; sentinel-gated; replay-safe.
7. The drift detection contract: 30-second persistence threshold for escalation; diagnostic-only per F15.0.1; no auto-correction; evidence digest privacy-safe.
8. The read consistency contract: read-your-own-write after local apply (synchronous); eventual consistency cross-install; latency budgets as design targets.
9. The privacy model: inherits `library.binding` domain policy; no new domain; cache row is device-local; envelopes are hash-only.
10. Proof expectations across 10 categories of harness assertions.
11. Eleven risks with mitigations.

F15.0.2 does not supersede F15.0.0 or F15.0.1 on any point. It commits the materialized cache contract that F15.0.1 §7.5 deferred. After F15.0.2, the F15.0.x design trilogy is two-thirds complete; F15.0.3 (store-layer deprecation plan) is the final design phase before F15.1 implementation begins.

**Next ratifiable phase:** F15.0.3 — existing labels/categories/tags store-layer deprecation plan (doc only). After F15.0.3, F15.1 begins implementation.
