# F15.0.3 - Existing Labels / Categories / Tags Store-Layer Deprecation Plan

**Status:** Architecture design - final for sections 1-12, provisional only for the exact F15.8 bulk-migration shim shape.
**Predecessors:** F15.0.0 at `cf681fc`, F15.0.1 at `e8e7db6`, F15.0.2 at `2caf0fc`.
**Scope:** Doc only. No runtime code. No store-layer changes. No Labels/Categories/Tags implementation. Does not begin F15.1.
**Authoritative position:** F15.0.3 is the authority for the existing store-layer deprecation and compatibility-shim plan. It does not supersede F15.0.0, F15.0.1, or F15.0.2 on canonical object, cross-lane, or materialized-cache semantics.

---

## 1. Executive Summary

F15 introduces `library.catalog` and `library.binding` as the source of truth for labels, tags, categories, and their chat/category edges. The existing Desktop SQLite stores remain the compatibility surface for F15 callers, but their write authority is deprecated and rewired at a single cutover point.

Top-level decisions:

| Decision | Verdict | Status |
|---|---|---|
| Existing store read APIs | **Preserved as-is. Signatures remain stable.** Internals may read F15-maintained tables, ledgers, or materialized caches after cutover. | final |
| Existing catalog write APIs | **Deprecate-and-rewire.** Calls become F15 catalog proposals, not direct catalog-row SQL writes. | final |
| Existing binding write APIs | **Deprecate-and-rewire.** Calls become F15 binding proposals, not direct `label_bindings` / `tag_bindings` writes. | final |
| Existing category-assignment write APIs | **Deprecate-and-rewire.** Calls become `chat-category` bind/unbind proposal sequences. | final |
| Direct `store/chats.tauri.js` category writes | **Blocked at SQLite trigger after cutover.** `patch(chatId, { categoryId })` and equivalent `upsert` paths cannot directly mutate `chats.category_id`. | final |
| Cutover point | **F15.8 only.** Shims and SQLite write-protection triggers ship together so compatibility and enforcement become true atomically. | final |
| Legacy API removal | **Not in F15.** Legacy write APIs remain callable and shimmed throughout F15. Physical API removal is F16+. | final |
| Shim settlement model | **Synchronous emit + wait with timeout.** The shim emits F15 proposal work and waits for settlement or a bounded pending state. | final |
| Tombstone shim behavior | Catalog delete/tombstone returns **`pending-review`** after F5 review enqueue/preview handoff, not after final F5 delete completion. | final |
| `replaceForChat` and replace-category | Decompose into independent F15 proposals. UI may group them, but the lanes receive bind/unbind proposals. | final |
| Subscribe notifications | Fire **after settlement**, not at proposal emission. | final |
| `saveNow()` | Waits for pending shim proposals to settle or reach the configured timeout. | final |
| Bundle import | Uses a special bulk-migration mode. Exact shape is provisional and deferred to F15.8. | provisional |
| Tag-category M:N | No legacy runtime API exists to migrate. It starts clean in F15. | final |
| F7 folder-binding overlap | Excluded from this plan until F15.11. | final |
| SQLite write protection | PRAGMA-based writer identity sentinel with debug/emergency bypasses. | final |

The deprecation plan is conservative: reads stay stable, writes are converted to operator-visible convergence work, direct SQL drift is blocked at the database layer, and old APIs are left in place until F16+ so existing callers do not break during F15.

## 2. Store-Layer Inventory

### 2.1 Existing Desktop store modules

| Module | Existing tables / columns | Current authority before F15 cutover | F15 role after cutover |
|---|---|---|---|
| `store/labels.tauri.js` | `labels`, `label_bindings` | Source of truth for label catalog rows and chat-label bindings | Compatibility read API + shimmed write API over `library.catalog` / `library.binding` |
| `store/tags.tauri.js` | `tags`, `tag_bindings` | Source of truth for tag catalog rows and chat-tag bindings | Compatibility read API + shimmed write API over `library.catalog` / `library.binding` |
| `store/categories.tauri.js` | `categories`; assignment through `chats.category_id` | Source of truth for category catalog rows and primary chat-category assignment | Compatibility read API + shimmed write API; `chats.category_id` becomes materialized cache per F15.0.2 |
| `store/chats.tauri.js` | `chats.category_id` plus other chat columns | Direct chat-row writer can currently write `category_id` through generic patch/upsert | Read API remains; direct category cache mutation is blocked after F15.8 |

### 2.2 Existing API families

The current APIs divide into five families:

1. **Store lifecycle:** `init`, `dispose`, `isReady`, `reload`, `saveNow`, `subscribe`, `diagnose`.
2. **Catalog reads:** `get`, `getAll`, `list`, `count` on labels/tags/categories.
3. **Catalog writes:** `create`, `upsert`, `patch`, `remove`, `delete` on labels/tags/categories.
4. **Chat binding reads:** `listChats`, `listForChat`, `getForChat`, chat list filters by `categoryId`.
5. **Chat binding writes:** `bindChat`, `unbindChat`, `replaceForChat`, `assignChat`, `clearChat`, plus direct `chats.patch/upsert` category writes.

F15.0.3 preserves family 1 and read signatures from families 2 and 4. It deprecates and rewires write families 3 and 5.

### 2.3 Non-goals

- No runtime implementation in F15.0.3.
- No store-layer code changes in F15.0.3.
- No new F15.1 implementation in this phase.
- No change to F15.0.0 subject identity.
- No change to F15.0.1 cross-lane blockers.
- No change to F15.0.2 materialized cache semantics.
- No F7 folder-binding migration before F15.11.

## 4. Write-Path Replacement Matrix

| Legacy write intent | Legacy path | F15 proposal decomposition | Settlement result exposed to legacy caller |
|---|---|---|---|
| Create label/tag/category | `create`, `upsert` insert path | One `library.catalog` create proposal | Created row projection after settlement; pending result on timeout |
| Rename catalog | `patch({ name })`, `upsert` update path | One `library.catalog` rename proposal | Updated row projection after settlement; pending result on timeout |
| Recolor label/category if supported | `patch({ color })` | One `library.catalog` recolor/update proposal | Updated row projection after settlement |
| Update non-color metadata | `patch`, `upsert` update path | One or more allowlisted catalog update proposals if F15 policy supports the field | Unsupported fields fail closed |
| Delete label/tag/category | `remove`, `delete` | One catalog tombstone proposal plus F5 review handoff path | `pending-review`; never waits for F5 delete completion |
| Bind chat to label/tag | `bindChat` | One `library.binding` bind proposal | `true` after settlement; pending result on timeout |
| Unbind chat from label/tag | `unbindChat` | One `library.binding` unbind proposal | `true` after settlement; idempotent no-op may return `false` or policy-specific noop result |
| Replace chat labels/tags | `replaceForChat` | N independent unbind proposals for removed edges plus M independent bind proposals for added edges | Aggregate result after all proposals settle or timeout |
| Assign category to chat | `assignChat` | If an active category exists: unbind old `chat-category`; then bind new `chat-category` | Aggregate result after settlement |
| Clear chat category | `clearChat` | Unbind active `chat-category` binding | `true` after settlement |
| Direct chat category patch | `chats.patch(chatId, { categoryId })` | Approved caller routes through same replace-category shim; unapproved direct SQL is blocked | Caller receives shim result or SQL trigger error |
| Bundle import category assignment | Import pipeline | Special bulk-migration mode that materializes F15 catalog/binding state, then refreshes cache | Shape provisional; deferred to F15.8 |

Replacement rules:

- The store API never emits a batch proposal envelope.
- `replaceForChat` and replace-category are UI-level aggregate operations only.
- The compatibility shim may group results for caller ergonomics, but each lane proposal remains independent.
- Notifications and read-cache updates happen only after proposal settlement.
- Failed or timed-out sub-proposals must be visible in the aggregate result.

## 5. SQLite Write-Protection Contract

### 5.1 Protected targets

F15.8 installs SQLite protection for all legacy direct-write surfaces that can diverge from F15 source-of-truth:

| Protected target | Direct write blocked after F15.8 | Authorized writer |
|---|---|---|
| `chats.category_id` | Any direct mutation outside settlement writer / approved migration | `execute-settlement-writer` and F15.8 bulk-migration mode |
| `label_bindings` | Direct insert/delete outside shim settlement | F15 binding settlement writer / approved migration |
| `tag_bindings` | Direct insert/delete outside shim settlement | F15 binding settlement writer / approved migration |
| `labels` | Direct catalog row insert/update/delete outside shim settlement | F15 catalog settlement writer / approved migration |
| `tags` | Direct catalog row insert/update/delete outside shim settlement | F15 catalog settlement writer / approved migration |
| `categories` | Direct catalog row insert/update/delete outside shim settlement | F15 catalog settlement writer / approved migration |

### 5.2 Writer identity sentinel

SQLite write protection is based on a connection-scoped writer identity sentinel. The F15.8 implementation must provide a PRAGMA-backed sentinel that can be set only by approved writer paths before protected statements run.

Required identities:

| Writer identity | Allowed scope |
|---|---|
| `f15.execute-settlement-writer` | Normal F15 catalog/binding settlement writes and materialized cache refresh |
| `f15.bulk-migration` | One-shot bundle/import/backfill mode |
| `f15.debug-bypass` | Developer-only bypass, disabled by default |
| `f15.emergency-repair` | Operator-explicit repair mode, audited and disabled by default |

Contract requirements:

- The sentinel is connection-local or transaction-local; it must not persist as durable domain state.
- The sentinel must be set before the protected write and cleared immediately after the transaction or write batch.
- SQLite triggers must reject protected writes when the sentinel is missing, unknown, or not authorized for the target.
- Debug and emergency bypasses must emit explicit audit evidence and must be impossible to enable silently.
- F15.8 must include a runtime proof that direct `chats.category_id` mutation fails without an authorized sentinel.

### 5.3 Trigger behavior

Protected triggers fail closed:

```
if writer_identity not in allowed_identities_for_target:
  abort write with f15-store-write-protected
```

The trigger layer is a backstop, not the primary routing mechanism. Correct code should reach protected tables only through F15 settlement or approved migration paths.

## 6. Read API Preservation Plan

### 6.1 Stable signatures

The following APIs keep their public signatures through F15:

- `get`, `list`, `getAll`, `count`
- `getForChat`, `listChats`, `listForChat`
- `chats.get`, `chats.list`, `chats.getAll`, `chats.count`
- `reload`, `diagnose`, `subscribe`, `saveNow`

Callers should not need to know whether a result came from a legacy table, a compatibility projection, a materialized cache, or a F15-maintained binding/catalog ledger.

### 6.2 Read semantics after cutover

| Read API family | Post-cutover source | Staleness policy |
|---|---|---|
| Catalog row reads | F15 catalog projection into existing table shape | Fresh after local settlement; eventually consistent cross-install |
| Label/tag binding reads | F15 binding projection into existing binding shape | Fresh after local settlement; eventually consistent cross-install |
| Category assignment reads | `chats.category_id` materialized cache per F15.0.2 | Stale reads permitted with drift diagnostics |
| Chat category filters | Same materialized cache | Stable signature, cache-backed semantics |

### 6.3 Read compatibility constraints

- Raw names may continue to be returned by local read APIs because they are local UI/store surfaces, not convergence envelopes.
- Convergence envelopes still carry only hashes/redacted fields per F15.0.0.
- Read APIs must not trigger proposal generation.
- Read APIs must not repair drift automatically.
- Read APIs may expose diagnostic metadata through `diagnose`, but normal row shapes stay stable.

## 7. Deprecation Timeline

| Phase | Scope | Store-layer outcome |
|---|---|---|
| F15.0.3 | This doc | Deprecation plan becomes source-of-truth; no runtime changes |
| F15.1 | Canonicalizers | `library-catalog-canonicalizer.tauri.js` and `library-binding-canonicalizer.tauri.js` planned/implemented |
| F15.2-F15.4 | Diagnostics, preflight, proposal candidates | Store writes still legacy; F15 lane can produce evidence/proposals |
| F15.5-F15.7 | Handoff/receipt/bookkeeping/proof/UI | Store writes still legacy unless explicitly behind development flags |
| **F15.8** | **Single cutover** | Compatibility shims and SQLite write-protection triggers ship together |
| F15.9-F15.10 | Stabilization and UI route cleanup | Legacy APIs remain callable; write calls route through shims |
| F15.11 | F7 folder-binding overlap review | Decide folder-binding absorption path; not part of F15.0.3 |
| F16+ | Removal phase | Legacy write API removal may begin after telemetry/proofs show no external dependency |

F15.8 is intentionally the only cutover point. Shipping shims without triggers leaves bypass paths open. Shipping triggers without shims breaks existing callers. They must land together.

## 8. Compatibility Shim Strategy

### 8.1 Shim contract

Every deprecated write API becomes a shim with this lifecycle:

1. Normalize legacy arguments.
2. Resolve local read-state needed for compatibility behavior.
3. Build one or more F15 proposal intents.
4. Emit proposal work synchronously to the F15 lane.
5. Wait for settlement with timeout.
6. Refresh compatibility projections/materialized cache as part of settlement.
7. Return the legacy-compatible result shape.
8. Notify subscribers after settlement, not at proposal emission.

Timeouts do not authorize direct SQL fallback. A timeout returns a pending result and leaves proposal settlement to the normal execute path.

### 8.2 Tombstone path

Catalog `remove` / `delete` is high impact and F5-owned. The compatibility shim must not wait for final F5 review completion.

Required behavior:

- Emit catalog tombstone proposal.
- Route to F5 review/handoff as defined by the F15 delete lane.
- Return `pending-review` after the review candidate is accepted.
- Do not clear bindings or cache immediately.
- Subscriber notifications fire only when an actual settlement changes the compatibility projection.

### 8.3 Replace operations

`replaceForChat` and replace-category are decomposed:

- Compute desired edge set.
- Compute current edge set.
- Emit one unbind proposal per removed edge.
- Emit one bind proposal per added edge.
- Wait for all emitted proposals to settle or timeout.
- Return an aggregate result with per-edge status.

No F15 kernel-level batch envelope is introduced.

### 8.4 `saveNow()`

`saveNow()` remains callable and returns a promise. After F15.8 it must:

- Wait for all pending shim proposals emitted by that store to settle, reach `pending-review`, or timeout.
- Never force direct SQL writes.
- Surface timeout/pending details through `diagnose`.
- Remain a no-op when there are no pending shim proposals.

### 8.5 `subscribe()`

Subscriptions remain in-process and legacy-compatible. The semantic shift is timing:

- Before cutover: notifications fire after direct SQL writes.
- After cutover: notifications fire after F15 settlement updates the compatibility projection/cache.
- Notifications must not fire merely because a proposal was emitted.
- Aggregate operations may emit one aggregate notification after all settle, or per-settled-edge notifications if existing UI depends on per-edge changes. F15.8 must choose and proof the final behavior.

### 8.6 Bundle import and bulk migration

Bundle import uses a special bulk-migration mode because it may need to create many catalog and binding records from existing imported data.

F15.0.3 requirements:

- Bulk mode is explicitly separate from normal shims.
- Bulk mode uses the authorized SQLite writer identity `f15.bulk-migration`.
- Bulk mode must be idempotent.
- Bulk mode must preserve privacy and hashing rules from F15.0.0.
- Exact request/receipt shape is provisional and deferred to F15.8.

## 9. Proof Expectations

F15.8 must include runtime and static proofs for this deprecation plan:

| Proof | Expected result |
|---|---|
| Read API signature proof | Existing read calls still return compatible row shapes |
| Shim route proof | `create`, `patch`, `remove`, `bindChat`, `unbindChat`, `replaceForChat`, `assignChat`, `clearChat` emit F15 proposals instead of direct SQL |
| Settlement proof | Subscriber notifications fire after settlement, not at proposal emission |
| `saveNow()` proof | Waits for pending proposals; never performs direct SQL fallback |
| Trigger proof | Direct `UPDATE chats SET category_id = ?` fails without authorized sentinel |
| Trigger allow proof | Settlement writer can update protected cache/table rows with the approved sentinel |
| Tombstone proof | Legacy delete returns `pending-review` and does not clear related bindings immediately |
| Replace proof | `replaceForChat` decomposes into independent bind/unbind proposals |
| Bundle import proof | Bulk migration mode is idempotent and isolated from normal shim mode |
| Privacy proof | F15 envelopes carry hashes/redacted fields only |
| Static grep proof | No non-settlement module writes protected targets after F15.8 |
| Drift proof | Category cache drift remains diagnostic-only per F15.0.2 |

F15.0.3 itself requires only docs validation because it ships no runtime behavior.

## 10. Risks and Mitigations

| Risk | Impact | Mitigation |
|---|---|---|
| Shim timeout changes legacy caller expectations | UI may receive pending instead of immediate boolean success | Return legacy-compatible result plus diagnostic pending details; make timeout bounded and visible |
| Trigger too early breaks callers | Existing UI writes fail before shims exist | Ship triggers and shims only together in F15.8 |
| Trigger bypass remains open | Direct SQL drift can continue | Mandatory PRAGMA-based sentinel and static grep proof |
| Replace operation partially settles | Chat may temporarily have partial label/tag/category state | Aggregate result lists per-edge status; UI can retry failed edges |
| Tombstone path appears successful too early | Caller may assume delete completed | Return explicit `pending-review`, not `true` |
| Subscriber timing regression | UI may miss optimistic updates | Notifications after settlement are intentional; UI can show pending state from shim result |
| Bundle import volume overwhelms proposal path | Import becomes slow or noisy | Separate bulk-migration mode, idempotent and audited |
| Read cache stale after cross-install activity | UI may display old category | F15.0.2 drift diagnostics and manual refresh affordance |
| Debug/emergency bypass abused | Source-of-truth can be bypassed | Disabled by default, operator-explicit, audited, and proofed |
| F7 folder-binding overlap accidentally pulled into F15 early | Scope creep and binding conflicts | Explicitly defer F7 overlap until F15.11 |

## 12. Authoritative Summary

F15.0.3 is the source-of-truth for how existing Labels / Categories / Tags store-layer APIs survive the transition to F15 lanes:

- Existing store read APIs are preserved as-is.
- Existing catalog write APIs are deprecate-and-rewire.
- Existing binding write APIs are deprecate-and-rewire.
- Existing category-assignment write APIs are deprecate-and-rewire.
- `store/chats.tauri.js` direct `categoryId` patch/upsert writes are blocked by SQLite trigger after cutover.
- F15.8 is the single cutover where shims and triggers ship together.
- Legacy APIs are not removed in F15; removal is F16+.
- Shims use synchronous emit + wait with timeout.
- Tombstone shims return `pending-review` instead of waiting for F5 completion.
- `replaceForChat` and replace-category decompose into independent F15 proposals.
- Subscribe notifications fire after settlement, not proposal emission.
- `saveNow()` waits for pending proposals to settle or timeout.
- Bundle import uses special bulk-migration mode; exact shape is deferred to F15.8.
- Tag-category M:N has no legacy runtime API to migrate and starts clean in F15.
- F7 folder-binding overlap is excluded until F15.11.
- SQLite protection uses a PRAGMA-based writer identity sentinel plus audited debug/emergency bypasses.
- F15.0.3 does not supersede F15.0.0, F15.0.1, or F15.0.2 on any canonical model, cross-lane, or materialized-cache decision.

With F15.0.3 complete, the F15.0.x design sequence is complete enough to start F15.1 implementation planning for:

- `library-catalog-canonicalizer.tauri.js`
- `library-binding-canonicalizer.tauri.js`
