# F12.0 — Folder delete convergence safety model

> **Contract document. F12.0.0 is docs-only.** No code, no delete
> implementation, no apply changes, no mobile write-back, no remote apply, no
> automatic delete. This file is the formal safety contract that every later
> F12.0.x phase must satisfy before any delete proposal is generated or any
> reviewed delete is handed to F5.

Delete is the capstone convergence operation and the most dangerous:
color / rename / move are non-destructive (reversible attribute or edge
changes), whereas delete removes a node, can cascade, and risks data loss.
Three properties govern this contract: **(1)** delete is destructive →
recovery is first-class; **(2)** delete has cascade blast radius → v1 is
**empty-folder-only**; **(3)** delete is **F5-owned** → convergence *proposes*,
F5 *reviews / tombstones / applies*.

## Grounding

- **F5 tombstone ownership** — F5 owns delete evidence, tombstone records,
  delete-vs-edit review, the reviewed delete/apply flow, and the
  synthetic-cleanup purge lifecycle. Convergence references tombstones; it
  never mints, applies, or purges them.
- **F6 conflict-queue ownership** — F6 owns the `sync_conflicts` queue and
  operator-driven, one-at-a-time ingest. Delete-vs-edit divergences route to
  F5; non-delete divergences route to F6.
- **F10.2 envelope kinds** (`envelope-v1.md`, `validate-kind.ts`) — the one-way
  pipeline `evidence → preview → proposal → conflict/review → audited apply`.
  Delete uses `operationIntent: "delete"`, which the validator gates with
  `delete-proposal-missing-f5-predicate` (proposal) and
  `delete-apply-event-missing-audit-id` (applyEvent).
- **Proven delete boundary** (`docs/systems/folders/folder-delete-preview-p8h.md`)
  — Native empty-folder delete applies only with **exact confirmation**;
  **non-empty delete apply is blocked**; Chrome has **no local fallback delete
  path** (routes through the Native owner); **Desktop delete apply remains
  disabled**. F12.0 inherits empty-only + exact-confirmation as hard rules.
- **F10.6 object model** (`folder-sync-object-model-f10.6.md`) — `folder`
  (`fold_<uuid>`), `folderBinding` (`folderBinding:<chatId>:<folderId>`),
  `tombstone` (F5-owned); `subjectId` redaction; sha256 `revisionHash` over
  the canonical field set; counts/hashes-only redaction posture.
- **Convergence machinery** (F10.6.x / F10.8.6x / F11.0) — canonicalizer,
  diff, planner buckets, proposal generation/publication, relay outbox/inbox,
  F10.7 transactional local apply, applyEvent, base watermark, lineage ledger.
  F12.0 reuses all of it; it adds only the delete-specific gates below.

## 1. Delete convergence lifecycle

```
measure  (folder present in base/local, absent-or-tombstoned in remote → delete candidate)
  -> planner DESTRUCTIVE bucket  (delete is NEVER auto-proposalEligible)
  -> operator elects             (explicit, one at a time)
  -> delete-proposal             (operationIntent: delete + F5 predicate)
  -> publish / review
  -> F5 reviewed-delete handoff  (F5 owns the decision)
  -> F5 mints tombstone + owning surface applies (empty-only, exact-confirmed, transactional)
  -> delete-apply-event          (carries F5 auditMaintenanceId)
  -> re-measure
  -> watermark advances ONLY when the folder is absent/tombstoned on BOTH peers (mutually observed)
```

Convergence **never deletes directly**. It produces a proposal and hands the
reviewed delete to F5; F5 mints the tombstone and the owning surface performs
the destructive apply.

## 2. F5 ownership boundaries

| Concern | Owner |
|---|---|
| Tombstone record (mint / shape / lifecycle) | **F5** |
| Delete-vs-edit review decision | **F5** |
| Reviewed delete / destructive apply | **F5** + owning surface |
| Synthetic-cleanup purge | **F5** |
| Delete *proposal* (request + justifying evidence) | F12.0 convergence |
| Referencing the F5 tombstone `recordId` in bookkeeping | F12.0 convergence |

Convergence MUST NOT mint, apply, or purge tombstones, and MUST NOT delete a
row outside the F5 reviewed-delete flow.

## 3. Tombstone ownership

- Tombstone shape (F5-owned): `{ recordKind: "folder" | "folderBinding",
  recordId, deleteReason, meta }`; binding tombstones use
  `deleteReason ∈ { "user-unbind", "folder-delete-cascade" }` (F5D).
- Convergence references a tombstone by F5 `recordId`; it never authors the
  record.
- **The tombstone is the recovery substrate** and MUST retain enough pre-delete
  state to undelete: the pre-state `revisionHash` plus a `device-local` raw
  snapshot of the folder record. This retention is a precondition of apply
  (§15).

## 4. Delete proposal model

- `kind: "proposal"`, `operationIntent: "delete"`, `subjectType:
  "folder.metadata"`, `operation: "folder-metadata-delete-proposed"`,
  `declaredAuthority: "proposal-source"` (**Desktop only**).
- Required payload (F10.2 §3.proposal + delete gate):
  `justifyingEvidenceDigests` (non-empty array of the remote tombstone /
  delete-evidence digests), `proposedOperation` (the redacted delete ledger
  entry), `expectedPostState` (folder absent / tombstoned), `predicateVersion`
  (non-empty, e.g. `h2o.folder-delete.predicate.v1`).
- The validator rejects a delete proposal lacking the non-empty
  `predicateVersion` + non-empty `justifyingEvidenceDigests`
  (`delete-proposal-missing-f5-predicate`).
- **Never auto-eligible** — a delete is always operator-elected from the
  planner's destructive bucket; the planner never marks delete `proposalEligible`
  automatically.
- `dedupeKey = sha256(canonicalJSON({ subjectId, operation: "delete", baseHash,
  actorPeerSyncHash }))`.

## 5. Recovery model (first-class)

- **Soft-delete first:** prefer a recoverable F5 tombstone over hard removal.
- Recovery substrate = the F5 tombstone with retained pre-state (§3).
- A **retention window** (§7) precedes any irreversible purge; convergence must
  not purge.
- Recovery is forward-gated (§6), never an implicit transaction rewind.

## 6. Undelete model

- Undelete = an **F5-owned reviewed restore**: re-create the folder from the
  tombstone's retained pre-state as a *forward gated create operation* through
  F5 / the owning surface.
- **Never auto-undelete.** No automatic resurrection of a tombstoned folder.
- Cross-peer, an undelete is observed by the other peer as a normal **remote
  create** and converges through the ordinary create path — there is no special
  remote-undelete command (consistent with "no remote apply").

## 7. Retention window model

- Between tombstone mint and irreversible purge there is a **retention/grace
  window** during which the folder is recoverable (§5/§6).
- The window MUST outlast convergence settling (both peers observing the delete
  through the boundary), so a delete is never purged before convergence is
  proven.
- **F5 synthetic-cleanup owns purge.** Convergence never purges and never
  shortens the window.

## 8. Empty-folder-only rule (v1)

- A folder is converge-deletable only when **membership count == 0** **and**
  it has **no child folders** (the proven boundary; non-empty delete apply is
  blocked).
- Non-empty / cascade delete is **deferred to F12.1** — it must mint child +
  binding cascade tombstones (`folder-delete-cascade`), each F5-gated, with the
  full subtree + binding blast radius surfaced and exact confirmation of the
  count.
- Emptiness is re-checked **at apply** (`folder-not-empty` / `not-empty-at-apply`);
  a folder that gained members or children since the proposal is blocked.

## 9. Delete-vs-rename

- Peer A deletes X; peer B renames X → the canonical **delete-vs-edit**
  conflict → **F5 review**. The rename contests the delete; the operator
  decides. Never auto-resolve, never last-writer.

## 10. Delete-vs-move

- Peer A deletes X; peer B moves X (or moves a child into/out of X) → **hard
  conflict → F5 delete-vs-edit**. Never auto-resolve.
- If a move's new parent is being deleted → orphan; the move is invalidated and
  routed to F5. (In v1 a parent with children is non-empty, hence not
  converge-deletable.)

## 11. Delete-vs-binding

- A folder with bindings (members) is **not empty** ⇒ **blocked in v1**, so a
  delete-vs-binding apply cannot arise.
- A binding added on peer B while peer A proposes deleting → the folder is now
  non-empty → block (`not-empty-at-apply`) → conflict / F5.
- Binding convergence stays deferred; the delete preview reports **membership
  as a count only** (the proven `membership = 0` gate), never raw `chatId`s.

## 12. Watermark interaction

- The delete proposal `baseHash` is anchored to the last-converged watermark.
- A delete **never advances** the watermark. The watermark advances only when
  the folder is **absent/tombstoned on both peers** at the export boundary and
  mutually observed (proven convergence).
- Stale base: if X was renamed / moved / gained members since the proposal's
  base, the delete is stale → re-validate (and likely now blocked as
  non-empty).

## 13. Replay interaction

- `dedupeKey` (§4) makes a re-applied delete of an already-tombstoned folder a
  **noop**.
- Delete envelopes below the watermark or for an already-tombstoned subject are
  inert (replay/dedupe + watermark monotonicity).
- Oscillation (delete ↔ recreate) is suppressed via lineage echo suppression;
  a recreate-after-delete is treated as a create contesting the tombstone (§14).

## 14. Resurrection protection

- A stale delete envelope MUST NOT re-delete a folder that was legitimately
  **re-created** with the same logical identity after the tombstone.
- Apply only if the **live folder `revisionHash` == the proposal `baseHash`**;
  a re-created folder has a different base ⇒ delete refused
  (`baseline-hash-not-verified`).
- Recreate-after-delete is a `create` that contests the tombstone → routed to
  F5, never silently re-deleted.

## 15. Required preflight checks (strictest set; re-run at apply)

| Check | Blocker (on failure) |
|---|---|
| Empty-only (membership 0 + no child folders) | `folder-not-empty` |
| Exact confirmation of the target (name + subjectId) | `exact-confirmation-required` |
| Base-fresh (`live revisionHash == baseHash`) | `baseline-hash-not-verified` |
| F5 predicate present (`predicateVersion` + `justifyingEvidenceDigests`) | `delete-proposal-missing-f5-predicate` |
| F5 reviewed-delete approval obtained | `f5-reviewed-delete-not-approved` |
| Delete-vs-edit clear (no contesting rename/move/color on peer) | `delete-vs-edit-conflict` |
| No open F6 conflict for the subject | `f6-blocker-present` |
| Owner/authority gate (apply only via owning surface; honors enablement state) | `delete-apply-not-authorized` |
| Recovery precondition (tombstone will retain pre-state) | `recovery-precondition-unmet` |
| Not stale / not replay + resurrection guard | `stale-or-replayed-delete` |
| Privacy: forbidden-field scan on output; hashed IDs; membership as count only | `delete-preflight-output-contains-forbidden-field` |

All checks re-run at apply time (TOCTOU), inside the transaction; any failure
rolls back with no tombstone and no applyEvent.

## 16. MUST / SHOULD / DEFER

- **MUST:** convergence **proposes**, F5 **reviews / tombstones / applies**
  (never convergence-owned delete); **empty-folder-only** in v1; exact
  confirmation + base-fresh re-check at apply; F5 predicate on the proposal and
  `auditMaintenanceId` on the applyEvent; delete-vs-edit (rename/move) → F5,
  never auto-resolve; resurrection guard (no re-delete of a re-created folder);
  soft-delete + retention window; tombstone retains pre-state for undelete;
  transactional apply with rollback + kill-switch; Desktop-local, manual, one
  at a time; redacted/hashed output with membership as count only; watermark
  advances only on proven both-sides-absent convergence.
- **SHOULD:** mirror the rename/move module suites (materialization-diagnostic,
  preflight, proposal-candidate-generator, action, apply-event, bookkeeping,
  proof, UI); an explicit undelete/recovery diagnostic + UI affordance; surface
  delete-vs-edit and membership count prominently in the confirm modal;
  reference the F5 tombstone `recordId` in bookkeeping.
- **DEFER:** non-empty / cascade delete convergence (child + binding cascade
  tombstones — F12.1, separate review); binding-cascade apply; multi-peer (>2)
  delete; irreversible purge automation; any automatic delete — **never**
  auto-delete, auto-cascade, auto-undelete, or remote apply.
