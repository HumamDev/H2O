# F14.5.1 - Capture canonical schemas

> **Docs-and-schema-only contract.** F14.5.1 introduces no runtime behavior,
> publication, apply, relay, write-back, storage migration, automatic
> materialization, or mutation. It freezes the reviewable canonical schema
> contracts for `CaptureEvent` and `CaptureArtifact` only.

F14.5 Capture needs stable object contracts before any implementation can
materialize capture evidence, publish it, or compare it against local state.
This phase defines the redacted canonical shape and safety keys that later
runtime phases must obey.

The companion static schema is
[`schemas/capture-canonical-v1.schema.json`](./schemas/capture-canonical-v1.schema.json).

## 1. Scope

| In | Out |
|---|---|
| Canonical `CaptureEvent` allowlist. | Capture ingestion implementation. |
| Canonical `CaptureArtifact` allowlist. | Publication, relay, proposal, apply, or write-back. |
| Identity composition rules. | Storage table creation or migration. |
| Replay keys and audit keys. | Watermark or consumed-operation writes. |
| Ownership invariants. | Any conversion to Chat, Snapshot, Notes, Bookmarks, Context, Library, or Dock state. |
| Denylist validation contract. | Mutation of existing capture-store blobs. |
| Schema versioning rules. | Background polling, automatic sync, or hidden refresh. |
| Required future indexes. | Runtime validators or generated code. |

## 2. Canonical objects

`CaptureEvent` is an append-only audit/evidence event: something was observed,
normalized, accepted, blocked, quarantined, or superseded.

`CaptureArtifact` is the redacted canonical subject that an event points at:
the durable capture evidence identity and its safe structural metadata.

Neither object is a command. Neither object can request or perform local
mutation. A future runtime may store or validate these objects only after a
separate implementation phase.

## 3. CaptureEvent allowlisted fields

No field outside this allowlist is valid for `schema:
"h2o.capture.event.v1"` and `schemaVersion: 1`.

```ts
type CaptureEvent = {
  schema: "h2o.capture.event.v1";
  schemaVersion: 1;
  eventId: string;
  eventKind:
    | "observed"
    | "normalized"
    | "artifact-linked"
    | "validation-blocked"
    | "quarantined"
    | "superseded";
  eventAtIso: string;
  producer: CaptureProducer;
  owner: CaptureOwner;
  source: CaptureSourceSummary;
  subject: CaptureSubject;
  replay: CaptureReplayKeys;
  audit: CaptureAuditKeys;
  validation: CaptureValidationSummary;
};
```

Allowed `producer` fields:

| Field | Contract |
|---|---|
| `platformId` | One of `desktop-studio`, `chrome-studio`, `native-extension`, `mobile`. |
| `surfaceKind` | One of `desktop-tauri`, `browser-studio`, `browser-runtime`, `mobile`. |
| `producerPeerIdHash` | SHA-256 hash. Raw peer IDs are forbidden. |
| `installIdHash` | SHA-256 hash. Raw install IDs are forbidden. |
| `sequence` | Non-negative integer or `null` when producer sequencing is unavailable. |

Allowed `owner` fields:

| Field | Contract |
|---|---|
| `ownerWorkspaceIdHash` | SHA-256 hash of the owning workspace identity. |
| `ownerAccountIdHash` | Optional SHA-256 hash of the owning account identity. |
| `ownerPeerIdHash` | Optional SHA-256 hash of the peer currently allowed to review local materialization. |

Allowed `source` fields:

| Field | Contract |
|---|---|
| `sourceKind` | One of `chatgpt-live`, `claude-live`, `chatgpt-import`, `manual-redacted`, `native-mirror`, `studio-bundle-import`. |
| `origin` | One of `live`, `import`, `manual`, `mirror`. |
| `observationSource` | Optional enum: `page-localStorage`, `native-mirror`, `extension-message`, `desktop-import`, `mobile-import`, `manual-entry`. |
| `sourceSubjectHash` | SHA-256 hash of the source-side conversation, import, or mirror subject. |
| `captureSessionIdHash` | Optional SHA-256 hash linking events from the same local capture session. |

Allowed `subject` fields:

| Field | Contract |
|---|---|
| `subjectType` | Literal `capture.artifact`. |
| `artifactIdHash` | SHA-256 canonical artifact identity. |
| `artifactRevisionHash` | SHA-256 canonical artifact revision identity. |
| `chatIdHash` | Optional SHA-256 hash of the source chat identity. |
| `snapshotIdHash` | Optional SHA-256 hash of the source snapshot identity. |
| `turnIdHash` | Optional SHA-256 hash of the source turn identity. |

`eventId` identifies this event row. It must be generated independently from
raw source IDs. `artifactIdHash` identifies the redacted subject and may repeat
across multiple events in the same lineage.

## 4. CaptureArtifact allowlisted fields

No field outside this allowlist is valid for `schema:
"h2o.capture.artifact.v1"` and `schemaVersion: 1`.

```ts
type CaptureArtifact = {
  schema: "h2o.capture.artifact.v1";
  schemaVersion: 1;
  artifactIdHash: string;
  artifactRevisionHash: string;
  artifactKind:
    | "chat-snapshot-digest"
    | "turn-selection-digest"
    | "attachment-metadata-digest"
    | "capture-store-digest"
    | "manual-redacted-evidence";
  artifactState: "observed" | "captured" | "blocked" | "quarantined" | "superseded";
  capturedAtIso: string;
  updatedAtIso: string;
  owner: CaptureOwner;
  source: CaptureSourceSummary;
  identity: CaptureArtifactIdentity;
  summary: CaptureArtifactSummary;
  replay: CaptureReplayKeys;
  audit: CaptureAuditKeys;
  validation: CaptureValidationSummary;
};
```

Allowed `identity` fields:

| Field | Contract |
|---|---|
| `sourceSubjectHash` | SHA-256 hash of the source-side conversation, import, or mirror subject. |
| `chatIdHash` | Optional SHA-256 hash of the source chat identity. |
| `snapshotIdHash` | Optional SHA-256 hash of the source snapshot identity. |
| `turnIdHash` | Optional SHA-256 hash of the source turn identity. |
| `sourceItemIdHash` | Optional SHA-256 hash of the source capture item identity. |
| `captureSessionIdHash` | Optional SHA-256 hash linking artifacts from the same local capture session. |

Allowed `summary` fields:

| Field | Contract |
|---|---|
| `redactionClass` | `redacted` or `metadata-only`. |
| `evidenceDigest` | SHA-256 over local canonical redacted evidence or structural evidence; never the raw evidence itself. |
| `lengthBucket` | Optional bucket enum: `empty`, `short`, `medium`, `long`, `very-long`, `unknown`. |
| `itemCount` | Optional non-negative integer for aggregate artifacts. |
| `turnCount` | Optional non-negative integer for chat/snapshot artifacts. |
| `attachmentCount` | Optional non-negative integer for attachment metadata artifacts. |
| `statusBucket` | Optional enum: `new`, `reviewed`, `archived`, `converted`, `dismissed`, `other`. |
| `kindBucket` | Optional enum: `captureSnippetKind`, `attachmentMetadataKind`, `aggregateDigestKind`, `otherKind`. |

The `summary` field is still redacted metadata. It must not contain raw message
text, HTML, Markdown, filenames, URLs, attachment bytes, model names, user
titles, tags, or route suggestions.

## 5. Identity composition

All canonical identity fields are hashes produced from canonical JSON tuples.
The tuple label and schema version are part of the hash input so v1 identities
cannot collide silently with a future v2 shape.

`artifactIdHash`:

```txt
sha256(canonical-json([
  "h2o.capture.artifact.identity.v1",
  owner.ownerWorkspaceIdHash,
  source.sourceKind,
  source.sourceSubjectHash,
  identity.chatIdHash || null,
  identity.snapshotIdHash || null,
  identity.turnIdHash || null,
  identity.sourceItemIdHash || null,
  artifactKind
]))
```

`artifactRevisionHash`:

```txt
sha256(canonical-json([
  "h2o.capture.artifact.revision.v1",
  artifactIdHash,
  summary.redactionClass,
  summary.evidenceDigest,
  summary.lengthBucket || null,
  summary.itemCount || null,
  summary.turnCount || null,
  summary.attachmentCount || null,
  summary.statusBucket || null,
  summary.kindBucket || null
]))
```

`eventId`:

```txt
uuidv7() or ulid()
```

`eventId` is intentionally not derived from raw source IDs. Duplicate defense
uses replay keys, not row identity reuse.

## 6. Replay keys

Both objects carry the same `replay` shape:

```ts
type CaptureReplayKeys = {
  lineageId: string;
  dedupeKey: string;
  payloadHash: string;
  eventDigest: string;
  sourceWatermark: string | null;
};
```

| Key | Composition |
|---|---|
| `lineageId` | Stable UUID/ULID for the local capture session or review chain. It is not a source chat ID. |
| `dedupeKey` | SHA-256 of schema, object kind, producer peer hash, owner workspace hash, artifact id hash, artifact revision hash, event kind if present, and source watermark if present. |
| `payloadHash` | SHA-256 of the canonical allowlisted object payload excluding `audit`, `validation`, `replay.payloadHash`, and `replay.eventDigest`. |
| `eventDigest` | SHA-256 of the canonical full object excluding warnings, blockers, and `eventDigest` itself. |
| `sourceWatermark` | Optional redacted source progress marker. It must be forward-only when a future phase writes watermarks. |

Replay keys are comparison and audit inputs only. They do not apply, publish,
advance watermarks, or record consumed operations in this phase.

## 7. Audit keys

Both objects carry the same `audit` shape:

```ts
type CaptureAuditKeys = {
  auditKey: string;
  custodyChainHash: string;
  validationResultHash: string;
  denylistScanHash: string;
};
```

| Key | Composition |
|---|---|
| `auditKey` | SHA-256 of owner workspace hash, producer peer hash, artifact id hash, artifact revision hash, object schema, and event digest. |
| `custodyChainHash` | SHA-256 over the redacted custody tuple: producer, source, owner, lineage id, and previous custody hash when present. |
| `validationResultHash` | SHA-256 over validation status, policy version, blocker codes, warning codes, and denylist policy version. |
| `denylistScanHash` | SHA-256 over the scanner name, scanner version, denylist version, scanned object schema, and scan result codes. |

Audit keys are immutable evidence links. A later correction produces a new
event or artifact revision; it must not rewrite an existing audit key.

## 8. Ownership invariants

| Invariant | Contract |
|---|---|
| Owner is explicit | Every object carries `owner.ownerWorkspaceIdHash`. |
| Custody is not ownership | `producer` and `source` identify custody and observation, not the owner allowed to review materialization. |
| Source account is not owner | Source-side account or chat identity may appear only as hashes under `source` or `identity`; it never grants authority. |
| Relay custody adds no authority | If a future relay carries these objects, relay possession must not change owner or producer authority. |
| Materialization is separate | A valid capture object may justify later local review, but it cannot create Chat, Snapshot, Library, Note, Bookmark, Context, or Dock rows by itself. |
| No cross-owner merge | Objects with different `ownerWorkspaceIdHash` values cannot dedupe, supersede, or replay each other without a future owner-handoff contract. |
| Blocked remains auditable | Blocked or quarantined objects keep audit and replay keys so the failure can be proven without retaining raw content. |

## 9. Denylist validation contract

Validation is deny-by-default:

- Unknown object fields are blockers.
- Unknown enum values are blockers.
- Missing required fields are blockers.
- Invalid hashes, timestamps, or negative counters are blockers.
- Raw or free-text fields are blockers, even when nested.

Forbidden key names at every depth:

```txt
body
content
contentHtml
contentText
html
markdown
messages
message
text
title
tags
routeSuggestion
attachments
attachmentBytes
file
filename
path
url
href
sourceUrl
accountId
chatId
snapshotId
turnId
itemId
messageId
msgId
model
email
password
apiKey
token
accessToken
refreshToken
```

Allowed fields whose names contain hash-suffixed identity, such as
`chatIdHash`, are not exceptions to the raw-ID ban; they are the only valid way
to carry those identities.

Validation output is code-only:

```ts
type CaptureValidationSummary = {
  status: "accepted" | "partial" | "blocked" | "quarantined";
  policyVersion: "f14.5.1";
  denylistVersion: "capture-denylist-v1";
  blockers: string[];
  warnings: string[];
};
```

`blockers` and `warnings` must contain fixed code strings only. They must not
carry raw excerpts, paths, URLs, user text, source IDs, or exception messages.

Required blocker codes for future validators:

| Code | Meaning |
|---|---|
| `capture-schema-unknown-field` | Object includes a field outside the allowlist. |
| `capture-schema-missing-required-field` | Required field absent or null. |
| `capture-schema-invalid-enum` | Enum value outside this contract. |
| `capture-schema-invalid-hash` | Hash field is not SHA-256 hex. |
| `capture-schema-invalid-timestamp` | Timestamp is not ISO UTC. |
| `capture-schema-invalid-counter` | Count field is negative or non-integer. |
| `capture-denylist-key` | Forbidden key name appears at any depth. |
| `capture-denylist-raw-identity` | Raw source or owner identity appears. |
| `capture-denylist-raw-content` | Raw message, title, tag, HTML, Markdown, attachment, path, URL, token, or free-text content appears. |
| `capture-owner-mismatch` | Owner hash does not match the local review owner. |
| `capture-replay-duplicate` | Replay keys prove the object has already been consumed or observed. |

## 10. Schema versioning rules

| Rule | Contract |
|---|---|
| Literal schema names | `h2o.capture.event.v1` and `h2o.capture.artifact.v1` identify the v1 contracts. |
| In-object version | `schemaVersion: 1` is required on both objects. |
| No field repurposing | Existing field names, meanings, enum values, and hash inputs cannot be repurposed. |
| Additive fields require review | Even optional additions require a later F14.5.x review and schema update. |
| Breaking changes require v2 | Removing required fields, changing identity composition, or allowing new raw data requires a new schema literal and `schemaVersion: 2`. |
| Readers fail closed | Unknown schema literals, unknown versions, unknown fields, or unknown enums block validation. |
| Migrations are explicit | Any future migration must produce new v2 objects or append migration events; it must not rewrite v1 audit evidence in place. |
| Hash inputs are versioned | Identity, replay, and audit hash labels include `v1` and must change on v2. |

## 11. Required indexes

These indexes are required for any future persistence implementation. This
phase does not create them.

### CaptureEvent indexes

| Index | Uniqueness | Purpose |
|---|---:|---|
| `capture_events_event_id` on `eventId` | Unique | Primary row lookup. |
| `capture_events_event_digest` on `replay.eventDigest` | Unique | Literal duplicate detection. |
| `capture_events_dedupe` on `replay.dedupeKey` | Non-unique | Replay and duplicate grouping. |
| `capture_events_artifact_time` on `subject.artifactIdHash, eventAtIso` | Non-unique | Artifact audit timeline. |
| `capture_events_owner_time` on `owner.ownerWorkspaceIdHash, eventAtIso` | Non-unique | Owner-scoped audit listing. |
| `capture_events_kind_status` on `eventKind, validation.status` | Non-unique | Blocked/quarantined review queues. |
| `capture_events_audit_key` on `audit.auditKey` | Unique | Audit proof linkage. |
| `capture_events_lineage` on `replay.lineageId` | Non-unique | Capture session/review chain replay. |

### CaptureArtifact indexes

| Index | Uniqueness | Purpose |
|---|---:|---|
| `capture_artifacts_artifact_id` on `artifactIdHash` | Unique | Primary subject lookup. |
| `capture_artifacts_revision` on `artifactRevisionHash` | Unique | Revision proof and supersession. |
| `capture_artifacts_owner_kind_time` on `owner.ownerWorkspaceIdHash, artifactKind, capturedAtIso` | Non-unique | Owner-scoped lists. |
| `capture_artifacts_source_subject` on `owner.ownerWorkspaceIdHash, source.sourceKind, source.sourceSubjectHash` | Non-unique | Source-scoped materialization review. |
| `capture_artifacts_chat` on `identity.chatIdHash` | Non-unique | Chat-adjacent evidence lookup without raw chat IDs. |
| `capture_artifacts_validation` on `artifactState, validation.status` | Non-unique | Blocked/quarantined queues. |
| `capture_artifacts_dedupe` on `replay.dedupeKey` | Non-unique | Replay grouping. |
| `capture_artifacts_audit_key` on `audit.auditKey` | Unique | Audit proof linkage. |
| `capture_artifacts_lineage` on `replay.lineageId` | Non-unique | Capture session/review chain replay. |

## 12. Acceptance for F14.5.1

F14.5.1 is complete when:

- The docs and static schema define `CaptureEvent` and `CaptureArtifact`.
- The allowlists, identity composition, replay keys, audit keys, ownership
  invariants, denylist validation contract, schema versioning rules, and
  required indexes are reviewable.
- No runtime files, storage migrations, relay code, publication code, apply
  paths, write-back paths, or mutation paths are introduced.
- `git diff --check` passes.
- `git diff --cached --check` passes after staging the F14.5.1 files.
