# F16.4 - `folder_bindings` Trigger Decision Contract

Status: F16.4.a docs-only decision contract.

F16.3 is closed. F16.4 decides how and when `folder_bindings` can be protected by SQLite writer-identity triggers without breaking the F7 compatibility fallback that F15.11 deliberately preserved.

## 1. Architecture Verdict

F16.4 should not protect `folder_bindings` immediately with unconditional SQLite triggers.

The recommended decision is to add a scoped compatibility identity first, then decide whether to enable `folder_bindings` triggers after proof.

Suggested scoped identity:

```text
f16.folder-legacy-fallback
```

Immediate trigger protection is unsafe because:

- F16.2 proves F7/F15 bridge behavior in deterministic simulation.
- F16.3 proves synthetic stress and performance characteristics.
- Neither F16.2 nor F16.3 proves SQLite trigger safety for existing legacy direct `folder_bindings` writes.
- `src-surfaces-base/studio/store/folders.tauri.js` still has legacy direct `folder_bindings` writes when F15 folder-binding delegation is off.
- Folder removal cleanup also deletes folder bindings directly.
- Enabling triggers before these paths are wrapped would likely break default F7 fallback compatibility.

F16.4 therefore keeps trigger protection deferred while allowing scoped identity work to proceed.

## 2. Decision Options

| Option | Decision | Reason |
|---|---|---|
| Keep deferred | Acceptable fallback | Safest if no F16.4 implementation is desired, but leaves `folder_bindings` unprotected |
| Protect now | Not recommended | Likely breaks flag-off F7 fallback and folder delete cleanup |
| Add scoped fallback identity | Recommended | Enables auditability without retiring F7 fallback |
| Make F15 delegation default-on | Later candidate | Should happen only after scoped fallback and trigger proofs pass |

## 3. Recommended F16.4 Split

| Phase | Scope |
|---|---|
| F16.4.a | Docs-only trigger decision contract |
| F16.4.b | Add and prove a scoped fallback identity such as `f16.folder-legacy-fallback` |
| F16.4.c | Optional guarded `folder_bindings` trigger implementation |
| F16.4.d | Proof/validator hardening and closure audit |

F16.4.b should not silently enable trigger protection. It should prove that legacy compatibility writes can be routed through a narrow identity before F16.4.c considers real `folder_bindings` triggers.

## 5. Proof Requirements Before Real Trigger Protection

| Proof | Expected |
|---|---|
| Unauthorized direct `folder_bindings` write | Blocks only when trigger mode is active |
| F15 delegated `chat-folder` write | Passes through settlement identity |
| Legacy fallback write | Passes only through scoped fallback identity |
| Delegation flag off | Existing F7-compatible behavior preserved |
| Delegation flag on | F15 `chat-folder` path used |
| Explicit fallback | Audited, not silent raw SQL |
| Folder delete cleanup | Covered by authorized identity or delegated path |
| F7 parity | Still green |
| Privacy | No raw chat/folder IDs, names, colors, paths, URLs, or tokens |
| Closure | Existing F15/F16 validators remain green |

## 6. Next Condition Before Enabling Triggers

`folder_bindings` triggers should be enabled only after all legacy write paths are either routed through F15 delegation or through a narrowly scoped audited identity.

The covered paths must include at least:

- `store.folders.bindChat`.
- `store.folders.unbindChat`.
- Folder removal cleanup that deletes `folder_bindings` rows.
- Any `desktopSetFolderBinding` wrapper path that can still reach legacy fallback.

Until that condition is met:

- Trigger protection remains deferred.
- Scoped identity work may proceed.
- F15 delegation should not be forced default-on.
- F7 fallback must remain available.

## 7. Scoped Fallback Identity Contract

The scoped fallback identity should be narrower than debug or emergency bypass identities.

Contract for `f16.folder-legacy-fallback`:

| Requirement | Contract |
|---|---|
| Scope | Authorizes only compatibility writes needed for `folder_bindings` fallback paths |
| Activation | Explicit runtime/config path only; never silently enabled |
| Auditability | Proof and reports must identify that fallback identity was used |
| Privacy | Reports remain hash-only and never expose raw chat or folder identifiers |
| F15 compatibility | Does not bypass the F15 `library.binding` lane when delegation is enabled |
| Failure behavior | No silent fallback from F15 delegation failure unless explicit fallback is allowed |

The identity must not become a generic protected-table bypass. It must not authorize labels, tags, categories, `chats.category_id`, or unrelated SQLite tables.

## 8. Trigger Implementation Guardrails

If F16.4.c implements triggers, the migration must be guarded by the proof results from F16.4.b.

Trigger implementation must preserve:

- F7 folder metadata hash parity.
- Existing folder color, rename, move, delete, popup, menu, Native, Chrome, and Desktop behavior.
- Existing `folder-binding-changed` event behavior.
- Existing cache busting and library index refresh behavior.
- Existing return shapes for F7 wrapper callers.
- F15 `chat-folder` endpoint decision: folders remain `folder.metadata`, not `library.catalog`.

Trigger implementation must not:

- Delete F7 modules.
- Force F15 delegation default-on without proof.
- Remove explicit fallback while the compatibility window remains active.
- Use debug or emergency bypass as the normal compatibility path.
- Re-enable broad direct store writes.

## 9. Privacy And Side Effects

F16.4 proof and decision artifacts must be hash-only.

Forbidden raw values:

- Raw chat IDs.
- Raw folder IDs.
- Folder names.
- Folder colors.
- Paths.
- URLs.
- Content.
- Tokens.
- `folder_id`.
- `chat_id`.
- `category_id`.
- `chats.category_id`.

Docs and proofs may use safe enum strings such as:

- `folder_bindings`.
- `chat-folder`.
- `folder.metadata`.
- `f16.folder-legacy-fallback`.

## 10. Risks And Mitigations

| Risk | Impact | Mitigation |
|---|---|---|
| Immediate triggers break F7 fallback | Folder binding writes fail when delegation is off | Add scoped fallback identity before enabling triggers |
| Folder delete cleanup is missed | Folder removal leaves stale bindings or fails under trigger protection | Include folder removal cleanup in proof gates |
| Scoped identity becomes broad bypass | Protected SQLite model weakens | Limit identity to `folder_bindings` compatibility paths only |
| F15 delegation failure silently falls back | Operators lose auditability and conflict proof meaning | Require explicit fallback flag and report fallback identity usage |
| Mixed install versions diverge | Some installs protect `folder_bindings` while others do not | Make trigger activation a documented migration phase with closure proof |
| Synthetic proofs miss real UI caller path | Runtime writes fail outside proof coverage | Include `desktopSetFolderBinding`, event notification, cache bust, and index refresh paths in validators |
