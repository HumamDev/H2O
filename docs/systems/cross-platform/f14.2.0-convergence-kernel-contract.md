# F14.2.0 - Convergence kernel contract

> **Docs-only architecture contract.** F14.2.0 introduces no runtime code, no
> kernel implementation, no domain refactor, no Chat/Snapshot/Capture
> implementation, no label/category implementation, no remote apply, no
> automatic merge, and no mobile write-back.

F10 through F13 proved the convergence lane pattern across folder color,
rename, move, delete, and binding add. F14.1a through F14.1d extended the
review surface to Chat, Snapshot, Capture, and cross-domain behavior. Those
reviews show that the next domains need a shared safety kernel before mutation
or convergence logic expands beyond folders.

This document is the formal contract for that kernel. The kernel is a safety
and lifecycle foundation. It is not an apply engine.

## 1. Strategic thesis

Folder convergence proved that safe convergence is possible when every domain
follows a strict lane:

```txt
canonicalize -> materialize -> preflight -> proposal -> reviewed local apply -> applyEvent -> bookkeeping -> proof -> UI
```

That lane worked because each folder domain kept authority narrow, used
explicit operator approval, preserved auditability, and separated remote
evidence from local mutation.

The Chat, Snapshot, and Capture reviews proved broader requirements:

- Chat requires owner identity, content redaction, custody tracking, and
  stricter body-level privacy rules.
- Snapshot requires restore/recovery semantics, larger retention questions, and
  explicit trust boundaries for saved historical state.
- Capture requires evidence provenance, source custody, origin trust, and
  stricter handling of raw observed content.
- Cross-domain convergence requires stable lifecycle states, shared replay
  defense, shared consumed-operation handling, and consistent watermark
  monotonicity.

The conclusion is direct: the convergence kernel must be extracted before
Chat, Snapshot, or Capture implementation. New domains must not copy the folder
lanes and re-invent privacy, identity, replay, ledger, and lifecycle mechanics.

## 2. Three-layer architecture

F14 splits convergence into three layers:

| Layer | Name | Responsibility | Apply authority |
|---|---|---|---|
| L0 | Primitives | Shared low-level correctness and privacy helpers. | Never applies |
| L1 | Convergence library | Shared lifecycle, publication, replay, and handoff mechanics. | Never applies |
| L2 | Domain policy | Domain-specific materialization, blockers, recovery, and apply. | Domain-owned only |

The kernel includes L0 and L1. L2 remains outside the kernel forever.

## 3. L0 primitive contract

L0 primitives are small, deterministic, and fail-closed. They may validate,
shape, hash, scan, and read safety evidence. They must not apply, publish,
upload, download, enqueue, or mutate domain state.

| Primitive | Contract |
|---|---|
| Envelope helpers | Build and validate envelope basics without weakening F10.2 rules. |
| Identity kit | Normalize peer, install, sync, owner, and subject identity shapes. |
| Canonical JSON | Provide stable key ordering and canonical serialization. |
| Hashing | Provide SHA-256 helpers, hash validation, and digest construction. |
| Forever-no scanner | Reject or flag forbidden raw fields before return, store, or publish. |
| Redaction-class enforcer | Enforce `redacted` vs `device-local` output boundaries. |
| Watermark monotonicity service | Validate that watermark writes are forward-only and subject-scoped. |
| Consumed-op log primitive | Normalize append-only consumed-operation rows and duplicate checks. |
| Tombstone reader | Read F5 tombstone state without creating, deleting, or repairing it. |
| Preflight blocker vocabulary | Provide shared blocker/warning codes and dedupe helpers. |

L0 must be safe to reuse across folder, Chat, Snapshot, Capture, labels,
categories, and future domains.

## 4. L1 convergence library contract

L1 composes L0 primitives into shared convergence mechanics. L1 can coordinate
state transitions and safety checks, but it still cannot apply or mutate domain
state.

| Library primitive | Contract |
|---|---|
| Publication kit | Shared generated-to-published lifecycle over local artifacts and relay outbox handoff. |
| Replay-defense composer | Combine eventDigest, dedupeKey, lineage, watermark, and consumed-op checks. |
| Owner-handoff primitive | Validate that a platform may hand an artifact to the correct owner for review. |
| Origin/custody/trust evaluator | Separate who observed data, who carries it, and who is trusted to act. |
| Lifecycle state-machine framework | Derive current states from append-only rows without editing rows. |
| Audit/proof framework | Standard proof result shape, negative proof reporting, and audit linkage checks. |

L1 may return decisions such as `blocked`, `expired`, `duplicate`, `replay`,
`handoff-ready`, or `publication-ready`. It must not turn those decisions into
domain mutation.

## 5. L2 domain policy contract

Each domain owns its policy. Domain policy is not extracted into the kernel
unless it is proven to be a generic mechanic and cannot widen authority.

| Domain policy surface | Domain responsibility |
|---|---|
| Canonical body shape | Define the redacted, hashable subject body. |
| Field allowlist | Declare which fields may be proposed, previewed, or applied. |
| Owner identity | Define the peer or local owner allowed to review/apply. |
| Materialization semantics | Resolve hashes into local actionable state, if allowed. |
| Domain blockers | Define domain-specific conditions that fail closed. |
| Restore/recovery semantics | Define recovery snapshots, inverse operations, or retention rules. |
| Divergence classifier | Classify safe divergence, conflict, stale, replay, and blocked states. |
| Cascade rules | Define whether related objects must block, defer, or be reviewed together. |
| Apply | Perform the local mutation inside domain-owned gates. |

Domain apply remains L2 forever. The kernel can prove that a request is safe
enough to hand to L2. It cannot execute the handoff as mutation.

## 6. Hard rules

| Rule | Requirement |
|---|---|
| Kernel never applies | L0/L1 modules must not call domain apply APIs or write domain rows. |
| Domain apply remains L2 forever | Color, rename, move, delete, binding, Chat, Snapshot, Capture, labels, and categories own their own apply gates. |
| No L0 bypass | Domains may not bypass L0 for envelopes, identity, privacy scan, or watermark advancement. |
| Authority before handoff | Authority must be checked against `PLATFORM_KIND_ALLOW` before owner handoff. |
| No weakened contracts | Kernel helpers cannot weaken any F5, F6, F7, F10, F11, F12, or F13 safety contract. |
| Transport is not mutation | Relay, inbox, outbox, and publication remain transport or lifecycle only. |
| Evidence is not proposal | Evidence may justify a proposal but never becomes one implicitly. |
| Proposal is not apply | A proposal may request review but never mutates state. |
| applyEvent is evidence | An applyEvent records a completed local apply and is never a remote command. |
| Cache is not authority | Cached or projected state may accelerate display but cannot authorize mutation. |

## 7. Kernel invariants

| Invariant | Name | Contract |
|---|---|---|
| K1 | Monotonic watermark | Watermarks advance only forward for a peer, subject, lineage, and revision. Older, equal-conflicting, or unverified evidence cannot move them. |
| K2 | Forward-only emission | Generated artifacts move forward through lifecycle states. Old artifacts can be superseded, expired, withdrawn, blocked, or consumed, but not rewritten. |
| K3 | No automatic apply | No kernel primitive, relay event, publication event, inbox event, proof, or projection can apply automatically. |
| K4 | Trust != custody | A trusted peer, a storage relay, and an observed source are separate concepts. Custody of an envelope does not grant ownership or apply authority. |
| K5 | Quarantine over silent transform | Invalid, private, expired, replayed, malformed, or over-authorized artifacts are blocked or quarantined, not silently rewritten to pass validation. |
| K6 | Tombstones/consumed evidence outlive bodies | Safety evidence for deletion, consumed operations, and replay defense must survive body deletion or cache eviction for the configured retention window. |

These invariants are mandatory for all F14.2.x extractions and all later domain
adoption.

## 8. Extraction priority

F14.2.x extracts kernel pieces in dependency order:

| Phase | Scope | Acceptance focus |
|---|---|---|
| F14.2.1 | Envelope library kernel role | Envelope helpers are documented as shared mechanics without relaxing F10.2 validation. |
| F14.2.2 | Forever-no scanner + redaction-class enforcer | Privacy checks are centralized and stricter-or-equal. |
| F14.2.3 | Identity kit | Peer, install, sync, owner, and subject identity helpers are normalized. |
| F14.2.4 | Preflight blocker vocabulary + result shape | Blocker/warning codes and result builders stop drifting. |
| F14.2.5 | Watermark monotonicity service | Forward-only watermark checks become shared before new domains write watermarks. |
| F14.2.6 | Consumed-op log primitive | Consumed-operation append/list/dedupe mechanics become shared. |
| F14.2.7 | Tombstone reader / F5 handoff primitive | Destructive flows share tombstone reads and reviewed-delete handoff mechanics. |
| F14.2.8 | Publication kit consolidation | Proposal and conflictCandidate publication share lifecycle mechanics. |
| F14.2.9 | Replay-defense composer | Replay, dedupe, lineage, consumed, and watermark checks are composed consistently. |
| F14.2.10 | Owner-handoff primitive | Cross-platform and cross-domain handoff rules are centralized. |
| F14.2.11 | Lifecycle state-machine framework | Current state derives from append-only ledgers without row mutation. |
| F14.2.12 | Audit/proof framework | Proof harnesses and audit linkage checks get deterministic shared structure. |

No extraction phase may combine unrelated primitive groups unless a later review
explicitly authorizes the combined scope.

## 9. Domain adoption strategy

New domains use the kernel first:

- F14.3 Chat must use the kernel for identity, envelope validation, privacy
  scan, replay defense, consumed operations, and watermarks before any apply
  path exists.
- F14.4 Snapshot must use the kernel for custody, restore safety evidence,
  redaction, lifecycle state, and proof structure.
- F14.5 Capture must use the kernel for origin/custody/trust evaluation,
  evidence retention, privacy scan, and publication lifecycle.

Existing folder domains migrate opportunistically:

- Color, rename, move, delete, and binding are already proven and should not be
  force-rewritten.
- A folder lane may adopt a kernel helper only when the extraction is proven and
  public API output remains stable.
- No shipped folder lane should be rewritten purely for style consistency.

This strategy preserves the working dogfood system while preventing new domain
drift.

## 10. Risk analysis

| Risk | Failure mode | Mitigation |
|---|---|---|
| Kernel-as-apply trap | A shared helper becomes a generic mutation engine. | Hard rule: L0/L1 never call apply APIs or write domain rows. |
| Over-abstraction | Domain-specific hazards are hidden behind generic names. | Keep L2 policy explicit and require per-domain safety docs. |
| Privacy regression | Central scanner misses a domain-specific raw field. | Enforce stricter-or-equal scans plus domain allowlists and negative proofs. |
| Authority drift | Chrome, native, mobile, or relay gains unintended capability. | Check `PLATFORM_KIND_ALLOW` before owner handoff and publication. |
| Migration risk | Existing folder lanes change behavior during extraction. | New domains adopt first; folder lanes migrate one primitive at a time. |
| Single point of failure | Kernel bug affects every domain. | Keep primitives small, versioned, independently tested, and revertible. |
| Validator false confidence | Broad scans pass while semantic output changes. | Add parity fixtures and public API output comparisons per extraction. |
| Ledger coupling | Shared ledger helpers accidentally change row schemas. | Preserve existing schemas unless a versioned migration is explicitly approved. |
| Watermark corruption | A generic helper advances a stale or cross-peer watermark. | Central monotonicity checks must reject stale, replay, and mismatched lineage. |
| Trust/custody confusion | Relay possession is treated as authority. | Enforce K4 and owner-handoff checks before any review or apply path. |

## 11. Acceptance criteria for F14.2.x phases

Every F14.2.x phase must satisfy:

- No runtime behavior change unless that phase explicitly implements a kernel
  helper and proves behavior parity.
- No domain apply changes.
- No Chat, Snapshot, Capture, label, or category convergence implementation
  unless a later domain phase explicitly authorizes it.
- No remote apply.
- No mobile write-back.
- No automatic merge.
- No hidden polling or automatic sync.
- No package or lockfile changes.
- No weakening of F10.2 envelope validation.
- No weakening of F5 tombstone, F6 conflict, F7 metadata, F10.7 apply, F10.8
  relay, F10.9 rename, F11 move, F12 delete, or F13 binding safety rules.
- Existing public APIs remain stable unless the phase explicitly documents a
  versioned compatibility surface.
- Existing ledger schemas remain stable unless the phase explicitly documents a
  versioned migration.
- Privacy scans are stricter-or-equal.
- `git diff --check` passes.
- `node --check` runs for changed JavaScript files when code is introduced.
- Cross-platform repo scan passes when code is introduced.
- Cross-platform envelope validator passes when code is introduced.
- F7 folder metadata hash parity validator passes when sync code is introduced.
- Runtime proofs remain manual/operator-gated where mutation is involved.

For docs-only phases such as F14.2.0, acceptance is limited to documentation
scope and diff hygiene.

## 12. Preparation for future domains

### F14.3 Chat

The kernel gives Chat convergence a safe base for identity, custody, content
redaction, replay defense, consumed-operation tracking, and owner handoff.
Chat policy must still own body shape, edit semantics, conflict taxonomy,
recovery, and apply.

### F14.4 Snapshot

The kernel gives Snapshot convergence shared tombstone reads, consumed evidence,
watermark monotonicity, audit/proof shape, and privacy scanning. Snapshot
policy must still own restore semantics, retention windows, snapshot body
hashing, and recovery constraints.

### F14.5 Capture

The kernel gives Capture convergence origin/custody/trust evaluation,
publication lifecycle, redaction enforcement, and replay defense. Capture
policy must still decide what evidence can be materialized and what content
must remain evidence-only.

### Later labels/categories

Labels and categories should use the kernel before implementation. They are
closer to folder metadata and binding lanes, but still need shared identity,
publication, watermark, consumed-operation, and replay primitives to prevent a
second generation of duplicated convergence code.

## 13. MUST / SHOULD / DEFER

### MUST

- Keep F14.2.0 docs-only.
- Treat the kernel as L0/L1 safety infrastructure, not an apply engine.
- Keep domain apply in L2 forever.
- Require L0 for envelopes, identity, privacy scans, and watermark advancement.
- Check authority against `PLATFORM_KIND_ALLOW` before owner handoff.
- Preserve all existing domain safety contracts.
- Keep `applyEvent` evidence-only.
- Keep transport and publication separate from mutation.
- Preserve append-only ledger semantics.
- Fail closed on privacy, replay, stale base, and authority uncertainty.

### SHOULD

- Extract one primitive group at a time.
- Add deterministic parity fixtures with each kernel helper.
- Make each kernel helper versioned and independently revertible.
- Prefer pure functions for L0 wherever possible.
- Keep storage writes behind explicit ledger primitives.
- Adopt kernel helpers in new domains before migrating folder lanes.
- Add domain-specific negative proofs before any new apply path.

### DEFER

- Generic convergence engine.
- Automatic merge.
- Automatic apply.
- Remote apply.
- Mobile write-back.
- Chat/Snapshot/Capture mutation.
- Labels/categories convergence.
- Domain lane rewrites for style only.
- Ledger compaction or migration.
- WebDAV authority expansion.

## 14. F14.2.0 status

F14.2.0 defines the convergence kernel contract only. No kernel implementation
exists in this phase, no runtime file is changed, and no domain behavior is
modified.
