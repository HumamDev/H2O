# F19.5 Premium Chrome/Desktop Sync Closure Evidence

Status: F19.5 closure evidence contract.

## Architecture Verdict

F19.5 is an evidence phase. It does not add a backend, identity UI, signing/notarization, or new propagation behavior.

Premium Sync cannot be declared complete from Node validators alone. Closure requires a live operator proof from both surfaces:

- Chrome Studio running in the MV3 extension context.
- Desktop Studio running in the Tauri runtime.
- Both configured to the same local sync folder.
- `chrome-latest.json` and `latest.json` both observed in that folder.
- F19.1 hash-only parity snapshots captured before and after both propagation directions.
- F19.2.b Chrome -> Desktop guarded import executed.
- F19.2.c Desktop -> Chrome guarded import executed.
- F19.4 hardening taxonomy present in propagation outputs.
- F19.5.b Chrome export-source coverage proves that `chrome-latest.json`
  covers the same supported Library row set that Chrome parity counts.
- F19.5.d Desktop -> Chrome import convergence proves that `latest.json`
  shell / linked rows materialize into the Chrome LibraryIndex-visible row set.

Without a validated live proof JSON, the final F19.5 verdict is `blocked` for Premium Sync closure.

## F19.5.b Chrome Export-Source Coverage Gate

Chrome Studio has two different read surfaces:

- `H2O.Studio.sync.libraryParity.captureSnapshot()` reads the live
  `H2O.LibraryIndex.getAll()` view that powers the Studio Library UI.
- `H2O.Studio.sync.autoImport.exportNow()` writes `chrome-latest.json` by
  asking the MV3 archive background for `exportFullBundle()`.

Those two sources are not automatically equivalent. F19.5 is blocked when
Chrome parity sees supported rows that do not appear in `chrome-latest.json`.
The required blocker code is:

`chrome-export-source-coverage-mismatch`

The Chrome exporter must report a redacted `chromeExportCoverage` object with:

- schema `h2o.studio.sync.chrome-export-coverage.v1`;
- `snapshotTotal`, `snapshotSaved`, `snapshotLinked`, `snapshotPinned`,
  and `snapshotArchived` from the Chrome LibraryIndex snapshot source;
- `bundleChatCount`, `bundleSavedCount`, `bundleLinkedCount`,
  `bundlePinnedCount`, and `bundleArchivedCount` from the transport bundle;
- `missingRowCount` and `missingRowTypeCounts`;
- privacy flags proving no raw IDs, titles, or content are returned in the
  diagnostic.

F19.5 live proof may continue only when:

- `chromeExportCoverage.ok === true`;
- `chromeExportCoverage.snapshotTotal === chromeExportCoverage.bundleChatCount`;
- `chromeExportCoverage.blockers` does not include
  `chrome-export-source-coverage-mismatch`;
- the export result has no blockers.

The selected F19.5.b source policy is `library-index-supported-rows`: Chrome
export aligns `chrome-latest.json` to the visible Chrome LibraryIndex supported
row set. When a LibraryIndex row is missing from the transcript archive bundle,
Chrome may add a minimal zero-snapshot row that carries saved/linked/pinned/
archived/category state only. It must not invent transcript messages or raw
content for linked-only rows. Labels, tags, projects, chat-folder bindings,
tombstones, sync apply events, and unsupported storage/KV fields remain
deferred unless implemented in a later phase.

## F19.5.d Desktop -> Chrome Convergence Gate

Desktop `latest.json` may contain zero-snapshot shell rows: saved, linked, or
imported/link-bearing Library rows that have state and catalog metadata but no
transcript content. Imported/link-bearing rows with no saved or linked state are
supported as total-only rows: they increment `total`, but must not increment
`saved` or `linked`.
The Chrome import path must not report success merely because the archive
background accepted the bundle. Closure requires a redacted convergence proof
after `latest.json` import:

- `sourceSummary.chatCount`, `savedCount`, `linkedCount`, `pinnedCount`,
  `archivedCount`, `folderCount`, and `categoryCount` are present;
- `importSummary.shellRowsIncoming`, `shellRowsMaterialized`,
  `shellRowsExisting`, `shellRowsSatisfied`, and `shellRowsFailed` are present;
- `importSummary.redactedErrorCategories` carries counts/codes only;
- `convergence.ok === true` and `convergence.mismatchCount === 0`;
- `total`, `saved`, `linked`, `folderCount`, and `categoryCount` converge
  exactly; positive incoming pinned/archived counts must be satisfied, but
  Chrome-local pinned/archived extras are allowed under merge semantics;
- no blocker `desktop-shell-row-import-unsupported`;
- no blocker `desktop-to-chrome-convergence-not-proven`.

Chrome may materialize shell rows into its Studio Chat Registry and then refresh
LibraryIndex. It must not invent transcript messages, prompts, responses, URLs,
or raw content. If shell rows cannot be represented safely, F19.5 remains
blocked with the blocker above instead of silently accepting the transport.

## F19.5.e Operator-Approved Merge Protocol

Desktop `latest.json` carries Desktop export-log lineage. During a live proof,
Chrome may already have a different `lastAppliedExportId` from an earlier
Desktop export while the current Desktop union export references Desktop's own
`previousExportId`. The default behavior is correct: Chrome blocks with
`library-propagation-simultaneous-update-conflict` / `simultaneous-update-conflict`
before writing.

For an operator-controlled local proof merge, use an explicit decision:

```js
const f19DesktopToChrome = await H2O.Studio.sync.folder.syncNow({
  reason: "f19.5-live-proof",
  conflictDecision: "approve-merge"
})
```

Approval requirements:

- approval is never default;
- only `library-propagation-simultaneous-update-conflict` may be approved;
- stale transport remains blocked;
- duplicate checksum idempotency remains preserved;
- the result must include `conflictDecision: "approve-merge"`;
- the result must include `conflictApproved: true` and a `conflictApproval`
  object with `staleTransportStillBlocks: true`;
- privacy remains redacted and no raw IDs/titles/content are returned.

Preferred clean proof run:

1. Disable Chrome auto-sync during proof unless explicitly testing it.
2. Use a fresh Chrome export to `chrome-latest.json`.
3. Import on Desktop, then write a fresh Desktop `latest.json`.
4. Run Chrome import with `conflictDecision: "approve-merge"` only if the
   simultaneous-update blocker appears.
5. If convergence still fails, keep F19.5 blocked and use the redacted blocker
   returned by the import result.

## Closure Verdicts

| Verdict | Meaning |
|---|---|
| `complete` | Live proof validates and all supported plus previously deferred fields converge. This requires labels, tags, projects, chat-folder bindings, tombstones, sync apply events, and unsupported storage/KV to be implemented or absent. |
| `conditionally complete for supported fields` | Live proof validates with `supportedMismatchCount: 0`, both propagation directions pass or are idempotent, privacy is clean, F19.4 hardening summaries are present, merge-local fields are explicitly classified, and deferred fields remain explicitly listed. This may be called `Premium Sync v1 supported fields complete`, not unqualified `Premium Sync complete`. |
| `blocked` | No live proof JSON exists, proof validation fails, supported field mismatches remain, a propagation direction fails, transport files are missing, privacy leaks occur, or deferred fields are hidden. |

Current F19.5 v1 supported fields:

- total Library rows,
- saved count,
- linked count,
- folders count/hash,
- categories count/hash,
- recents hash,
- supported row hash.

Current merge-local fields:

- pinned count,
- archived count.

Pinned and archived are local/merge-safe row state for F19.5. A positive
incoming pinned/archived count must be satisfied during direction-specific
import convergence, but a Chrome-local or Desktop-local pinned/archived extra
does not block final supported-field parity. The final proof must report those
mismatches under `mergeLocalMismatches` / `mergeLocalMismatchCount`, not hide
them and not count them as supported mismatches.

Current deferred fields:

- labels,
- tags,
- projects,
- chat-folder bindings,
- tombstones,
- sync apply events,
- unsupported storage/KV fields.

Deferred fields must stay explicit. They do not block `Premium Sync v1 supported fields complete`; they do block the unqualified phrase `Premium Sync complete` unless product scope accepts a v1-supported-fields definition.

## Shared Sync Folder Setup

Use one operator-chosen folder accessible to both surfaces.

Expected files:

- `chrome-latest.json`: Chrome-origin export consumed by Desktop.
- `latest.json`: Desktop-origin export consumed by Chrome.

Filesystem checks:

```bash
SYNC_FOLDER="/absolute/path/to/H2O Studio Sync"
test -f "$SYNC_FOLDER/chrome-latest.json"
test -f "$SYNC_FOLDER/latest.json"
shasum -a 256 "$SYNC_FOLDER/chrome-latest.json" "$SYNC_FOLDER/latest.json"
ls -l "$SYNC_FOLDER/chrome-latest.json" "$SYNC_FOLDER/latest.json"
```

The proof must not copy raw transport payloads into release evidence. Only hashes, sizes, timestamps, and redacted summaries are allowed.

## Pass Criteria

F19.5 may report `conditionally complete for supported fields` only when:

- the proof JSON validates with `validate-f19-live-parity-proof.mjs --proof`;
- `latest.json` and `chrome-latest.json` were observed in the shared folder;
- Chrome -> Desktop propagation is `ok:true` or idempotent;
- Chrome export source coverage is `ok:true` and
  `snapshotTotal === bundleChatCount`;
- Desktop -> Chrome propagation is `ok:true` or idempotent;
- Desktop -> Chrome convergence is `ok:true` with `mismatchCount: 0`;
- if simultaneous conflict was approved, Desktop -> Chrome result records
  `conflictDecision:"approve-merge"` and `conflictApproved:true`;
- final `supportedMismatchCount` is `0`;
- final supported fields match;
- final merge-local pinned/archived mismatches, if present, are reported under
  `mergeLocalMismatches` and do not appear in `supportedMismatches`;
- deferred fields are present in `deferredFields` and/or propagation warnings when present in payload;
- privacy is `ok:true` with `rawLeakCount:0`;
- no raw forbidden values appear in proof output;
- no unexpected side effects are reported;
- F19.4 hardening taxonomy is present in propagation output.

F19.5 remains `blocked` when:

- no live proof JSON is created;
- proof validation fails;
- supported mismatches remain;
- either propagation direction fails for supported fields;
- Chrome export coverage is missing or includes
  `chrome-export-source-coverage-mismatch`;
- Desktop -> Chrome import reports `desktop-shell-row-import-unsupported` or
  `desktop-to-chrome-convergence-not-proven`;
- simultaneous conflict approval is missing, implicit, or attempts to approve
  stale transport;
- sync folder files are missing;
- duplicate/stale/conflict/permission failures are hidden;
- deferred fields are hidden;
- proof output leaks raw IDs, titles, names, paths, URLs, tokens, content, `category_id`, `folder_id`, or `chat_id`.

## Release Evidence Placement

Recommended out-of-tree path:

```bash
../release-evidence/f19.5/<headShortSha>/f19.5-live-parity-proof.json
```

Do not write the live proof inside the source tree unless it is explicitly redacted and intentionally committed.

## Final Closure Rule

If no validated live proof JSON exists, answer:

`F19.5 blocked: live Chrome/Desktop parity proof not captured.`

If the live proof validates and only deferred fields remain, answer:

`F19.5 conditionally complete for Premium Sync v1 supported fields.`

Only answer:

`Premium Sync complete`

when supported and deferred fields all converge or the product scope explicitly redefines Premium Sync as the supported-field v1 subset.
