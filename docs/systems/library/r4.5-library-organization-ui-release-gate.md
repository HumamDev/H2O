# R4.5 Library Organization UI Release Gate

## Status

R4.5 records the Desktop-first Library Organization UI gate. Single-item
operations (R4.5.1.a through R4.5.3) and multi-select batch operations
(R4.5.4) are proven for all four catalog targets: folders, categories,
labels, tags.

Gate decision: the Desktop Studio is now the canonical owner of Library
organization workflows on the Desktop platform. The Native UI on
chatgpt.com is preserved as the MV3 fallback path and as the turn-level
tag extraction owner (R4.3 boundary). **Native UI deprecation is not
part of this gate.**

## Proven End to End

### Single-item organization (R4.5.1.a / R4.5.2 / R4.5.3)

- Desktop Studio operator can create, rename, recolor, and delete each
  catalog target (folders, categories, labels, tags) from the sidebar
  or programmatically via `H2O.Studio.OrganizationModals.openXxxEditor`.
- Catalog modes per target:
  - Folders: `create / rename / color / delete`
  - Categories: `create / rename / delete` (no color column)
  - Labels: `create / rename / color / delete`
  - Tags: `create / rename / delete` (no color, no extraction)
- Delete confirmation is enriched with the target name and the bound-
  chat count via `LibraryIndex.facets()` (best-effort, falls back to
  name-only when the facet is unavailable).
- Each successful mutation routes through `H2O.Studio.actions.<target>.*`
  which dispatches the canonical `evt:h2o:library-index:refresh-request`
  with reason `<target>-actions:<op>`.
- The S0Z1g sidebar's existing per-row rename / delete handlers (which
  previously called Native `H2O.archiveBoot.renameCategory` /
  `H2O.Labels.renameLabel` etc. directly) now branch through
  OrganizationModals on Desktop before falling through to the existing
  Native ladder on MV3. The MV3 path is preserved verbatim.
- The S0Z1g sidebar gains four new `data-h2o-<catalog>-create-button`
  helpers — one per catalog — for Desktop. Tags create button is
  defensive: it targets `.wbSidebarSection--tags` which is not yet
  rendered; the helper bails on missing section and will auto-mount
  the moment a future slice adds the section.

### Multi-select batch organization (R4.5.4)

- `H2O.Studio.BatchToolbar` exposes a selection API
  (`add / remove / clear / has / size / all`) plus `enable / disable /
  isEnabled / diagnose`.
- When enabled, a sticky bottom toolbar mounts when `size() > 0`. It
  displays "N chats selected" and exposes five action buttons: Set
  Folder / Set Category / Add Label / Add Tag / Clear Selection.
- Multi-select modifiers on Explorer chat rows (`.wbChatRow[data-chatId]`):
  - Plain click: existing navigation (unchanged); updates `lastAnchor`.
  - Cmd/Ctrl+click: toggle selection (preventDefault + stopPropagation).
  - Shift+click: DOM-order range select from `lastAnchor`.
  - Click on injected checkbox: toggle selection.
- The "checkbox column" is rendered via DOM mutation through a
  `MutationObserver` that watches Studio body for newly-rendered
  `.wbChatRow` elements. The observer is **Studio-internal** — it
  never observes chatgpt.com structure.
- Batch actions fan out via `Promise.all` over selected chats, calling
  `H2O.LibraryActions.<method>` per chat.
- Refresh strategy:
  - Each `H2O.LibraryActions.*` call dispatches its own
    `evt:h2o:library-index:refresh-request` when it succeeds.
  - S0F1c (LibraryIndex) guards on `state.refreshInFlight` — N parallel
    dispatches sharing the in-flight Promise collapse to ~1 actual
    refresh.
  - AFTER `Promise.all` settles, S0F1n dispatches ONE additional
    refresh with reason `batch-toolbar:<op>:<count>`.
  - Worst case: 2 refreshes. Best case: 1 refresh. Either way,
    dramatically better than N.
- Adversarial-review-driven hardening:
  - `state.opInProgress` guard rejects concurrent batch ops while one
    is in-flight (prevents out-of-order refresh dispatch from rapid
    back-to-back button clicks). `try` / `finally` guarantees release.
  - `selectionRemove` clears `state.lastAnchor` when the removed chat
    matches the anchor (prevents stale anchor → Shift+click falling
    back to plain-add).

### Hard boundary preserved (R4.3)

- **Turn-level tag extraction stays in Native 0F5a.** Studio is
  catalog-only. The matrix validator asserts that no UI module
  (S0F1m / S0F1n / S0Z1g) contains any of the forbidden patterns:
  `chatgpt.com`, `data-testid="conversation-turn"`,
  `data-message-author-role`, `extractTagsFromTurn`, `deriveTags`,
  `scanTurn`.
- `H2O.Studio.OrganizationModals.diagnose()` reports
  `tagExtraction: false`, `targets.tags.extraction: false`,
  `targets.tags.observesChatGptDom: false`.
- `H2O.Studio.BatchToolbar.diagnose()` reports
  `tagExtraction: false`, `domAccess: false`,
  `observesChatGptDom: false`.
- Native `src-runtime-base/0F5a. Tags.js` (273 KB) remains the
  extraction owner. The R4.5.5 matrix validator confirms it still
  exists, is non-empty, and still contains `MutationObserver` or
  turn-observation patterns — proving the extraction code path lives
  on the Native side and was NOT duplicated into Studio.

### What the new matrix validator (R4.5.5) covers

The new `validate-studio-library-organization-ui.mjs` is organized into
8 sections, totaling 107 assertions:

- **A. Studio actions modules** — for each target (folders, categories,
  labels, tags), assert the module is Tauri-gated, registers on
  `H2O.Studio.actions.<target>`, exposes `create / rename / remove`,
  has the target-appropriate binding op (`bindChat` or `assignChat`),
  dispatches the canonical refresh with `<target>-actions:<op>`
  reason, and contains no `chrome.*` runtime calls.
- **B. LibraryActions facade** — for each of `setFolder / setCategory /
  setLabels / addLabel / removeLabel / setTags / addTag / removeTag`,
  assert the method exists and routes through
  `H2O.Studio.actions.<target>` on Desktop. Confirm at least 10
  `native-context-required` references (MV3 fallback per method) and
  no direct `plugin:sql` calls.
- **C. OrganizationModals** — assert all 4 editors exist with the
  documented modes per target (folders + labels = 4 modes including
  color; categories + tags = 3 modes with no color, no extraction).
  Confirm each handler calls `H2O.Studio.actions.<target>.*` and that
  no Native mutation API, no `plugin:sql`, and no `dispatchEvent` is
  present (single-source refresh via actions.*).
- **D. BatchToolbar** — assert Tauri-gated, registers
  `H2O.Studio.BatchToolbar`, exposes selection API + enable/disable/
  isEnabled/diagnose, routes through `H2O.LibraryActions` facade (not
  actions.* directly), uses `Promise.all` fan-out + ONE final
  `batch-toolbar:<op>:<count>` refresh, has the `opInProgress` guard
  (review fix #1), and clears `lastAnchor` on anchor removal (review
  fix #2).
- **E. Sidebar S0Z1g re-wiring** — assert the folder-create button
  branches through OrganizationModals on Desktop (R4.5.1.a); the
  promptRenameItem and deleteMenuItem functions have Desktop branches
  for categories (R4.5.2) and labels (R4.5.3) INSERTED BEFORE the
  basic `W.confirm` to avoid double-prompting; ensure*CreateButton
  helpers exist for all four catalogs with distinct data attributes;
  the MV3 ladder (`H2O.archiveBoot.*` / `H2O.Labels.*`) is preserved.
- **F. Cross-file boundary** — for each UI module (S0F1m / S0F1n /
  S0Z1g), scan source-with-comments-stripped for the 6 forbidden
  patterns. Assert `MutationObserver` appears ONLY in S0F1n bounded
  to the Studio-internal `.wbChatRow[data-chatId]` selector. Confirm
  diagnose markers `domAccess:false`, `observesChatGptDom:false`,
  `tagExtraction:false`.
- **G. Refresh event single-source** — assert no novel
  `evt:h2o:batch-toolbar:*` / `evt:h2o:modals:*` event names exist;
  reasons follow `<target>-actions:<op>` and `batch-toolbar:<op>:<count>`
  patterns; modals dispatches no events.
- **H. Native 0F5a extraction owner** — assert the file still exists,
  is substantial (>= 100 KB), and still contains MutationObserver or
  conversation-turn observation patterns. Proves the extraction code
  path lives on the Native side and was not duplicated into Studio.

## Rollback strategy

R4.5.x ships as Desktop-first additive surfaces. No production runtime
flags need to be flipped to "enable" the work — the surfaces are
automatically active on Desktop because the modules are Tauri-gated
at load.

If a regression surfaces:

- **Toolbar regression** (R4.5.4): remove the `<script>` tag for
  `S0F1n` from `studio.html` and re-run prepare-dist. The toolbar
  silently disappears; existing single-item organization continues to
  work.
- **Modals regression** (R4.5.1.a / R4.5.2 / R4.5.3): remove the
  `<script>` tag for `S0F1m`. The sidebar's existing per-row handlers
  fall through to the MV3 ladder (which works the same way on Desktop
  via the canonical-folder-create / canonical category service paths).
  Pre-R4.5 Desktop folder/category/label rename/delete UX is restored.
- **S0Z1g re-wiring regression**: the re-wired handlers all have a
  `try { var modals = ... } catch (_) { /* fall through */ }` pattern
  with the MV3 ladder unchanged underneath. If modals dereferences
  fail, the existing path executes. No explicit rollback needed.
- **Actions module regression** (R4.1–R4.4): each actions module is
  the SQLite write path. Removing one breaks Desktop write capability
  for that target. Revert the corresponding R4.x commit, NOT the R4.5
  UI commits.

No persistence-layer schema changes were introduced in R4.5; rolling
back UI never requires a SQLite migration.

## Deferred — R4.6 Native UI deprecation (NOT in this gate)

The Native sidebar UI on chatgpt.com (in `src-runtime-base/0F*.js`)
still contains category rename/delete UI that calls
`H2O.archiveBoot.renameCategory` etc. directly. Similarly, the Native
labels UI uses `H2O.Labels.renameLabel`. R4.5.x **preserved** all of
these as the MV3 fallback path.

R4.6 is a separate, dedicated effort to:

1. Decide whether to remove the Native organization UI entirely or to
   keep it as a co-existing UI with explicit deprecation messaging.
2. If removing: identify every Native call site for catalog mutations
   and either remove the UI affordance or route it through the
   Studio facade via cross-surface message-passing.
3. If keeping with deprecation: add a deprecation banner in the
   Native UI pointing users to Desktop Studio + sunset timeline.
4. Update the platform support matrix and user-facing docs.

R4.6 must NOT break the MV3 organization workflow until the
replacement (Chrome Studio batch toolbar, Chrome Studio modal layer,
or full deprecation) is in place. R4.5.5's matrix validator includes
explicit assertions that the MV3 fallback ladder remains intact in
S0Z1g, so R4.6 work cannot accidentally regress the boundary without
the validator catching it.

Other work explicitly out of scope for R4.5.5:

- Drag-selection in the Explorer batch toolbar (Cmd/Ctrl/Shift click
  patterns ship; drag deferred to R4.5.4.b or later if needed).
- A tags sidebar section in S0Z1g (currently no `renderTags` exists
  alongside `renderFolders` / `renderCategories` / `renderLabels`).
  `ensureTagCreateButton` is in place and will auto-mount the moment
  the section is added.
- `setSnapshotCategory` refactor — `S0F1b.desktopSetSnapshotCategory`
  still writes to `store.categories.assignChat / clearChat` directly
  rather than through `H2O.Studio.actions.categories.assignChat`. The
  R4.5.2 validator includes a "deferred" assertion that records the
  current implementation so accidental regressions are caught. A
  dedicated R4.5.2.b slice can refactor with R4.4-style regression
  assertions.

## Explicit statement on Native UI

**Native UI is not removed in R4.5.5.** The chatgpt.com Native
sidebar and its category / label / folder mutation handlers remain
fully functional. R4.5.x added a Desktop-first ALTERNATIVE; it did
NOT replace the Native UI.

Validator-level guarantees that Native UI is preserved:

- `H2O.archiveBoot?.renameCategory` and `H2O.archiveBoot?.deleteCategory`
  references in S0Z1g's `promptRenameItem` and `deleteMenuItem` are
  asserted to still exist (the MV3 fallback ladder).
- `H2O.Labels?.renameLabel` and `H2O.Labels?.deleteLabel` references
  are asserted to still exist.
- The folder-create Native panel (`openFolderCreatePanel(button)`) is
  asserted to still be the MV3 fallback for the sidebar "+" button.

The matrix validator (Section H) additionally asserts that Native
0F5a Tags module is still present, substantial, and still contains
the extraction code patterns — proving extraction was NOT moved into
Studio.
