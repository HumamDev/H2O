## Purpose

This is the **versioned umbrella** for the H2O Studio saved-chat package format. It
indexes the per-version schema specs and defines how versions relate, so the
package contract can grow (assets, future fields) without breaking already-written
packages.

It is a **design contract only**. This Phase C C1 patch implements no code, adds no
migration, changes no Tauri capability, and ships no CAS. Schema **v2** described
here is a **future** (design-only) contract; **v1 remains the only implemented
version**.

- **Schema v1** = the existing **asset-less Phase B** contract. Its authority is
  [saved-chat-package-v1.md](saved-chat-package-v1.md). Nothing in this document
  changes v1.
- **Schema v2** = the **future asset-capable Phase C** contract, defined here at
  the design level. A separate `saved-chat-package-v2.md` will be authored when v2
  is implemented; until then this umbrella holds the v2 design.

## Phase C verdict (summary)

The binding decisions live in [ADR-0010](../../decisions/ADR-0010-saved-chat-asset-cas.md).
In short:

- Phase C is **Desktop-only**. Desktop Studio owns archive, index, and the content
  CAS.
- **Chrome remains light** — it may request/hand off a save, but never materializes
  packages and never owns the CAS.
- **Sync is later transport only.** Phase C integrates no sync.
- **Packages remain deterministic projections**, never a second live mutable store.

## Schema Version Matrix

| Version | Status | Asset-capable | `payloadVersion` | `contentHash` construction | Authoritative spec |
|---|---|---|---|---|---|
| **v1** | Implemented (Phase B) | No (`assets: []`) | 1 | `sha256(canonical snapshot.json)` | [saved-chat-package-v1.md](saved-chat-package-v1.md) |
| **v2** | Design-only (Phase C, future) | Yes | 2 | `sha256(canonical preservation-payload descriptor)` | this umbrella → future `saved-chat-package-v2.md` |

Rules:

- A package with no assets **may remain v1** and re-projects byte-identically. v1
  packages are never auto-upgraded.
- v2 is used **only when a package carries assets** (or is generated by a v2-aware
  writer that opts in).
- Readers/validators **must branch on `schemaVersion` (and `payloadVersion`)**. A
  v1 reader must reject v2 cleanly with a clear message; a v2 reader must validate
  both v1 and v2.
- `manifest.schema` (`h2o.savedChatPackage`) and `snapshot.schema`
  (`h2o.savedChatSnapshot`) are unchanged across versions; only `schemaVersion` /
  `payloadVersion` and the asset-bearing fields differ.

## Asset Descriptor Shape (v2 `manifest.assets[]`)

Each entry in `manifest.assets[]` describes one content-addressed binary copied
into the package at materialization:

| Field | Required | Type | Semantics |
|---|---|---|---|
| `sha256` | yes | string | `sha256-<hex>` over the **raw asset bytes**. This is the asset **identity**. |
| `path` | yes | string | Package-relative path, `assets/sha256-<hex>.<ext>`. `/` separators only. |
| `mimeType` | yes | string | e.g. `image/png`. |
| `ext` | yes | string | File extension matching `path` (e.g. `png`). |
| `byteLength` | yes | number | Raw byte length. |
| `originalName` | no | string | **Display-only**, sanitized. **Never part of any hash.** |
| `source` | yes | string | `chatgpt-capture` \| `import` \| `manual`. |
| `sourceMessageId` *(or `turnRef`)* | no | string | Provenance: which message/turn the asset came from. |
| `alt` | no | string | Image alternative text (display). |
| `caption` | no | string | Optional caption (display). |

Clarifications:

- **`sha256` is the identity.** Dedup, references, and integrity all key on it.
- **`originalName` / `alt` / `caption` / `mimeType` / `byteLength` / `path` are
  metadata** and are **excluded from `contentHash`** (see Hash Contract). Only the
  asset `sha256` set participates in the preservation hash.
- **`path` is package-relative and content-addressed**, e.g.
  `assets/sha256-<hex>.png`. Asset filenames are the hash, so they are
  deterministic and collision-free.

## Snapshot Integration (v2)

- **Per-message `assetRefs`** (already reserved in `snapshot.json`
  `messages[].assetRefs`) hold **a list of `sha256-` ids only** — the assets a turn
  uses. No descriptor data is duplicated into messages.
- **`manifest.assets[]`** holds the **package-level descriptors** (paths, mime,
  size, provenance). Identity (`sha256`) is the join between the two.
- **No duplicated descriptors inside messages.** Messages reference; the manifest
  describes.
- **`contentHtml` references package-relative asset paths after materialization.**
  Inline `data:` / blob image sources in captured HTML are extracted to the CAS and
  rewritten to relative `assets/sha256-<hex>.<ext>` paths so `chat.html` is
  offline-faithful and `snapshot.json` carries no host-specific bytes. The rewrite
  is deterministic (path = content hash).

## Minimal First Asset Scope (v2)

Phase C's first asset slice is intentionally narrow:

- **Images only** — `png`, `jpeg`, `gif`, `webp`.
- **Only bytes already present** — inline `data:image/*` URLs in captured HTML, and
  imported file bytes.
- **No remote URL fetching.** Resolving remote image URLs is a *capture* concern
  (cross-origin / CSP), not a projector concern — deferred.
- **PDFs and other files come later**, after the image path is proven. The
  descriptor shape above is already file-agnostic so no schema change is needed to
  add them.

## Hash Contract

All hashes use SHA-256, lowercase hex, `sha256-` prefix. Per-file
(`files.*.sha256`) and per-asset (`assets[*].sha256`) hashes are recorded
**separately** in both versions for tamper detection; they do not define preserved
chat identity.

### v1 (preserved, unchanged)

- `schemaVersion: 1`, `payloadVersion: 1`.
- `manifest.contentHash = sha256(canonical snapshot.json bytes)`.
- With no assets, `contentHash === files.snapshot.sha256`.

### v2 (design)

- `schemaVersion: 2`, `payloadVersion: 2` — used for asset-bearing packages
  (asset-less packages may remain v1).
- `manifest.contentHash = sha256(canonical preservation-payload descriptor)`, where
  the descriptor is the canonical JSON:

  ```json
  { "snapshot": "sha256-<snapshot.json hash>", "assets": ["sha256-...", "..."] }
  ```

- **`assets` is sorted by hash** (deterministic order, independent of message
  order).
- **`originalName`, `mimeType`, `byteLength`, `path` are excluded** from
  `contentHash` — only the snapshot hash and the sorted asset `sha256` set
  participate. This keeps the hash stable across machines and re-projections.
- `files.snapshot.sha256` still equals `sha256(canonical snapshot.json)`; in v2 it
  is one input to the payload descriptor, not the whole `contentHash`.

### Versioning / back-compat rules

- Validators **branch by `schemaVersion` / `payloadVersion`**.
- **v1 packages remain valid forever**; a v2-aware validator validates both.
- An asset-less chat re-projects deterministically as v1 (byte-identical),
  preserving the Phase B guarantee.
- `payloadVersion` is emitted explicitly so the `contentHash` construction is
  self-describing, not inferred.

## Live CAS vs Package Export Layout (C3.0, locked)

The **live content-addressed store** and the **per-package export copy** use
**intentionally different layouts**. The binding decision is recorded in
[ADR-0010 → "C3.0 — Live CAS Layout & Boundaries"](../../decisions/ADR-0010-saved-chat-asset-cas.md);
summarized here:

| Aspect | Live CAS (app-owned, internal) | Package export copy (portable) |
|---|---|---|
| Root | `$APPLOCALDATA/archive/assets` | `<chatId>.h2ochat/assets/` |
| Blob path | `archive/assets/<aa>/sha256-<hex>` (`<aa>` = first 2 hex chars) | `assets/sha256-<hex>.<ext>` |
| Extension | **extension-less** (hash only) | **includes `.<ext>`** (from manifest `ext`/`mimeType`) |
| Directory shape | prefix-sharded (avoids huge flat dir) | flat per package |
| Resolution | `get`/`exists` by hash alone, **no registry lookup** | descriptor-driven (`manifest.assets[*]`) |

C4 applies the `.<ext>` when copying a CAS blob into a package. The live store is
never required to know an asset's extension to read it back; `ext`/`mimeType` are
registry/manifest metadata only.

## Materialized Package Location & Write Form (C4.0, locked)

C4 **materializes packages into the app-owned archive**, not a user folder. The
binding decision (and the write order / boundary rules) is recorded in
[ADR-0010 → "C4.0 — Package Materialization with Assets"](../../decisions/ADR-0010-saved-chat-asset-cas.md);
summarized here:

- **Materialization root:** `$APPLOCALDATA/archive/packages/<chatId>.h2ochat/`.

  ```text
  $APPLOCALDATA/archive/packages/<chatId>.h2ochat/
  |-- manifest.json
  |-- snapshot.json
  |-- chat.md
  |-- chat.html
  `-- assets/
      `-- sha256-<hash>.<ext>
  ```

- **Why app-owned, not a user folder:** reuses the C2a capability
  (`$APPLOCALDATA/archive/**`) and avoids adding broad user-folder filesystem
  permissions now. **User-folder export / save dialog is deferred.**
- **Binary write only:** every package file under the archive is written with the
  tauri-plugin-fs **v2 binary `write_file` body+headers form** (path/options in
  headers, bytes in body) — `write_text_file` is **not** granted under the archive
  scope. JSON/MD/HTML are written as UTF-8 bytes. The C3.3 real-Desktop smoke
  proved this form works with `baseDir: 15` (commit `e3d5bb9`).
- **Write order:** package asset files first, then `manifest.json` / `snapshot.json`
  / `chat.md` / `chat.html` — so an interruption never yields a complete-looking
  package whose `snapshot.json` references a missing asset.
- **First C4 asset scope:** inline `data:image/*` (png, jpeg/jpg, gif, webp) already
  present in sanitized `contentHtml`; no remote fetch, no PDFs/files, no
  generated-file capture.

## Desktop Archive / Index Design (future ownership)

Design-only; no table or migration is added in C1.

- **Reuse `chats` / `snapshots` / `snapshot_turns` as the source of truth.** Do not
  build a parallel archive store.
- **`assets` registry table (landed in C2b as SQLite Migration v14)** is the
  canonical content registry: PK `sha256`, plus `mime_type`, `ext`, `byte_size`,
  `created_at`, `updated_at`, `refcount`, `meta_json`. Bytes live in the CAS on disk;
  the table holds metadata + refcount.
- **Turn↔asset association: shipped as the join table** `snapshot_turn_assets`
  (`snapshot_id`, `turn_idx`, `sha256`, `relation`, …) rather than
  `snapshot_turns.meta_json` refs. Trade-off (decided): the join table makes dedup,
  refcount, and future GC queries first-class and indexable; `meta_json` refs would
  have avoided a migration but make refcount/GC and "which turns use asset X"
  queries scan-heavy. The join table was the chosen design and shipped in C2b
  (Migration v14).
- **Search / FTS is design-only** and may be a later slice (today search is
  in-memory per the storage contract). When built, index **text and content
  metadata, not raw HTML**: chat `title`, turn `contentText` (via the existing
  HTML→text extraction), folder/category/tag **names**, `source` URL, and asset
  `originalName` + `mimeType`.
- **Rebuild from store** (derive index/diagnostics from SQLite snapshots/turns/
  assets) is a **Phase C/D diagnostic**.
- **Rebuild DB from packages** is **Phase E import/recovery — not Phase C.**
- **DB vs package files:** DB holds registry, refcounts, and index; package files
  hold the portable projection + an asset-bytes copy. Live CAS bytes sit in
  app-owned storage (DB-adjacent, see ADR-0010); the package `assets/` is the
  export copy.

## Sanitizer Design

- The Phase B **regex sanitizer + CSP is interim** (the validator is headless, so a
  DOM sanitizer cannot run there without a shim — this is why regex was chosen).
- Phase C should **centralize the sanitizer contract** into a shared,
  surface-neutral module consumed by the projector and any future renderer/import,
  rather than duplicating per-feature sanitizers. **Locked targets (C3.0):**
  module `src-surfaces-base/studio/platform/html-sanitizer.js`, global
  `H2O.Studio.html.sanitize`. The extraction + projector rewire happens in **C3.1**
  (behavior-preserving; the existing projector validator must stay green) — **not
  in C3.0**.
- The shared sanitizer should be **allowlist-based**. A **DOM/allowlist
  implementation may be deferred** (headless-validator constraint); at minimum the
  **allowlist contract is centralized and documented** now, with the hardened regex
  as the headless fallback.
- **Keep CSP on `chat.html`** as defense-in-depth regardless of sanitizer
  implementation.

### Minimum allowlist (ChatGPT-like content)

- Block: `p`, `br`, `hr`, `h1`–`h6`
- Lists: `ul`, `ol`, `li`
- Quotes: `blockquote`
- Tables: `table`, `thead`, `tbody`, `tr`, `th`, `td`
- Code: `pre`, `code`
- Inline: `strong`/`b`, `em`/`i`, `sup`, `sub`, `span` (class allowlist for
  math/katex)
- Links: `a` with **safe `href` only** (`http`, `https`, `mailto`); force
  `rel="noopener noreferrer"`
- **Images only after CAS support** — then `img` with `src` restricted to relative
  `assets/` paths or `data:image/*`, plus `alt`
- Everything else stripped; all `on*` handlers, `style`, and
  `script`/`style`/`iframe`/`object`/`embed`/`svg`/`math`/`base`/`meta`/`form`
  removed.

## Deferred (explicitly out of scope for this C1 patch)

- CAS implementation (any byte read/write).
- Binary Tauri capability changes.
- SQLite migrations (no v7 in C1).
- UI wiring (Save-to-Folder button/menu).
- Sync integration / transport.
- WebDAV / cloud / relay.
- Import / recovery (rebuild DB from packages → Phase E).
- Remote URL asset fetching (capture concern).
- PDFs / non-image file types beyond the initial image slice.
- Automatic asset garbage collection (refcount only when assets land; GC design
  must later coordinate with the Sync lane).

## Non-Goals for this patch

This C1 patch is documentation. It changes no runtime JS/TS/Rust, adds no
migration, modifies no Tauri capability, and ships no CAS. v1 remains the only
implemented package version.
